# basil-cose: public API and broker COSE-unseal RPC

- Status: proposed (feeds human review `basil-o6mm`; gates milestones C1–C3)
- Date: 2026-07-01
- br: `basil-ysdj`
- Parent plan: [cose-sealed-invocation.html](cose-sealed-invocation.html) §7.1, §10 (C1, C6), §11
- Settled context: [cose-questions.md](cose-questions.md)

This document settles two things the parent plan deferred to a design gate:

1. **Part A**: the public API of the new broker-free, publishable
   `crates/basil-cose` crate: the three constructions (signed, sealed,
   seal-only), the `Signer`/`Verifier`/`Recipient` traits, caller-supplied
   `external_aad` on both `Sig_structure` and `Enc_structure`, PartyU/PartyV
   identity in the ECDH KDF context, the content-algorithm parameter, the
   strict-decode/validation entry points with parameterized bounds, and the
   SHA3-256 request-hash helper.
2. **Part B**, the broker COSE-unseal RPC: `COSE_Encrypt` bytes + a catalog
   key id in, plaintext out, under `Op::Decrypt`, via the X25519
   materialize-to-use arm, `Zeroizing` end-to-end.

Everything here is API/wire design; no code changes accompany this document.

## 1. Problem statement

The parent plan replaces the bespoke `SecureMessage` sealed-invocation format
with COSE (RFC 9052/9053) and requires a shared profile crate that both the
basil broker/client and `edgebox-rs` consume. Two shapes were left "settled in
the C1 API design document":

- The exact public surface of `basil-cose`, enough for the broker invocation
  path (C2), the Rust client (C3), the interop suite (C4), the C6 Alice/Bob
  demo, and edgebox's three protocol surfaces (signed control RPC, sealed
  enrollment framing, encrypt-only outer layer).
- The broker RPC that lets a workload unseal a `COSE_Encrypt` addressed to a
  backend-custodied X25519 key without ever holding the private key (C6:
  "Bob verifies and unseals via the broker unseal-in-place operation").

## 2. Constraints

Carried from the plan, AGENTS.md, and user API preferences; restated here
because they shaped every choice below:

- **Strict validation everywhere.** No forward-compatible `Unknown` enum arms
  in the profile decoders. Unknown/duplicate labels, claims in unprotected
  headers, indefinite lengths, non-deterministic encodings, untagged
  structures, and wrong CBOR types for known labels are all rejected
  (plan §11.4).
- **minicbor-backed in-tree profile codec.** (Reviewer decision 2026-07-02,
  amending the original coset plan: `minicbor` to align with edgebox-rs and
  avoid a second CBOR stack across the ecosystem; it is also cleanly
  `no_std`/`alloc`.) The COSE structure encode/decode is implemented in-tree
  over `minicbor` behind the codec seam. The design's "in-tree deterministic
  encoder fallback" is the primary and only path. Deterministic encoding is
  by construction (canonical label order, definite lengths, minimal integer
  heads), plus the RFC 8949 §4.2 re-encode-and-compare check on every decode.
- **Dependency budget:** `minicbor`, RustCrypto primitives, `sha3`,
  `zeroize` (+ `getrandom` for nonce/ephemeral randomness). Nothing else at
  runtime. No tokio, no tonic, no serde in `basil-cose`.
- **Broker-free and publishable** (precedent: `basil-nats`). Keys stay behind
  traits; nonces and ephemerals are generated by the library, never supplied
  by the far side; no `unsafe`; workspace deny-level clippy clean.
- **API auditability:** structs over tuples, enums over chars/strings for
  every public parameter position.
- **Zeroizing end-to-end** for every materialized secret intermediate
  (private key bytes, ECDH shared secret, derived CEK, plaintext buffers on
  the open path).
- **edgebox convergence is a v1 requirement:** ChaCha20-Poly1305 as a content
  algorithm and a bare `COSE_Sign1` (sign-only) construction must exist from
  the start; `external_aad` and KDF party identities must be caller-suppliable
  (edgebox binds a NATS subject into control-RPC signatures, a purpose/version
  context into enrollment AEAD, and `d2h`/`h2d` direction strings into its
  HKDF info).

One deliberate divergence from the existing arm: `x25519_seal.rs` derives its
AEAD key with a bespoke HKDF info (`label || ephemeral_pub || recipient_pub`).
The COSE path MUST instead use the RFC 9053 §5.2 `COSE_KDF_Context` structure
as the HKDF info. `basil-cose` owns that derivation; `basil-core` reuses only
the custody scaffolding (materialize via `kv_get_secret`, contributory check
posture, zeroization discipline), not the bespoke KDF. The existing
`x25519_seal` arm continues to serve the untouched
`WrapEnvelope`/`UnwrapEnvelope` surface unchanged.

## 4. Part A: the `basil-cose` public API

### 4.1 Crate layout and dependencies

```
crates/basil-cose            # publishable, broker-free
├── src/lib.rs               # public API, re-exports
├── src/alg.rs               # closed algorithm enums + catalog of codepoints
├── src/label.rs             # private-label registry (-70001..-70005), crit handling
├── src/claims.rs            # Claims struct, CWT map codec, temporal/audience checks
├── src/sign.rs              # signed construction (COSE_Sign1)
├── src/seal.rs              # sealed construction (Sign1 over Encrypt)
├── src/encrypt.rs           # seal-only construction (COSE_Encrypt)
├── src/kdf.rs               # COSE_KDF_Context builder, PartyU/PartyV
├── src/keys.rs              # local key types (Ed25519, X25519), trait impls
├── src/traits.rs            # Signer / Verifier / Recipient
├── src/codec/mod.rs         # the coset seam (see §4.8)
└── src/hash.rs              # SHA3-256 request-hash helper
```

Runtime dependencies: `minicbor` (the in-tree profile codec's CBOR
reader/writer; reviewer decision 2026-07-02, edgebox-rs alignment),
`zeroize`, and RustCrypto: `ed25519-dalek`, `x25519-dalek`, `hkdf`, `sha2`
(HKDF-256), `sha3` (request hash), `aes-gcm`, `chacha20poly1305`, plus
`getrandom` (nonce/ephemeral randomness; the `wasm-js` crate feature maps to
`getrandom/js` for `wasm32-unknown-unknown` in JS hosts). No CBOR type
appears in the public API. Dev-dependencies only: fixture tooling.

`basil-cose` does NOT depend on `basil-proto`. Basil-specific values (the
content-type string registry, plaintext body schemas, catalog key-id
semantics) stay in `basil-proto` per the plan; `basil-cose` treats content
types and key ids as opaque validated newtypes. Dependency edges added:
`basil-core -> basil-cose`, `basil-client -> basil-cose` (client stays free of
OpenBao/tonic-server/age/argon2; `basil-cose` adds none of those).
`edgebox-rs` consumes `basil-cose` alone.

### 4.2 Core identifier and algorithm types

Everything a caller passes is a named struct or a closed enum: no bare
`&[u8]`/`&str`/tuples in public positions, and no `Unknown` arms.

```rust
/// COSE `kid` (bstr). Basil catalog names are UTF-8; edgebox may use raw ids.
pub struct KeyId(Vec<u8>);
impl KeyId {
    pub fn from_str(id: &str) -> Self;          // UTF-8 bytes of a catalog name
    pub fn from_bytes(id: Vec<u8>) -> Result<Self, ProfileError>; // 1..=128 bytes
    pub fn as_catalog_name(&self) -> Option<&str>; // Some iff valid UTF-8
}

/// CWT `cti` (bstr, 1..=64 bytes, sender-unique inside the replay window).
pub struct MessageId(Vec<u8>);

/// CWT `iss`/`aud` and the `-70005` response subject (tstr).
pub struct Subject(String);
pub struct ResponseSubject(String);

/// Content type (header 3, tstr). Registry values live in `basil-proto`
/// (e.g. `basil.sign-request`); edgebox registers its own strings.
pub struct ContentType(String);

/// Seconds since the Unix epoch (CWT `iat`/`exp`).
pub struct UnixTime(pub i64);

// Closed algorithm enums; decode of any other codepoint is an error.
pub enum SignatureAlgorithm { EdDsa }                    // -8
pub enum KeyAgreementAlgorithm { EcdhEsHkdf256 }         // -25
pub enum ContentAlgorithm { A256Gcm, ChaCha20Poly1305 }  // 3, 24
```

`ContentAlgorithm` is a parameter of every encrypting entry point from day
one (plan §11 deferred-item 4 pulled forward): basil invocation v1 passes
`A256Gcm`; edgebox passes `ChaCha20Poly1305` (its existing
`x25519-hkdf-sha256-chacha20poly1305-v1` suite is exactly ECDH-ES + HKDF-256
with ChaCha20-Poly1305 content). Both live in the strict decoder's allow-set;
the broker still rejects any wire `alg` that disagrees with catalog policy
(that check is basil-core's, on top of this crate).

### 4.3 External AAD and KDF party identity

```rust
/// Caller-supplied external_aad for exactly one COSE layer. Empty is the
/// explicit default, not an implicit one.
pub struct ExternalAad(Vec<u8>);
impl ExternalAad { pub fn empty() -> Self; pub fn from_bytes(b: Vec<u8>) -> Self; }

/// Per-layer AAD for the sealed (two-layer) construction.
pub struct SealedAad {
    pub signature: ExternalAad,   // fed to Sig_structure
    pub encryption: ExternalAad,  // fed to Enc_structure
}

/// One party's identity slot in the COSE_KDF_Context (RFC 9053 §5.2).
/// v1 exposes identity only; nonce/other are pinned nil by the profile.
pub struct PartyIdentity(Option<Vec<u8>>);

/// PartyU = message sender, PartyV = recipient, per RFC 9053 §5.1.
pub struct KdfParties {
    pub party_u: PartyIdentity,
    pub party_v: PartyIdentity,
}
impl KdfParties { pub fn anonymous() -> Self; } // both nil (basil invocation v1)
```

Usage split, per the plan's edgebox review: basil invocation passes
`ExternalAad::empty()` on both layers and `KdfParties::anonymous()`; edgebox
binds its delivery NATS subject via the signature-layer AAD, its
purpose/version context via the encryption-layer AAD, and its
direction-separating `d2h`/`h2d` strings via `KdfParties` identities.

**Wire placement of party identities (decision):** when either party identity
is non-nil at seal time, the profile writes it into the recipient structure's
protected headers using the registered ECDH parameters (PartyU identity −21,
PartyV identity −24) and lists them in that layer's `crit`. The opener
reconstructs the `COSE_KDF_Context` from the message and additionally checks
it against `expected_parties` in its params when the caller pins them
(mismatch = reject). Rationale: the KDF context must be reproducible by an
opener that is not the sealer (notably the broker unseal RPC, §5), and the
registered header parameters are the standard, signature/AEAD-covered home
for exactly this. Nonce (−22/−25) and "other" (−23/−26) parameters are
rejected on decode in v1 (strict profile; nothing needs them).

### 4.4 Claims and the private label registry

`basil-cose` owns the claim structs and the private label constants; it does
not own basil's content-type strings or body schemas.

```rust
/// Label registry (RFC 9052 private range, single source of truth).
pub mod label {
    pub const IN_REPLY_TO: i64      = -70001; // bstr, responses
    pub const REQUEST_HASH: i64     = -70002; // bstr(32), responses, SHA3-256
    pub const SENDER_KEY_ID: i64    = -70003; // bstr, must equal outer kid
    pub const RESPONSE_KEY_ID: i64  = -70004; // tstr-encoded KeyId, requests
    pub const RESPONSE_SUBJECT: i64 = -70005; // tstr, requests, optional
}

/// The claim set carried in the COSE_Encrypt protected header (CWT map,
/// header 15, plus basil private labels). One struct; role validators
/// enforce which fields must / must not be present.
pub struct Claims {
    pub issuer: Option<Subject>,               // iss (1)
    pub audience: Option<Subject>,             // aud (3)
    pub expires_at: Option<UnixTime>,          // exp (4); absent => iat + default_ttl
    pub issued_at: UnixTime,                   // iat (6)
    pub message_id: MessageId,                 // cti (7)
    pub sender_key_id: Option<KeyId>,          // -70003
    pub response_key_id: Option<KeyId>,        // -70004
    pub response_subject: Option<ResponseSubject>, // -70005
    pub in_reply_to: Option<MessageId>,        // -70001
    pub request_hash: Option<RequestHash>,     // -70002
}

/// Which claim shape a validator demands.
pub enum MessageRole {
    /// Requires sender_key_id + response_key_id; forbids in_reply_to/request_hash.
    Request,
    /// Requires in_reply_to + request_hash; forbids response_key_id/response_subject.
    Response,
    /// Peer message (C6 demo, edgebox): requires sender_key_id; forbids the
    /// request/response correlation labels.
    Peer,
}
```

Every basil label present, plus content type (3) and the CWT map (15), is
listed in `crit` (2) on encode; decode rejects a missing `crit`, a basil
label absent from `crit`, any label outside the registry (critical or not),
and any claim found in an unprotected header (plan §9).

### 4.5 Traits: `Signer`, `Verifier`, `Recipient`

Native `async fn` in traits (AFIT, stable on our 1.96 toolchain), consumed
through generics: no `async_trait` proc-macro dep, no `dyn` in the public
API. Local implementations complete synchronously; broker-backed
implementations (which live in `basil-client`, not here) await an RPC.

```rust
/// Produces the signature over Sig_structure bytes. The broker backs this
/// with sign-in-place (SigningService.Sign, as Ed25519 transit accepts the
/// full arbitrary-length structure); clients and edgebox use local keys.
pub trait Signer {
    fn key_id(&self) -> &KeyId;                    // becomes the outer kid
    fn algorithm(&self) -> SignatureAlgorithm;
    async fn sign(&self, sig_structure: &[u8]) -> Result<Signature, SignError>;
}
pub struct Signature(Vec<u8>);

/// Resolves a kid and verifies a signature. Implementors enforce their own
/// key/alg pinning (the broker checks the wire alg against the catalog
/// record inside its impl; a client checks against its pinned broker key).
/// Async because kid resolution may be remote: the basil-client impl that
/// resolves an unfamiliar peer kid via the GetPublicKey RPC (the C6
/// Alice/Bob flow) awaits a network call; local pinned-key impls and the
/// broker's in-memory catalog lookup complete synchronously.
pub trait Verifier {
    async fn verify(
        &self,
        key_id: &KeyId,
        algorithm: SignatureAlgorithm,
        sig_structure: &[u8],
        signature: &Signature,
    ) -> Result<(), VerifyError>;
}

/// Opens one COSE_Encrypt addressed to key_id(): ECDH-ES decapsulation +
/// HKDF-256 (COSE_KDF_Context) + content AEAD, returning zeroizing
/// plaintext. The broker-backed impl forwards the exact encrypt bytes to
/// the UnsealCose RPC (§5); the local impl holds a Zeroizing X25519 private.
pub trait Recipient {
    fn key_id(&self) -> &KeyId;
    async fn open(
        &self,
        request: &OpenRequest<'_>,
    ) -> Result<Zeroizing<Vec<u8>>, OpenError>;
}

/// Everything an opener needs, already strictly validated by the entry
/// points in §4.6, plus the raw encrypt bytes so a remote (broker-backed)
/// recipient can forward them verbatim without re-encoding.
pub struct OpenRequest<'a> {
    pub cose_encrypt: &'a [u8],          // exact tagged COSE_Encrypt bytes
    pub external_aad: &'a ExternalAad,   // encryption-layer AAD
    pub expected_parties: Option<&'a KdfParties>, // pin, or None to accept message values
}
```

**Options considered for `Recipient` granularity:**

- *CEK-deriver* (`derive_cek(ephemeral_pub, kdf_context) -> Zeroizing<[u8;32]>`):
  rejected. A broker-backed impl would require the broker to return the CEK
  over the wire, a strictly stronger disclosure than returning this one
  message's plaintext (a CEK holder can forge a same-message ciphertext), and
  it moves AEAD-open outside the audited decrypt operation.
- *Whole-message opener* (chosen): the broker performs decapsulate + open in
  place and returns only plaintext, which is exactly the existing
  `UnwrapEnvelope` privilege shape, and the local impl is symmetric.
- *Sync traits with a blocking broker client*: rejected; the broker client is
  tonic-async, and `block_on` inside async callers is a known hazard.

Shipped local implementations (in `basil-cose::keys`):

```rust
pub struct Ed25519Signer { /* SigningKey (ZeroizeOnDrop), KeyId */ }
pub struct Ed25519Verifier { /* map or single pinned VerifyingKey per KeyId */ }
pub struct X25519Recipient { /* Zeroizing<[u8; 32]> private, KeyId */ }
pub struct X25519RecipientPublic { pub key_id: KeyId, pub public: [u8; 32] } // seal target
```

`X25519Recipient::open` enforces the contributory-secret check (reject
low-order ephemerals before key derivation), mirroring
`x25519_seal.rs`; the shared secret and derived CEK live in `Zeroizing`
buffers on every path, and an X25519 open proves confidentiality, never
sender identity (sender identity comes only from the outer `COSE_Sign1`).

### 4.6 The three constructions

All builders take a params struct; all outputs are complete tagged COSE
bytes. All verify/open entry points strict-decode first (§4.7) and return
typed views: callers never touch coset types.

**(1) Signed: bare `COSE_Sign1`** (edgebox control RPC/push; also any
future sign-only basil surface):

```rust
pub struct SignParams<'a> {
    pub content_type: ContentType,
    pub payload: &'a [u8],
    pub claims: Option<Claims>,        // optional for sign-only messages
    pub external_aad: ExternalAad,     // Sig_structure external_aad
}
pub async fn build_signed<S: Signer>(
    params: &SignParams<'_>, signer: &S,
) -> Result<CoseBytes, BuildError>;

pub struct VerifySignedParams<'a> {
    pub external_aad: ExternalAad,
    pub validation: Option<&'a ValidationParams>, // temporal/audience iff claims expected
}
pub async fn verify_signed<V: Verifier>(
    bytes: &[u8], verifier: &V, params: &VerifySignedParams<'_>,
) -> Result<VerifiedSigned, VerifyError>;

pub struct VerifiedSigned {
    pub content_type: ContentType,
    pub payload: Vec<u8>,
    pub claims: Option<Claims>,
    pub signer_key_id: KeyId,
}
```

**(2) Sealed: `COSE_Sign1` over embedded tagged `COSE_Encrypt`** (basil
invocation requests/responses, C6 peer messages):

```rust
pub struct SealParams<'a> {
    pub content_type: ContentType,
    pub plaintext: &'a [u8],
    pub claims: Claims,                       // required; sender_key_id must equal signer kid
    pub role: MessageRole,
    pub recipient: X25519RecipientPublic,
    pub content_algorithm: ContentAlgorithm,
    pub aad: SealedAad,
    pub kdf_parties: KdfParties,
}
pub async fn build_sealed<S: Signer>(
    params: &SealParams<'_>, signer: &S,
) -> Result<CoseBytes, BuildError>;
```

The library generates the 12-byte content nonce and the ephemeral X25519
keypair internally (fresh per message, `Zeroizing`); there is no
caller-supplied-nonce path in the public API (secure-by-construction, per
AGENTS.md). A `#[cfg(feature = "fixtures")] build_sealed_with_parts` exists
for deterministic test vectors only, mirroring `x25519_seal::seal_with_parts`.

Verification and opening are two explicit stages so the broker (and any
verifier) can run replay/expiry/audience preflight before decryption
(plan §3.4 order), and so opening can be remote:

```rust
pub async fn verify_sealed<V: Verifier>(
    bytes: &[u8], verifier: &V, params: &ValidationParams,
) -> Result<VerifiedSealed, VerifyError>;

pub struct VerifiedSealed {
    pub claims: Claims,                  // signature-trusted, pre-decrypt
    pub content_type: ContentType,
    pub signer_key_id: KeyId,            // outer kid; == claims.sender_key_id (checked)
    pub recipient_key_id: KeyId,         // from the recipient structure
    pub content_algorithm: ContentAlgorithm,
    // private: exact embedded COSE_Encrypt bytes for open()
}
impl VerifiedSealed {
    pub async fn open<R: Recipient>(
        &self, recipient: &R, aad: &ExternalAad,
        expected_parties: Option<&KdfParties>,
    ) -> Result<Opened, OpenError>;      // rejects recipient.key_id() != recipient_key_id
}
pub struct Opened {
    pub plaintext: Zeroizing<Vec<u8>>,
    pub content_type: ContentType,
}
```

`verify_sealed` performs, in order: strict decode (nesting: payload MUST be a
tagged `COSE_Encrypt`; `COSE_Encrypt0`/`COSE_Mac0`/double-`Sign1`/plain
bodies rejected), signature verification via the `Verifier`, temporal +
audience checks from `ValidationParams`, the strip-and-re-sign cross-check
(`-70003` == outer kid), and role-shape validation. Replay-cache lookup stays
a basil-core concern (`VerifiedSealed.claims.message_id` is the input); the
crate stays state-free.

**(3) Seal-only: bare `COSE_Encrypt`** (edgebox enrollment outer layer;
tamper evidence from the AEAD plus an inner signed envelope):

```rust
pub struct EncryptParams<'a> {
    pub content_type: ContentType,
    pub plaintext: &'a [u8],
    pub recipient: X25519RecipientPublic,
    pub content_algorithm: ContentAlgorithm,
    pub external_aad: ExternalAad,
    pub kdf_parties: KdfParties,
}
pub fn build_encrypted(params: &EncryptParams<'_>) -> Result<CoseBytes, BuildError>;

pub fn decode_encrypted(bytes: &[u8]) -> Result<EncryptedMessage, DecodeError>; // strict decode only
pub struct EncryptedMessage {
    pub content_type: Option<ContentType>,
    pub recipient_key_id: KeyId,
    pub content_algorithm: ContentAlgorithm,
    // private: exact bytes
}
impl EncryptedMessage {
    pub async fn open<R: Recipient>(
        &self, recipient: &R, aad: &ExternalAad,
        expected_parties: Option<&KdfParties>,
    ) -> Result<Opened, OpenError>;
}
```

`build_encrypted` is fully local and synchronous: sealing needs only the
recipient public key, no `Signer` and no broker. Claims are not part of the
seal-only construction in v1 (edgebox's outer layer carries none; anything
needing claims should be signed).

### 4.7 Strict decode and `ValidationParams`

```rust
pub struct ValidationParams {
    pub now: UnixTime,                    // injected, never sampled internally
    pub max_clock_skew: Duration,
    pub max_ttl: Duration,
    pub default_ttl: Duration,            // effective exp when absent (60s today)
    pub allowed_audiences: BTreeSet<Subject>, // empty = no audience restriction configured;
                                          // when non-empty, a present aud must match and
                                          // requests without aud are accepted per today's rule
    pub role: MessageRole,
}
```

Bounds are parameters (the broker feeds its `[invocation]` config; edgebox
feeds its own), preserving today's semantics: skew check on `iat`, effective
`exp = iat + default_ttl` when absent, `max_ttl` cap on the explicit span,
audience list check before policy. `now` is a parameter for testability and
because a broker-free crate must not own time policy.

The strict decoder (shared by every entry point) rejects: untagged top-level
structures, wrong tags, indefinite-length items, non-deterministic encodings
(RFC 8949 §4.2, verified by re-encoding through the codec seam and comparing
bytes), duplicate labels, unknown labels anywhere, known labels with wrong
CBOR types, `crit` violations, unknown `alg`/`kty`/`crv` codepoints, absent
`alg`, claims in unprotected headers, and recipient arrays with length ≠ 1
(single-recipient profile in v1; multi-recipient is plan deferred-item 5).
Error enums are closed and diagnostic (`DecodeError::IndefiniteLength`,
`DecodeError::UnknownLabel { label: i64 }`, …) but never echo secret bytes.

### 4.8 Codec seam and determinism check

`src/codec/mod.rs` is the only module that names CBOR-library types.
(Reviewer decision 2026-07-02: the CBOR backend is `minicbor`, aligning with
edgebox-rs; the in-tree COSE profile encoder over `minicbor` is the primary
and only path; the originally-planned coset backing and its fallback story
are superseded.) The seam is realized as the crate-internal module boundary:
encode functions emit the deterministic form by construction (canonical
label order, definite lengths, minimal integer heads), and the strict decode
functions re-encode the parsed semantics through the same builders and
require byte equality with the input.

C1 lands a determinism suite: every fixture must survive
encode→decode→re-encode byte-identically, and the codec's output for our
structures must match an independently hand-encoded expectation for at least
one vector per construction. The re-encode determinism check also runs on
every decode in release builds (it is cheap relative to the crypto and is
itself a strictness guarantee, not just a test). A byte-level pre-scan
additionally rejects indefinite-length items and non-minimal integer heads
with precise diagnostics before structure decoding.

### 4.9 SHA3-256 request-hash helper

```rust
pub struct RequestHash(pub [u8; 32]);
/// SHA3-256 over the complete tagged request COSE_Sign1 bytes (claim -70002).
pub fn request_hash(request_cose_sign1: &[u8]) -> RequestHash;
```

Used by the broker when building responses and by clients when checking
`-70002`; living here keeps the primitive choice (plan §11.4: SHA3-256, the
conservative pick) in one place.

### 4.10 What basil-client adds on top (C3 preview, for review context)

Not part of this crate, but the trait design is validated by naming them:

## 5. Part B: the broker COSE-unseal RPC

### 5.1 Decision: a new `UnsealCose` RPC on `AeadService`

Options considered:

1. **Extend `Decrypt`.** `DecryptRequest` carries a `CiphertextEnvelope`
   (`alg`/`key_version`/`nonce`/`ciphertext`) shaped for transit symmetric
   decrypt. A `COSE_Encrypt` cannot be decomposed into it: the AEAD AAD is
   the `Enc_structure`, which embeds the *exact serialized protected-header
   bytes*: any parse-to-protobuf/re-encode round trip that is not
   bit-perfect breaks the tag. Cramming whole COSE bytes into
   `CiphertextEnvelope.ciphertext` with placeholder `alg`/`nonce` fields
   abuses every field's documented meaning. Rejected.
2. **Extend `UnwrapEnvelope` (`KemEnvelope` grows a COSE arm).** Same
   exact-bytes objection (a `oneof` carrying raw bytes next to the parsed
   fields makes one RPC speak two protocols with different validation
   pipelines), and it would entangle the ML-KEM/software-custody dispatch
   already in that handler with the COSE strict decoder. The plan explicitly
   keeps the `WrapEnvelope`/`UnwrapEnvelope` surface unconverted. Rejected.
3. **New RPC, opaque COSE bytes** (chosen). Matches the plan's thin-envelope
   philosophy (§4: carriers move opaque COSE bytes), keeps strict validation
   in exactly one decoder (`basil-cose`), and leaves the existing surfaces
   byte-for-byte untouched.

Proto change (`crates/basil-proto/proto/basil/broker/v1/broker.proto`,
`AeadService`):

```proto
service AeadService {
  // ... Encrypt, Decrypt, WrapEnvelope, UnwrapEnvelope unchanged ...

  // Open a COSE_Encrypt sealed to a backend-custodied X25519 recipient key,
  // in place. Strict COSE profile; gated by op:decrypt on `key_id`.
  rpc UnsealCose(UnsealCoseRequest) returns (UnsealCoseResponse);
}

message UnsealCoseRequest {
  // Catalog name of the recipient sealing key (class: sealing, engine: kv2,
  // keyType: x25519).
  string key_id = 1;
  // Complete tagged COSE_Encrypt bytes, exactly as received.
  bytes cose_encrypt = 2;
  // Encryption-layer external_aad; empty when the protocol binds none.
  optional bytes external_aad = 3;
}

message UnsealCoseResponse {
  // The recovered plaintext.
  bytes plaintext = 1;
}
```

No party-identity fields on the request: per §4.3 the KDF party identities
travel inside the `COSE_Encrypt` recipient protected headers, so the broker
reconstructs the `COSE_KDF_Context` from the message itself. (Pinning
expected parties and/or `external_aad` server-side is an optional per-key
catalog refinement: the `sealingPin`, implemented in `basil-2rqj`; see open
question 2.)

### 5.2 Authorization: `Op::Decrypt`, no new op

The unseal is precisely the decrypt-in-place privilege on the named sealing
key, identical in power to today's `UnwrapEnvelope` on the same key (both
return plaintext sealed to a `Class::Sealing` catalog key, both are gated by
`Op::Decrypt` in `crates/basil-core/src/service/aead.rs`). Introducing a new
op would (a) split one privilege into two names an operator must keep in
sync, and (b) contradict the existing grant model where reading/writing/
rotating are the separated axes, not wire formats. The `Op` enum in
`crates/basil-core/src/core/catalog/policy.rs` is unchanged.

Oracle scope is likewise unchanged: an `op:decrypt` grantee could already
unseal arbitrary `KemEnvelope`s addressed to the key; `UnsealCose` adds a
second encoding of the same capability. A successful unseal proves
confidentiality, not sender identity: sender authentication is the caller's
`verify_sealed` step (or the invocation service's, on that path), never this
RPC's.

### 5.3 Handler flow (`crates/basil-core/src/service/aead.rs`)

1. `authorize(&request, Op::Decrypt, &body.key_id)`: SO_PEERCRED actor, PDP
   decision, audit decision record, exactly like `unwrap_envelope`.
2. Size cap: `cose_encrypt.len()` against `limits().max_encrypt_size`
   (same cap the sibling handlers use), before any parse.
3. Strict decode via `basil_cose::decode_encrypted`: closed enums throughout;
   any unknown alg/kty/crv/label, wrong nesting, or non-deterministic
   encoding is `InvalidArgument` (no partial acceptance, no `Unknown` arms).
4. Cross-checks against the catalog record for `key_id`:
   recipient `kid` (when present) must equal `key_id`; recipient `alg` must
   be `ECDH-ES + HKDF-256`; ephemeral key must be OKP/X25519; content `alg`
   must be in the catalog-permitted set (A256GCM, ChaCha20-Poly1305); the
   catalog entry must be `class: sealing`, `engine: kv2`, `keyType: x25519`.
   Wire algorithms are checked against configuration, never trusted.
5. `BackendManager::unseal_cose(&key_id, &encrypted, external_aad).await`
   (§5.4), then move the plaintext into the response at the last possible
   point. The plaintext leaving the broker is the RPC's purpose (as with
   `Decrypt`/`UnwrapEnvelope`); everything upstream of the response stays
   `Zeroizing`.

Errors are uniform and non-oracular: every authentication failure inside the
open (wrong key, tampered ciphertext, bad AAD, low-order ephemeral) surfaces
as one opaque decrypt-failed status, per the existing `SealError::OpenFailed`
posture. No secret byte, derived key, or shared secret ever reaches an error
string, log, or audit record.

### 5.4 Manager arm (`crates/basil-core/src/core/manager.rs`)

New method beside `unwrap_envelope` (~L1173):

```rust
pub async fn unseal_cose(
    &self,
    key_id: &str,
    encrypted: &basil_cose::EncryptedMessage,
    external_aad: &basil_cose::ExternalAad,
) -> Result<Zeroizing<Vec<u8>>, ManagerError>
```

- Routes like the existing X25519 arm: `require_class(Class::Sealing)`;
  `rotate`/`import`/`get`/`set` remain refused; the private stays
  un-gettable; reconcile behavior is untouched (same key entries, same
  `publicPath` discipline; this RPC adds a use, not a key shape).
- Materializes the private **only** via
  `Backend::kv_get_secret(path) -> Zeroizing<Vec<u8>>` (never
  `kv_get`/`KvValue`), converts with `try_into` to the fixed array (never
  indexes), per `docs/agents/materialize-key-custody.md`.
- Wraps it in `basil_cose::X25519Recipient` and calls
  `EncryptedMessage::open`, so the ECDH (with the contributory/low-order
  check), the RFC 9053 `COSE_KDF_Context` HKDF-256 derivation, the
  `Enc_structure` AAD assembly, and the AEAD open all execute in the one
  implementation the interop suite certifies. `basil-core` does not grow a
  second COSE KDF.
- Zeroizing chain: KV body text and decoded bytes (`kv_get_secret`),
  private array, `StaticSecret` (ZeroizeOnDrop), shared secret, derived CEK,
  and the returned plaintext buffer. The single non-zeroized copy is the
  protobuf response, created last.
- No panic path: all errors map to `ManagerError` arms; malformed or
  mismatched envelopes fail closed before any plaintext exists.

The bespoke-HKDF `x25519_seal::open` path and the `unwrap_envelope` manager
arm are not modified; the two arms share custody discipline, not key
derivation. (They MAY later share a private materialize helper, an
implementation detail for C2, not an API question.)

### 5.5 Client and Go surface

- `basil-client`: `AeadClient` (or the demo-facing sub-client) gains
  `unseal_cose(key_id, cose_encrypt, external_aad)` and the
  `BrokerRecipient` adapter of §4.10.
- `clients/go`: stub regeneration picks up `UnsealCose`; a thin
  `aead.UnsealCose` wrapper follows the existing sub-client pattern. Go
  COSE *construction* helpers remain deferred (plan deferred-item 6); the
  RPC itself is bytes-in/bytes-out and needs no Go COSE library.

## 7. Open questions (all resolved in the basil-o6mm review)

1. **`Verifier` synchronicity.** RESOLVED (review, 2026-07-01): `verify` is
   `async`. The sync rationale only covered the broker's in-memory catalog
   lookup and pre-pinned local keys; a basil-client verifier that resolves
   an unfamiliar peer kid via the `GetPublicKey` RPC (the C6 Alice/Bob flow)
   is a network call. AFIT makes async free for the sync implementors,
   while retrofitting async later is a breaking change.
2. **Server-side KDF-party/purpose pinning.** RESOLVED (review, 2026-07-02):
   deferred from v1; **IMPLEMENTED 2026-07-02** (`basil-2rqj`). A `class: sealing`
   catalog key may carry an optional `sealingPin` pinning the KDF `PartyU`/`PartyV`
   identities and/or an allowed `external_aad` set; the manager `unseal_cose` arm
   cross-checks the envelope against the pin and refuses `PermissionDenied`
   (`ManagerError::UnsealContextNotPermitted`) when it is not among the pinned
   contexts, narrowing the `UnsealCose` oracle from "any envelope addressed to the
   key" to "only envelopes bound to these contexts". Absent = unchanged.
3. **`no_std` posture for `basil-cose`.** RESOLVED (review, 2026-07-02):
   pursue `no_std` if it comes cheap, not because embedded is a requirement,
   but so the crate does not exclude a WASM target (`wasm32-unknown-unknown`).
   The ingredients cooperate: coset and the RustCrypto stack are
   `no_std`-capable, and `Zeroizing<Vec>`/`BTreeSet` are `alloc`, not `std`.
   C1 should attempt `#![no_std]` + `alloc` from the start (getrandom/wasm
   nonce+ephemeral generation is the one part to verify); if that turns into
   real friction, ship v1 as `std` but keep dependencies `no_std`-capable so
   the door stays open. Either way, do not add `std`-only public API surface.
4. **Detached payloads.** RESOLVED (review, 2026-07-02): deferred; the v1
   API omits them. Adding later is additive (`SignParams.payload` becomes an
   enum of embedded/detached).
5. **`allowed_audiences` empty-set semantics.** RESOLVED (review,
   2026-07-02): empty set means no audience restriction (status-quo rule
   confirmed); when non-empty, a present `aud` must match, and messages
   without `aud` are accepted.

### 7.1 C1 implementation notes (2026-07-02, br `basil-9j2s`)

Recorded deviations/clarifications from the shapes sketched above, made
while landing `crates/basil-cose`:

1. **CBOR backend is `minicbor`** (reviewer amendment, see §2/§4.1/§4.8).
2. **`verify_sealed` takes `VerifySealedParams { signature_aad, validation }`**
   rather than `&ValidationParams` alone: the sealed construction binds a
   caller-supplied `Sig_structure` external AAD at build time (§4.3), so the
   verifier must be able to supply the same bytes or non-empty
   signature-layer AAD could never verify.
3. **`EncryptedMessage.content_type` is `ContentType`, not
   `Option<ContentType>`**: the profile requires a content type on every
   construction (builders always write one; `Opened.content_type` is
   non-optional), so the decoded view carries it unconditionally.
4. **`KeyId::from_str` is `KeyId::from_text(&str) -> Result<Self, ProfileError>`**:
   an inherent `from_str` collides with the `FromStr` idiom (deny-level
   clippy) and the 1..=128-byte bound must be enforced on this path too.
5. **`ContentType` enforces a `type/subtype` media-type shape** (non-empty,
   exactly one `/`, no surrounding whitespace) per RFC 9052's tstr content
   type semantics; basil-proto registry strings should therefore be of the
   `application/basil.sign-request` form.
6. **`verify_signed` claims-presence rule**: `validation: Some(_)` demands
   claims and validates them; `validation: None` demands their absence
   (strict both ways; `VerifyError::ClaimsPresenceMismatch`).
7. **Party identities**: carried in recipient protected headers −21/−24 and
   listed in that layer's `crit` exactly as §4.3 specifies (the in-tree
   codec has no registry-representation limits).
8. **Transcript removal sequencing** (br `basil-lp6g`): plan §7.2 deletes the
   v1 transcript canonicalization from `basil-proto` in C1, but the C2/C3
   rewrites that stop calling it land later. To keep the workspace green
   without a compatibility shim in `basil-proto`, the deleted functions moved
   verbatim into private `mod transcript` blocks inside
   `basil-core/src/service/invocation.rs` and
   `basil-client/src/sealed_invocation.rs` (each crate temporarily carries
   the `blake3` dependency); both blocks are deleted wholesale with the old
   envelope in C2 (`basil-znut`) and C3. The `basil-proto` public surface is
   already the post-COSE one.
