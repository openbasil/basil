The broker loads JSON (the export-tool output). Throughout this document the schema is shown in Nix (the form an operator authors) and §5 specifies the Nix→JSON export. The JSON the broker consumes is the mechanical projection of the Nix below (symbolic names resolved to numbers; see §4–§5).

1. Problem statement & constraints

The broker is the policy decision point (PDP) for a Unix-socket signing/secrets service that fronts a Vault-compatible backend (OpenBao or HashiCorp Vault). Today it can sign, verify, and mint, but it has no declarative source of truth for (a) which keys exist, (b) which backend/engine serves each key, or (c) who may do what to which key. This document specifies that source of truth: a catalog (key inventory + backend routing) and a policy (authorization rules) that the broker loads at startup.

The schema is generic and provider-neutral: no edge/realm/host-specific terms appear in the broker's view. Realm semantics and catalog wiring live in the consuming layer, which emits the JSON this broker consumes (§5).

Two constraints drive every default (README "Principles", §17):

  • Secure by default (default-deny). Nothing is permitted until a policy rule grants it. An empty policy authorizes nothing. There is no ambient authority from "the backend is unlocked." A public key is world-readable, but only for read ops and only because the catalog explicitly classed it public (§3.5).
  • Least privilege (PLP). Every grant is the narrowest that works: a rule binds explicit principals to an explicit action over an explicit target. Writes (set/rotate/import/new_key) live in a separate operator role and are never implied by any read/use/mint grant. mint is its own capability, never implied by sign (§3.1).

Authorization binds to the connecting peer's SO_PEERCRED uid/gid (a kernel-captured snapshot, §17.3). The catalog destination owner becomes the allowed caller uid. /proc/<pid>/exe is advisory only and never a decision input.

⚠️ Always isolate runtime environments: never mix development and production keys in the same server. Mixing keys from multiple environments can increase the risk of accidental data breaches or accidental data destruction. One broker instance serves exactly one environment; environment is therefore not encoded in key names.

Scope

In scope: the schema for catalog + policy, glob-match semantics, user/group resolution, the Nix→JSON export contract, and the Rust type contract for vault-flk. Out of scope (other issues): the PDP evaluation engine internals (vault-1l8), multi-backend manager wiring (vault-jso), startup reconcile execution (vault-zrg), audit log (vault-vq5), master-unlock (vault-er3).

2. Catalog schema

The catalog is the key inventory + backend routing table: one document loaded at startup, per-generation immutable (a new generation = a new file = a restart).

2.1 Key names

Keys are dotted, lowercase segment paths, e.g. web.tls.signing_key. Grammar:

key      = segment ("." segment)*
segment  = [a-z0-9] [a-z0-9_-]*          # lowercase; no dots inside a segment

A segment may contain _ and - but never .: the dot is the only separator, so glob matching (§3.4) is unambiguous. Names are the broker's stable identifiers, decoupled from backend paths (the backend path is an opaque routing detail that can change without renaming the key). Environment prefixes (prod./dev.) are forbidden by convention; one broker = one environment (§1).

2.2 Per-key backend routing

Each key routes to exactly one backend instance (named in the top-level backends table). engine selects the sub-engine within that backend; it is optional and inferred from class for the Vault-compatible backend kind (asymmetric/symmetric → transit; value/public → kv2). Set engine explicitly only to override the inference or when a backend exposes multiple engines that the class cannot disambiguate. The pki engine is never inferred: set it explicitly on a class:asymmetric CA key whose mint issues X.509-SVID leaves or broker TLS leaves (the CA key signs in place; see designs/spiffe-workload-api.html §4). path is the backend-native locator, opaque to policy.

2.3 Concrete example (Nix)

{
  schemaVersion = 1;

  backends = {

      requires = [ ];

  keys = {

      path = "nats-account";            # transit key NAME (the backend builds transit/<verb>/<name>)
      writable = true;
      missing = "error";

      description = "NATS account signing key; mints user JWTs in place (seed never leaves transit).";
    };

    "web.tls.signing_key" = {
      class = "asymmetric";
      keyType = "ed25519";
      backend = "bao";
      path = "web-tls";                 # transit key NAME, not the keys/ HTTP path
      writable = true;
      missing = "error";
      description = "Signing key for the web TLS leaf; signs in place.";
    };

    "backups.aead_key" = {
      class = "symmetric";
      keyType = "aes-256-gcm";
      backend = "bao";
      path = "backups-aead";            # transit key NAME
      writable = true;
      missing = "generate";            # crypto key: created in transit from keyType (no recipe block)
      description = "Symmetric AEAD key for ENCRYPT_PAYLOAD/DECRYPT_PAYLOAD of backup blobs.";
    };

    "web.tls.ca_cert" = {
      class = "public";                # world-readable by definition (§3.5)
      backend = "bao";
      engine = "kv2";
      path = "secret/data/web/tls/ca-cert";
      writable = false;                # externally managed; replaced out-of-band (§2.4.2)
      missing = "warn";
      description = "Public CA certificate (PEM); readable by every TLS client.";
    };

    "grafana.admin_password" = {
      class = "value";
      backend = "bao";
      engine = "kv2";
      path = "secret/data/grafana/admin";
      writable = true;
      missing = "generate";
      generate = { format = "ascii-printable"; bytes = 24; };
      description = "Opaque config value a service cannot take as a token; RAM-only GET.";
    };
  };
}

2.3.1 Backend capability enforcement

In the Nix module, implementation is a preset such as OPENBAO_2_5, VAULT_2_0, or VAULT_2_0_ENT. The export expands that preset into the JSON kind = "vault", engines, capabilities, and mintKeyTypes fields; hand-authored JSON supplies those fields directly.

Each backend may declare what it provides through engines, capabilities, and mintKeyTypes. Basil derives what the catalog requires from the routed keys (their effective engine, any algorithm-specific capability such as future PQC transit support, and any missing=generate transit key type) and unions that with explicit backend-level requires. Startup and check enforce required ⊆ provided without backend I/O.

A backend with engines = [], capabilities = [], and mintKeyTypes = [] is capability-unknown: enforcement is skipped for that backend with a warning. Once a provides set is declared, --capability-policy / BASIL_CAPABILITY_POLICY controls gaps: strict (default, fail closed), degraded (defer to each key's missing policy; required-key gaps remain fatal), or off.

There is no credential class: a credential a service merely reads is a value; a NATS .creds is an API mint output (a JWT, decomposable to a keypair + JWT for BYOK), not a stored catalog key. There is no public:true flag: public material is the public class (§3.5).

2.4.2 writable semantics

writable is a catalog-level hard cap: "the broker may mediate writes (set/rotate/import/new_key) to this key." A write is allowed only if both writable = true and the operator role grants it (defense-in-depth). writable = false ⇒ the key is externally managed: no broker-mediated write is ever permitted, even for an operator. Replacement happens out-of-band: directly in the backend (e.g. an operator rotates a CA key in bao) or via re-provisioning/import at deploy time. The flag makes externally-managed keys explicit and enforced.

2.5 generate recipes (value / public material)

Formats mirror the edge catalog generators (secret-defs.nix). The recipe applies only to non-crypto material; crypto keys generate from keyType.

2.6 labels

labels is an optional list of tags, each either name=value or a bare slug. It captures distinctions the structural schema cannot. Reserved convention:

  • nats_type=<letter>: for asymmetric keys used as NATS NKeys, where <letter> is the single NKey prefix letter: A account, O operator, U user, N server, C cluster, X curve. The raw Ed25519 public is identical across NATS roles; the NKey-encoded form differs only by this prefix letter, which the minter must prepend when building iss (and the subject). An ed25519-nkey key without a nats_type label is a load-time warning (mint will fail for it).
  • PQC/provider metadata labels: reserved for provider-agile keys and validated by the broker loader. Each reserved label may appear at most once per key and must use name=value form:
  • crypto_provider=vault-transit|local-software
  • crypto_provider_version=<non-empty backend or software-provider version>
  • pqc_algorithm=ml-dsa-44|ml-dsa-65|ml-dsa-87|ml-kem-512|ml-kem-768|ml-kem-1024
  • pqc_custody=backend-native|software-encrypted
  • crypto_provider_policy=backend-preferred|backend-required|local-software
  • migration_target=<non-empty catalog key name>

Labels are otherwise free-form and reserved for future policy use (e.g. a rule targeting keys by label). Unknown labels are preserved, not rejected, and may repeat.

3. Policy schema

The policy is the authorization allow-list. Default-deny: a request is permitted only if some rule matches (principal, action, target). Authored in Nix; exported to JSON (§5). Catalog and policy are separate documents.

3.1 Operations

read ops :  get | list | get_public_key | verify
use  ops :  sign | encrypt | decrypt                     # use the key in place; no export, no mutation
mint op  :  mint                                          # capability emission (§3.1.1)
validate :  validate                                      # verify a presented JWT-SVID vs the issuer JWKS (§3.1.2)
write ops:  set | rotate | import | new_key               # operator-only; never implied

3.1.1 mint is its own capability

mint (e.g. a NATS user JWT, a JWT-SVID) is internally the atomic combination of sign the signing input + assemble the credential, but in the authorization model it is a distinct grant, separate from raw sign, and neither implies the other. This is the least-privilege point: a service that mints NATS user JWTs from an account key must not thereby gain raw sign over that key (raw sign is strictly more powerful: it could forge any signature or any JWT). mint is therefore neither a read, a use-sign, nor a write op; it is its own category, granted explicitly (typically via the minter role).

3.1.2 validate

validate authorizes a caller to have the broker verify a presented JWT-SVID for an audience against the issuer's published JWKS, the SPIFFE Workload API ValidateJWTSVID (see designs/spiffe-workload-api.html §9). It reads only public key material (never a private key, never a stored value), so it neither signs, mints, nor mutates. It is a separately grantable capability (its own validator role), gated on the issuer key (uid, validate, issuer_key), so the policy names exactly which callers may use Basil as a token verifier rather than making validation ambient. (It is not folded into reader: a caller that reads values is not thereby a token verifier.)

3.2 Roles

A role is a reusable named op set, referenced from a rule's action as role:<name>. Roles carry only ops. The operator role is the canonical home for write ops and is never merged into a read/use/mint role.

roles = {
  reader    = [ "op:get" "op:list" "op:get_public_key" ];
  signer    = [ "op:sign" "op:verify" "op:get_public_key" ];
  verifier  = [ "op:verify" "op:get_public_key" ];
  minter    = [ "op:mint" "op:get_public_key" ];
  validator = [ "op:validate" ];
  encrypter = [ "op:encrypt" "op:decrypt" ];
  operator  = [ "op:set" "op:rotate" "op:import" "op:new_key" ];
};

3.3 Rules

A rule binds principal → action → target, with an optional comment. All three are flat prefix lists (URI-like terms); field names are singular.

  • principal terms: user:<name> | group:<name> | * (any caller, §3.6).
  • action terms: role:<name> | op:<op> | * (any op). The effective op set is the union of inline op: terms and the ops of referenced role:s.
  • target terms: bare dotted key names or globs (targets are always keys, so no qualifier prefix) | * (any key, §3.6, root-only).
  • comment: optional free text (recommended for auditability).
rules = [
  { id = "nats-account-minter";
    principal = [ "user:svc-nats" ];
    action    = [ "role:minter" ];
    target    = [ "nats.account" ];
    comment   = "the nats service mints user JWTs from the account key"; }

  { id = "grafana-value-reader";
    principal = [ "user:svc-grafana" ];
    action    = [ "role:reader" ];
    target    = [ "grafana.**" ];
    comment   = "grafana reads its own opaque values"; }

  { id = "root-ssh-keys-writer";
    principal = [ "group:wheel" ];
    action    = [ "role:operator" ];
    target    = [ "user.root.ssh.authorized_keys" ];
    comment   = "users in wheel group can replace root ssh keys"; }

  { id = "root-all";
    principal = [ "user:root" ];
    action    = [ "*" ];
    target    = [ "*" ];                 # any-key target -> principal must be root (§3.6)
    comment   = "root may do anything"; }
];

A request (uid/gid, op, key) is allowed iff some rule's resolved principal matches the caller, its action lists the op, and any of its target globs matches the key. Otherwise denied. Rules never deny: denial is the absence of an allow (no explicit deny rules; default-deny makes them redundant and avoids precedence ambiguity).

Every term must carry a known prefix (user:/group:/role:/op:) or be the bare-key / * form valid for its position; an unknown or missing prefix, or an unknown referenced name/role, is a fatal load error (no silent miss).

3.4 Glob (target) matching semantics

A target glob matches a dotted key segment-by-segment. Globs are only valid in the last position. Load-bearing definition (must match across export tool, PDP, and tests):

  • * as the entire term is the any-wildcard; see §3.6 (matches any key, subject to the root-only safety rule).
  • * in the last segment matches exactly one segment, any value; it does not cross a dot. (web.tls.* matches web.tls.signing_key, not web.tls.a.b.)
  • ** in the last segment matches one or more segments (a non-empty tail). It is valid only as the last segment. (web.** matches web.tls, web.tls.signing_key; it does not match bare web.)
  • A literal segment matches that segment exactly (lowercase, case-sensitive).
  • No intra-segment wildcards. web* is a fatal syntax error (not a literal, not a partial match): it is too easy to write an accidentally-broad grant. Per-system patterns like user.*.ssh.authorized_keys (glob not in last position) are likewise rejected; that is system-level policy, not secrets policy.

3.5 public keys are world-readable

A key of class: public is readable by every caller for get / get_public_key, by definition of the class: no rule, and no users:["*"], is required or written. This replaces the former public_read stanza. Only public-class keys get this; every other key is default-deny. (A public key is still subject to default-deny for any non-read op: nobody writes it through the broker unless writable + an operator rule allow.)

3.6 Safety rules for *

* as a whole term means "any of that type," bounded by:

  • * target ⇒ root-only. A rule whose target contains * (any key) is valid only if its principal is exactly user:root (uid 0). Any other principal with a * target is a fatal load error. This prevents an accidentally catch-all grant.
  • * principal (any caller) is permitted but is read-only in spirit (most "everyone may read" needs are already met by the public class, §3.5); the engine still applies default-deny elsewhere.
  • * action (any op) is permitted only on a root rule (it pairs naturally with the root-only * target).

3.7 Missing-key policy

At startup the broker reconciles the catalog against its backends (executed by vault-zrg). Each key's missing decides behavior when the backend has no material:

missingBehavior when absent
error (default)Hard error. With a --require lint flag the process exits non-zero (§17.7).
warnLog a warning, continue; the key's ops fail at request time until it exists.
generateCreate material: crypto keys from keyType; value/public from the generate recipe (§2.5). A value/public key with missing="generate" but no generate block is a load error.

4. User/group resolution & logging

Policy is authored with symbolic names (svc-nats, wheel). The broker decides on numeric uid/gid from SO_PEERCRED. The export tool resolves names → numbers at export time: that is exactly when the mapping is known, which avoids drift (a name resolved at any later time risks disagreeing with the system the policy was built for).

The export therefore emits, alongside the numeric rules, two tables (§5):

  • names: uid → name and gid → name, used only for logging. The broker logs principals as name(num), e.g. svc-nats(9002) or wheel(10): the number is what the kernel passes (authoritative), the name is for human readers.
  • memberships: uid → [gid], each uid's full declared group set (primary + supplementary). Supplementary groups are needed for service permissions and for group rules like group:wheel. SO_PEERCRED yields only uid + primary gid, so the broker expands the kernel-trustworthy uid to its declared group set via this table. Keeping memberships in the declarative export (rather than reading /etc/group at runtime) keeps the whole model declarative and auditable; on a system built from the same source, declared == actual.

⚠️ NixOS containers / VMs: mind where the exporter runs. If the export is invoked in the host config, it will not see user/group mappings that exist only inside a container; an export run inside container.<name>.config will not see host mappings. Scope the exporter to the same boundary as the users the policy references.

4.1 PDP decision (per request)

The PDP (vault-1l8) reads the peer's SO_PEERCRED → (uid, gid, pid) and decides on uid plus the declared group set for that uid:

  1. resolve the request's (op, key);
  2. allow iff uid matches a rule's user: principal, or any gid in the uid's declared group set matches a rule's group: principal, or the key is class: public and the op is a public read (§3.5);
  3. else deny.

pid / /proc/<pid>/exe are recorded for audit but are advisory only (the exe read is racy; SO_PEERCRED is the kernel snapshot). The kernel-trustworthy anchor is the uid; group membership is expanded from the declarative table keyed on that uid.

5. Export contract (Nix → JSON)

The export tool projects the authored Nix into the JSON the broker loads. Everything not listed here is a straight pass-through of the §2/§3 fields; the export tool's non-obvious jobs are:

  1. Resolve every symbolic user:/group: name to a numeric uid/gid, and emit the numeric rules plus the names table (for name(num) logging, §4).
  2. Emit memberships: each uid's full declared group set (primary + supplementary), from users.users.<n> / users.groups.<n> (§4).
  3. Derive read/use/mint rules from each catalog destination's owner (the uid that may read/use the key), with the role chosen by class (asymmetric → signer/minter, symmetric → encrypter, value → reader, public → covered by §3.5). A destination group becomes an optional group: grant for the same role.
  4. Derive the operator rule from an explicit operators declaration (e.g. xyz.secrets.operators = [ "secret-operators" ]), never from a key's owner: a service that reads a key must not thereby gain write authority (PLP).
  5. Drop file mode: a /run/secrets filesystem concept; the broker has no files.
  6. Rename legacy UPPER_SNAKE catalog keys to dotted-lowercase.

The broker loads the result once, validates it (unknown name, ** not last, intra-segment glob, non-root * target, generate without recipe on value/public, missing nats_type on an ed25519-nkey key, duplicate key, unknown backend/role/engine reference, blank description → hard errors), and treats it as immutable for the process lifetime.

6. Rust-type sketch (contract for vault-flk)

Names the implementer creates; all serde-derived, loaded from the exported JSON. No code is written by this task; this is the contract.

// ---- Catalog ------------------------------------------------------------
pub struct Catalog {
    pub schema_version: u32,
    pub backends: BTreeMap<String, BackendRef>,
    pub keys: BTreeMap<String, KeyEntry>,        // dotted-lowercase name -> entry
}
pub struct BackendRef {
    pub kind: BackendKind,
    pub addr: String,
    pub engines: Vec<Engine>,
    pub capabilities: Vec<Capability>,
    pub requires: Vec<Capability>,
}
pub enum BackendKind { Vault /* HashiCorp Vault or OpenBao */ }

pub enum Class { Asymmetric, Symmetric, Value, Public }
pub enum Engine { Transit, Kv2, Pki }   // Pki: X.509-SVID / TLS issuer CA; never inferred (§2.2)
pub enum Capability { ByokImport, PrehashSign, PqcTransit, PkiCrl, Other(String) }

// Union of crypto algorithms across asymmetric + symmetric keys. The wire layer
// (vault-9j9) splits this by op into KeyType (asym) + AeadAlgorithm (sym);
// catalog `keyType` maps onto those. Keep the mapping explicit; do not conflate
// `Class` with `KeyAlgorithm`.
pub enum KeyAlgorithm { Ed25519, Ed25519Nkey, Rsa2048, Aes256Gcm, ChaCha20Poly1305 }

pub struct KeyEntry {
    pub class: Class,
    pub key_type: Option<KeyAlgorithm>,          // required for asym/sym; None for value; optional for public
    pub backend: String,                         // -> Catalog.backends key
    pub engine: Option<Engine>,                  // inferred from class when None
    pub path: String,
    pub writable: bool,
    pub missing: MissingPolicy,                  // default Error
    pub generate: Option<GenerateSpec>,
    pub labels: Labels,                          // "name=value" | "slug"
    pub description: String,                      // validated non-empty
}
pub enum MissingPolicy { Error, Warn, Generate }
pub enum GenerateSpec {
    AsciiPrintable { bytes: u32 },
    Base64 { bytes: u32 },
    Hex { bytes: u32 },
    AgeX25519,
    SelfSignedTls { common_name: String, validity: String },        // test/dev, feature-gated
    SelfSignedTlsPairOf { pair_of: String },                        // test/dev
}

// Parsed labels; the reserved nats_type accessor returns the single NKey prefix letter.
pub struct Labels(Vec<String>);
impl Labels {
    pub fn get(&self, key: &str) -> Option<&str>;   // "name=value" lookup
    pub fn nats_type(&self) -> Option<char>;        // 'A'|'O'|'U'|'N'|'C'|'X' (§2.6)
}

// ---- Policy -------------------------------------------------------------
pub struct Policy {
    pub roles: BTreeMap<String, BTreeSet<Op>>,
    pub rules: Vec<Rule>,
    pub config: Config,
}
pub struct Rule {
    pub id: String,
    pub principal: Vec<Principal>,               // parsed from "user:"/"group:"/"*"
    pub action: Vec<ActionTerm>,                 // parsed from "role:"/"op:"/"*"
    pub target: Vec<KeyGlob>,                     // bare key/glob, or AnyKey ("*")
    pub comment: Option<String>,
}
pub enum Principal { User(u32) /* uid */, Group(u32) /* gid */, Any }
pub enum ActionTerm { Role(String), Op(Op), AnyOp }
pub enum Op { Sign, Verify, Get, List, GetPublicKey, Validate, Mint,
              Encrypt, Decrypt, Set, Rotate, Import, NewKey }
impl Op { pub fn is_write(self) -> bool; /* Set|Rotate|Import|NewKey */ }

// ---- Glob ---------------------------------------------------------------
pub enum KeyGlob { AnyKey, Pattern(Vec<GlobSeg>) }     // AnyKey = "*", root-only at validation
pub enum GlobSeg { Literal(String), Star, DoubleStar } // Star/DoubleStar only in tail
impl KeyGlob {
    pub fn parse(s: &str) -> Result<Self, GlobError>;  // rejects mid-glob **, intra-segment *
    pub fn matches(&self, dotted_key: &str) -> bool;   // §3.4 semantics (** = one-or-more)
}

// ---- Resolution / config (export-resolved) ------------------------------
pub struct Config {
    pub names: NameTable,                        // uid->name, gid->name (logging: name(num))
    pub memberships: BTreeMap<u32, BTreeSet<u32>>,// uid -> full declared gid set
}
pub struct NameTable { pub users: BTreeMap<u32, String>, pub groups: BTreeMap<u32, String> }

// resolved, ready-for-PDP form (built by the loader; consumed by vault-1l8)
pub struct ResolvedPolicy {
    pub by_uid: BTreeMap<u32, Vec<(Op, KeyGlob)>>,
    pub by_gid: BTreeMap<u32, Vec<(Op, KeyGlob)>>,
    pub any_principal: Vec<(Op, KeyGlob)>,       // from "*" principal rules
}

vault-1l8 consumes ResolvedPolicy + Catalog + Config and implements the (uid, gid_set, Op, &dotted_key) -> Allow|Deny decision per §3/§4. vault-jso reads Catalog.keys[k].{backend,engine,path} for routing; the minter reads labels.nats_type(); vault-zrg reads {missing,generate,key_type} for reconcile.

7. Decisions

  • Key naming: dotted-lowercase, no environment prefix (one broker = one environment). The export tool renames legacy UPPER_SNAKE keys.
  • Classes (4): asymmetric / symmetric / value / public, encoding crypto shape and whether a public half exists. No credential class (a read credential is a value; nats_creds is an API mint output). No public:true flag, since public is a class, and such keys are world-readable for reads by definition (no public_read stanza).
  • keyType: KeyAlgorithm union {ed25519, ed25519-nkey, rsa-2048, ecdsa-p256, aes-256-gcm, chacha20-poly1305}; required for crypto classes, omitted for value, optional for public. No opaque.
  • engine: optional, inferred from class for the Vault-compatible backend kind; pki is explicit-only (never inferred), for an X.509-SVID issuer CA or broker TLS certificate issuer (§2.2).
  • writable: kept, as a catalog-level hard cap; false = externally managed, replaced out-of-band; both writable and the operator role must allow a write.
  • missing: error | warn | generate. Generators: ascii-printable, base64, hex, age-x25519, self-signed-tls(+pair-of) (step-cli, test/dev).
  • labels: optional name=value/slug list; reserved nats_type=<letter> (single NKey prefix letter) drives the prefix on iss; open for future policy use.
  • description: required, non-empty.
  • mint: its own capability category, separate grant from sign, neither implies the other (least privilege). Writes are the operator role, never implied.
  • validate: its own separately-grantable capability (the validator role), gated on the issuer key; verifies a presented JWT-SVID against the JWKS using public material only, for the SPIFFE Workload API ValidateJWTSVID (§3.1.2).
  • Rules: singular fields principal/action/target (+ comment), prefix/URI-form flat lists (user:/group:/role:/op:), bare-key targets, strict prefix validation.
  • Globs: last-position only; * = one segment, ** = one-or-more tail segments, web* = fatal, whole-term * = any (root-only for targets).
  • Resolution authority: the export tool resolves names → numbers (avoids drift); the broker evaluates on numbers and logs name(num) via the emitted names table.
  • Supplementary groups: declared in the JSON export (memberships), expanded from the trustworthy uid; not read from /etc/group.
  • Catalog and policy: separate documents.
