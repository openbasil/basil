Subject-Based Authentication and Signed Requests

A design for moving Basil authorization from Unix-only uid/gid principals to named subjects authenticated by typed proofs, while keeping the request authorization path simple, default-deny, auditable, and ready for signed NATS transports, executable identity, cgroups, and X.509 proof-of-possession.

1. Summary

Today Basil authorizes requests by the Unix uid/gid captured from SO_PEERCRED on the Unix socket. That model is excellent for host-local workloads, but it does not cover messages relayed over NATS, remote gRPC bridges, X.509 proof-of-possession, container identity, or executable identity without making the policy expression language grow a new syntax branch for every authenticator.

This design introduces a stable policy concept: rules grant to named subjects. A subject is resolved from one or more typed principals, where a principal is a configured proof or local identity selector such as a Unix uid/gid, process hash, NATS public NKey, Ed25519 public key, SSH Ed25519 public key, X.509 SAN under a trust bundle, or cgroup fact.

Core decision: replace rule-level principal: "user:9001 || group:10" expressions with subjects: ["svc.web", "ops.wheel"]. Rules OR their listed subjects. The old expression parser is no longer needed because allOf and anyOf carry the AND/OR logic in the subject registry, where it is structured, typed, validated by Serde, and extensible without redesigning the rule parser.

2. Goals and non-goals

Goals

  • Keep Basil default-deny: no request is authorized until a subject is resolved and a policy rule grants the operation.
  • Keep runtime inputs at four files: agent config, catalog, policy, and sealed bundle. Subject definitions live in the policy document.
  • Separate the transport peer from the authorized subject so bridges cannot become confused deputies.
  • Support the immediate NATS use case with signed and encrypted request envelopes and signed and encrypted responses.
  • Leave clean extension points for executable identity, cgroups, Kubernetes pods, and X.509 proof-of-possession.
  • Prefer simple rule APIs over a complex string language. Use structured schemas for typed identity evidence.
  • Make audit logs tell the full story: subject, proof, presenter, transport, policy generation, decision, and target.

Non-goals

  • Do not build a general OAuth2 or OpenID Connect authorization server.
  • Do not introduce bearer tokens as a replacement for signed proof-of-possession.
  • Do not require a NATS-specific identity model. NATS is one proof kind, not the policy model.
  • Do not add a fifth runtime configuration file.
  • Do not implement all future authenticators now. The first implementation should cover Unix subjects and sealed request subjects.

3. Terminology

Subject
The authenticated actor that Basil authorizes. Policy rules grant to subjects by stable names such as content.publisher, svc.web, or support.admin.
Principal
One configured identity selector or proof that can establish a subject: Unix uid/gid, process digest, public signing key, X.509 identity, or cgroup fact.
Presenter
The entity that delivered the request to Basil. For a local Unix socket call, this is the process on the other end of the socket. For a NATS bridge, this is the local bridge process, not necessarily the actor whose request it carries.
Transport peer
The low-level peer facts captured from transport, such as SO_PEERCRED uid/gid/pid, process path, executable digest, cgroup, or TLS peer certificate. These facts are evidence, not automatic authority.
Actor
The subject selected for authorization after evaluating all presented evidence. Most code should use AuthenticatedActor rather than raw uid/gid arguments.

4. Security model

4.1 Security principles

  • Authenticate before authorize. Policy never evaluates an unauthenticated string supplied by the request.
  • The subject gets the grant. The presenter is evidence and audit context; it does not inherit the subject's rights.
  • Bridges are couriers. A NATS or gRPC bridge forwards an envelope; it does not assert identity and does not need the sender's private key.
  • Bind signatures to intent. Signed proofs cover the message type, target key, ciphertext, sender subject, recipient subject, nonce, and expiry.
  • Encrypt bridged payloads by default. Request and response bodies are sealed to the recipient. A bridge should not see key material in either direction.
  • Fail closed on missing evidence. A missing executable hash, unreadable cgroup, expired certificate, unknown signer, or replayed nonce resolves no subject.
  • Prefer proof-of-possession over bearer credentials. A stolen message body without the signing key must be useless.
  • Keep public trust material out of the sealed bundle. Subject public keys and trust roots belong in policy; backend bootstrap secrets remain in the bundle.
  • Keep private broker signing material out of policy. Policy may name the broker identity key, but private material is in the backend or unlocked runtime state.

4.2 Threat model

Threat Required defense
Bridge sets subject=content.publisher without a valid sender signature. Subject resolution ignores bridge claims. The named subject resolves only when the sealed invocation carries a valid proof from a configured principal for that subject.
Attacker replays a sealed request from NATS or a log. Signed envelopes carry recipient subject, message id, issued time, and expiry. Basil stores a bounded replay cache per signer or subject.
Bridge changes the target key or operation after receiving a signed message. Signature covers canonical message type, ciphertext, and target key fields.
Transport security is absent or untrusted. Integrity, identity, and payload confidentiality are supplied by the invocation envelope. Operators may still use TLS as defense in depth, but the bridge must not need TLS to avoid seeing secret request or response bodies.
A request or response carries key material, such as import-key input or get-secret output. The invocation protocol seals every request body to Basil and every response body to the caller's response recipient key id with a unique nonce. There is no per-message opt-in switch to forget.
Process path is spoofed or replaced. Executable identity uses digest as the primary signal. Path alone is advisory unless paired with digest.
Cgroup or Kubernetes label is mutable or ambiguous. Selectors prefer stable facts such as namespace, service account, pod UID, container ID, and validated projected token claims. Labels are optional and treated as lower-trust constraints.
Certificate is valid but used for the wrong request. X.509 support is proof-of-possession: the private key signs the Basil request envelope. Chain validation alone is not enough.
Response is forged by the bridge. Basil signs response envelopes with a configured broker identity key. Clients verify before trusting results.

4.3 Rules of thumb

  • If the evidence is typed, keep it out of a string parser.
  • If a request is bridged, record the bridge as presenter in audit, but authorize only the subject proved by the envelope.
  • If a proof can cross a network, it needs a recipient subject, message id, timestamp, and expiry.
  • If a proof depends on local machine state, it must define what happens when that state cannot be read. The answer is always deny.
  • If two workloads share a subject, they share policy authority. Subject names should be operationally meaningful and shared only intentionally.
  • If a proof kind is not fully understood by the current Basil build, the loader rejects it rather than preserving it as opaque.

5. Policy schema

Subject definitions live in policy.json. They are authorization trust material and must reload atomically with rules. The catalog remains the resource inventory. The sealed bundle remains secret bootstrap material. The agent TOML remains process wiring.

5.1 Policy document shape

{
  "schemaVersion": 2,
  "subjects": {
    "svc.web": {
      "allOf": [
        { "kind": "unix", "uid": 9001 }
      ]
    },
    "ops.wheel": {
      "allOf": [
        { "kind": "unix", "gid": 10 }
      ]
    },
    "content.publisher": {
      "anyOf": [
        {
          "kind": "signature-key",
          "algorithm": "nats-nkey",
          "public": "U..."
        },
        {
          "kind": "signature-key",
          "algorithm": "ed25519",
          "public": "base64url..."
        }
      ]
    }
  },
  "roles": {
    "reader": ["get", "list", "get_public_key"],
    "signer": ["sign", "verify", "get_public_key"]
  },
  "rules": [
    {
      "id": "web-reads-web-keys",
      "subjects": ["svc.web"],
      "action": ["role:reader"],
      "target": ["web.**"]
    },
    {
      "id": "publisher-signs-feed",
      "subjects": ["content.publisher"],
      "action": ["role:signer"],
      "target": ["content.feed.signing_key"]
    }
  ],
  "config": {
    "names": {
      "users": { "9001": "svc-web" },
      "groups": { "10": "wheel" }
    },
    "memberships": { "9001": [9001] }
  }
}

5.2 Rule shape

A rule grants to one or more subject names. The list is an OR. There is no inline boolean expression in rules. 

{
  "id": "support-admin-can-reload",
  "subjects": ["support.admin", "breakglass.root"],
  "action": ["op:reload"],
  "target": ["broker.reload"]
}

The loader rejects a rule that references an undefined subject. It also rejects empty subject lists. A convenience singular field subject may be accepted by source tooling, but the exported JSON should use subjects for one canonical runtime shape.

5.3 Subject expression shape

A subject definition has exactly one top-level composition key: allOf or anyOf. Each list item is a principal spec. There is no extra principal wrapper, and the first implementation should not support nested boolean expressions. More complex OR cases can be expressed by defining multiple subjects and listing them together on a rule. Negation is deliberately omitted for the first version. Most useful policy can be expressed positively, and negation over missing evidence is easy to misunderstand. 

{
  "breakGlass": false,
  "allOf": [
    { "kind": "unix", "uid": 9001 },
    {
      "kind": "process",
      "exePath": "/nix/store/.../bin/web-api",
      "sha256": "hex..."
    }
  ]
}

breakGlass is optional and defaults to false. It is not a grant. It only marks a subject as eligible to appear in rules whose target is the global wildcard *. The loader rejects target: ["*"] unless every subject named by that rule has breakGlass: true.

5.4 Break-glass wildcard target

A global wildcard target is deliberately harder to express than ordinary key globs. The subject must be marked in the subject registry, and the rule must still explicitly grant the desired actions. The marker exists so code can enforce operator intent; comments are not sufficient. 

{
  "subjects": {
    "breakglass.root": {
      "breakGlass": true,
      "allOf": [
        { "kind": "unix", "uid": 0 }
      ]
    }
  },
  "rules": [
    {
      "id": "root-breakglass",
      "subjects": ["breakglass.root"],
      "action": ["*"],
      "target": ["*"],
      "comment": "Emergency operator path; keep out of normal service policy."
    }
  ]
}

5.5 Principal kinds

Kind Status Purpose
unix first implementation Matches the SO_PEERCRED uid and/or the export-resolved group set for that uid. A gid match uses the full configured group membership, not only the primary gid.
unauthenticated first implementation Matches only when no actor proof is present and policy explicitly configured unauthenticatedSubject to this subject. Recommended subject name: guest.
signature-key first implementation Verifies a signed Basil request envelope with a configured public key. Algorithms: ed25519, nats-nkey, and later ssh-ed25519.
process roadmap Matches executable digest and optionally path. Digest is primary; path is advisory or an additional constraint.
cgroup roadmap Matches cgroup/container facts for Docker or systemd scopes.
kubernetes roadmap Matches namespace, service account, pod UID, and projected service-account token claims after issuer/audience validation.
x509-pop roadmap Validates certificate chain and SAN constraints, then requires private-key proof over the Basil request envelope.

5.6 NATS bridge as a courier

The NATS bridge does not assert identity. It forwards an invocation envelope whose encrypted body and sender signature are verified by basil-agent. The bridge can be authenticated as a local Unix process for audit and operational controls, but that presenter identity is not the subject being authorized. 

{
  "subjects": {
    "transport.nats-bridge": {
      "allOf": [
        { "kind": "unix", "uid": 9100 }
      ]
    },
    "content.publisher": {
      "anyOf": [
        {
          "kind": "signature-key",
          "algorithm": "nats-nkey",
          "public": "U..."
        }
      ]
    }
  },
  "rules": [
    {
      "id": "nats-bridge-can-submit-invocations",
      "subjects": ["transport.nats-bridge"],
      "action": ["op:invoke"],
      "target": ["broker.invocation"]
    }
  ]
}

The first implementation does not have systemd service authentication, so the bridge presenter subject resolves from Unix uid/gid evidence. A dedicated op:invoke grant over broker.invocation lets that presenter submit sealed envelopes without granting data-plane authority. Basil still authorizes the actor proved by the sealed request for the actual operation and target. 

[nats]
url = "nats://127.0.0.1:4222"
creds = "/run/credentials/basil-nats-bridge/nats.creds"

[basil]
socket = "/run/basil/agent.sock"

[bridge]
request-subject = "basil.invocation.v1"
queue-group = "basil-nats-bridge"
max-message-bytes = 1048576

The NATS request payload is protobuf-encoded SecureMessage; the reply payload is protobuf-encoded response SecureMessage. NATS request/reply inboxes provide transport correlation, while the signed response header still binds request_id to the request message_id. If the bridge cannot obtain a sealed Basil response, it replies with an empty payload and NATS headers: Basil-Bridge-Error, Basil-Bridge-Message, and Basil-Bridge-Retryable. Stable error tokens include MALFORMED_REQUEST, MESSAGE_TOO_LARGE, BASIL_UNAVAILABLE, BASIL_REJECTED, TIMEOUT, and INTERNAL. If Basil returns a sealed response, the bridge forwards it unchanged and does not add bridge error headers.

A future delegation feature may allow one subject to delegate to another, but that is a separate capability and should not be confused with the NATS bridge path. The bridge path is simpler: no private sender key, no impersonation.

6. Subject resolution

Authorization should run over an AuthenticatedActor, not a uid. The actor carries the selected subject plus the evidence chain that established it. 

struct AuthenticatedActor {
    subject: SubjectName,
    authenticated_by: Vec<ProofSummary>,
    presenter: PresenterInfo,
    transport: TransportInfo,
}

6.1 Resolution order

  1. Collect transport evidence from request extensions: Unix uid/gid/pid, executable path/digest when configured, cgroup facts when configured, TLS peer certificate when available. The pid is informational and logged when available; it is not used for authentication.
  2. Parse request-level authentication evidence: sealed envelope, requested subject, sender signature, certificate proof, nonce, and timestamps.
  3. Resolve the presenter subject, if any, from transport evidence.
  4. Resolve the actor subject from direct proof in the envelope.
  5. Reject ambiguous actor resolution unless the request explicitly names one subject and that subject resolves.
  6. Run the PDP on the resolved actor subject.

6.2 Local Unix compatibility

Existing local calls can remain ergonomic. A request with no explicit auth evidence resolves exactly one subject from the presenter's Unix facts. The Nix exporter should generate subjects for configured users and groups. 

"subjects": {
  "unix.user.svc-web": {
    "allOf": [
      { "kind": "unix", "uid": 9001 }
    ]
  },
  "unix.group.wheel": {
    "allOf": [
      { "kind": "unix", "gid": 10 }
    ]
  }
}

6.3 Unauthenticated subject

Basil may support a NATS-style unauthenticated subject mapping for public or low-risk operations such as health checks or fetching public identity material. This is disabled by default. When enabled, the operator names the subject; guest is the recommended conventional name, not a hard-coded rule. 

{
  "unauthenticatedSubject": "guest",
  "subjects": {
    "guest": {
      "anyOf": [
        { "kind": "unauthenticated" }
      ]
    }
  },
  "rules": [
    {
      "id": "guest-can-read-public-identities",
      "subjects": ["guest"],
      "action": ["op:get_public_key"],
      "target": ["identity.public.**"]
    }
  ]
}

The loader rejects kind: "unauthenticated" unless unauthenticatedSubject names that same subject. This keeps anonymous access explicit and easy to audit.

6.4 Authorization API

The PDP should move from decide(uid, op, key) to decide(actor, op, key). The policy matching step becomes a lookup over subject names rather than a uid/gid predicate expression. 

impl Pdp {
    fn decide(&self, actor: &AuthenticatedActor, op: Op, key: &str) -> Decision;
    fn decide_admin(&self, actor: &AuthenticatedActor, op: Op) -> Decision;
}

Audit records should carry both the actor and the presenter: 

{
  "actor_kind": "subject",
  "actor_id": "content.publisher",
  "authenticated_by": "signature-key:nats-nkey:sha256:...",
  "presenter_kind": "unix_peercred",
  "presenter_id": "svc-nats-bridge(9100)",
  "outcome": "allow"
}

7. Request and response protection

Superseded by COSE: the protobuf Header/SecureMessage/MessageType envelope in this section is historical. The implemented sealed invocation path is the COSE design in designs/basil-cose-api.md and designs/cose-sealed-invocation.html: callers send raw tagged COSE_Sign1 bytes, the broker opens the embedded tagged COSE_Encrypt, and responses bind the complete request COSE bytes with the -70002 SHA3-256 claim.

7.1 Sealed invocation envelope

The immediate signed-message use case should use a Basil request envelope, not a NATS-specific authorization grammar. The envelope can be carried over NATS, gRPC, a file queue, or any future transport. For bridged transport, the body is always encrypted to Basil; encryption is not an optional flag. 

enum MessageType {
  MESSAGE_TYPE_UNSPECIFIED = 0;
  MINT_JWT_REQUEST = 1;
  MINT_NATS_USER_REQUEST = 2;
  SIGN_REQUEST = 3;
  SIGN_RESPONSE = 4;
  MINT_JWT_RESPONSE = 5;
  MINT_NATS_USER_RESPONSE = 6;
}

message Header {
  uint32 version = 1;
  string message_id = 2;
  optional string request_id = 3;
  MessageType message_type = 4;
  uint32 issued_at_unix = 5;
  optional uint32 expires_at_unix = 6;
  string sender_sign_id = 7;
  optional string sender_subject = 8;
  string recipient_key_id = 9;
  optional string recipient_subject = 10;
  optional string response_encryption_key_id = 11;
  bytes encapsulated_key = 12;
  bytes nonce = 13;
}

message SecureMessage {
  Header header = 1;
  bytes body = 2;
  bytes signature = 3;
}

The runtime wire shape is a protobuf SecureMessage, not JSON. Protobuf bytes fields are length-delimited raw bytes on the gRPC wire; they are only base64-encoded by protobuf JSON mappings or human-readable debug renderings. No base64 conversion is part of the Basil wire protocol.

  • version is the envelope format version. v1 is the only supported value. Example: 1.
  • message_id is a sender-generated message id, unique for that sender within the replay window for every envelope, request or response. A UUIDv7 works, but so does a durable monotonic counter. Request callers generate request message ids; Basil generates response message ids. Example: uuid7-or-persisted-counter.
  • request_id correlates a response to the request it answers. It is absent on first-leg requests and present on responses, where it equals the request envelope's message_id.
  • message_type is the complete payload discriminator. Each request and response body type gets a stable enum value such as MINT_JWT_REQUEST, MINT_NATS_USER_REQUEST, or SIGN_RESPONSE. No separate body_type, method string, or protobuf type URL is carried in v1.
  • issued_at_unix is the signed Unix issue time. Basil checks it against the configured clock-skew allowance. Example: 1782740000.
  • expires_at_unix is the optional signed expiry. It may be omitted, in which case its effective value is issued_at_unix + 60 seconds. Basil still enforces the configured maximum TTL.
  • sender_sign_id names the configured signing key record that signs the transcript digest. Encryption and signature algorithms are resolved from configured key records, not from caller-supplied algorithm fields. Example: publisher.signing.2026q3.
  • sender_subject is the subject asserted by the signer of this envelope. For requests, it is the authorization principal evaluated by policy; for responses, it is the broker identity subject. It may be omitted when sender_sign_id resolves to exactly one subject in policy; if the signer can prove multiple subjects, the message must name the intended subject and the signature must cover it. Example: content.publisher.
  • recipient_key_id names the configured recipient encryption key the body is sealed to. For requests this is the broker request-encryption key; for responses it is the caller's response encryption key. If using PQC encryption, the message carries per-message KEM encapsulation output, not recipient public keys. Callers resolve this catalog key id to key type and public key bytes before encryption; Section 7.5 describes the existing lookup APIs. Example: broker.invocation_encryption.2026q3.
  • recipient_subject is the intended recipient identity bound into the signature. For requests, it is the broker identity or configured recipient subject. For responses, it is the caller or response recipient identity. It may be omitted when recipient_key_id resolves to exactly one recipient subject; otherwise it is required. If present, Basil verifies it matches the registry. Example: basil://prod/us-east-1/agent-a.
  • response_encryption_key_id is present only on requests. It names the requester-controlled response recipient key Basil must use when encrypting the response. The agent validates that the key exists in the backend, has the expected response-encryption use, and has not been revoked before executing the operation. Responses omit this field; their recipient_key_id is the request's response_encryption_key_id.
  • encapsulated_key is the raw KEM encapsulation output for the per-message body key (if using PQC encryption). On the protobuf wire this is binary bytes, not base64 text.
  • nonce is the raw AEAD nonce for the encrypted body. On the protobuf wire this is binary bytes, not base64 text.
  • body is the encrypted CBOR body bytes. On the protobuf wire this is binary bytes, not base64 text.
  • signature is the raw signature over the BLAKE3 canonical transcript digest. On the protobuf wire this is binary bytes, not base64 text.

The plaintext message body is serialized as CBOR before encryption. MessageType selects the CBOR schema for the body, and every message type must define that schema in one registry. Implementations use the same deterministic CBOR profile for maps, integers, byte strings, text strings, arrays, booleans, and nulls. Unknown fields are rejected in v1 rather than ignored, so client implementations fail closed when their schema registries diverge. M2 defines the CBOR structs in Rust first and adds every request and response body to the MessageType enum as the Rust implementation needs them. M2 and M3 use Rust-generated fixtures to debug the protocol quickly; Go implementation is deferred until after M4's Rust bridge is stable. Interoperability tests cover both the CBOR plaintext bytes and the encrypted SecureMessage.body bytes.

Response bodies are operation-specific on the wire but share common declared fields: status and policy_generation. Rust and Go helper APIs may expose this as an ergonomic InvocationResponse<T> wrapper or a common status trait, but the stable CBOR registry still declares each response body explicitly. 

enum InvocationStatusCode {
  OK = 1;
  DENIED = 2;
  INVALID_REQUEST = 3;
  INTERNAL_ERROR = 4;
}

struct InvocationStatus {
  1: code,
  2: reason,
  3: message?,    # sanitized, optional
  4: retryable
}

struct SignInvocationResponse {
  1: status,
  2: policy_generation,
  3: signature
}

struct MintJwtInvocationResponse {
  1: status,
  2: policy_generation,
  3: jwt,
  4: expires_at_unix?
}

struct MintNatsUserInvocationResponse {
  1: status,
  2: policy_generation,
  3: jwt,
  4: expires_at_unix?
}

Request binding is not a separate body field. Implementations compute the binding from the canonical request fields and transcript described at the end of this section.

The signature input is the 32-byte unkeyed BLAKE3 digest of a canonical transcript. BLAKE3 uses its normal parameters. The transcript starts with the fixed domain-separation string basil-secure-message-v1, then includes every field in Header in protobuf field-number order, then the encrypted body. This deterministic transcript encoding is separate from the protobuf wire encoding used by gRPC; protobuf still sends enum and integer fields using its normal numeric wire format. For the transcript only, integers and enum values are encoded as u32 little-endian. Optional fields are encoded as a single u8 presence flag (0 or 1) followed by the value when present. Strings are encoded as u32 little-endian byte length followed by UTF-8 bytes. Byte fields are encoded as u32 little-endian length followed by raw bytes. The authenticated encryption associated data is the same canonical header transcript, without body, so a bridge cannot transplant ciphertext into a different message type, sender, recipient, key id, nonce, request id, or expiry.

7.2 Payload confidentiality

Every bridged invocation body is sealed, including apparently harmless operations. This avoids requiring every future API author to remember which messages carry secrets. It also protects sensitive metadata that can appear in requests and responses over time.

  • Requests are encrypted to the broker's configured request-encryption public key.
  • Requests name a response_encryption_key_id in the signed header; responses are encrypted to that validated requester key.
  • Every sealed message, request or response, uses a fresh nonce and, if using PQC encryption, a fresh KEM encapsulation. A response never reuses the request's encapsulated_key (if using PQC encryption) or nonce.
  • The bridge may see envelope routing fields needed to deliver the message, but not serialized broker request or response bodies.
  • Local typed gRPC over Unix sockets may continue to send plaintext protobuf on that local connection; this sealed envelope is the required shape for bridged or remote invocation.

7.3 Replay protection

  • Every sealed request must include a message_id that is unique for the sender within the replay window.
  • Every sealed request must include a short expiry. A default maximum of 60 seconds is appropriate for message bridges.
  • Basil keeps a bounded replay cache keyed by signer fingerprint plus message_id until the expiry plus skew allowance.
  • Clock skew should be explicit and small. A default of 30 seconds is enough for same-fleet systems.

7.4 Broker response protection

For async or bridged transports, callers need to know that a response came from Basil and corresponds to the exact request. Basil signs response envelopes with a configured broker identity key and encrypts the response body to the response recipient key id named by response_encryption_key_id on the request. The response header sets request_id to the request envelope's message_id. The encrypted response body schema is selected by MessageType. Request binding is computed from the canonical request fields and transcript rather than carried as a trusted request_hash field.

Denials and errors should also be signed. Otherwise a bridge can hide Basil's decision or forge failure modes. Operation status and other trusted response metadata live inside the encrypted response body. The clear gRPC status is transport status only; it says whether a sealed response was returned, not whether the broker operation succeeded.

If Basil cannot construct a signed and encrypted response after accepting a request, it fails closed at the transport level with a generic INVOCATION_RESPONSE_PROTECTION_FAILED reason and no operation result details. It must audit the failure with the trusted context available at that point and never fall back to an unsigned or plaintext response.

7.5 Broker identity and encryption keys

Broker response-signing and request-encryption keys are configured by catalog key id. Private key material lives in the backend; the catalog holds metadata, key type, and backend location. Policy can trust public material, but it must not contain the broker's private signing material. 

[broker-identity]
id = "basil://prod/us-east-1/agent-a"
response-signing-key-id = "broker.response_signing.2026q3"

[invocation]
enable = true
audience = ["basil://prod/us-east-1/agent-a"]
request-encryption-key-id = "broker.request_encryption.2026q3"
max-ttl-secs = 60
clock-skew-secs = 30
replay-cache-capacity = 4096

Response signing is always required whenever invocation is enabled; there is no response-signing = "disabled" mode. Response encryption is also mandatory, but the recipient key is selected per request through response_encryption_key_id because Basil does not know the remote requester set at startup.

Header key ids are Basil catalog names, not self-describing algorithm handles. A caller that builds a sealed message must know the recipient catalog key's public bytes and key type before encryption, because the key type selects the public-key encoding and encryption suite. Basil already has forward lookup APIs for callers that are connected over an authenticated local or otherwise trusted channel: 

service SecretService {
  rpc ListCatalog(ListCatalogRequest) returns (stream CatalogEntry);
}

message CatalogEntry {
  string name = 1;
  CatalogKind kind = 2;
  optional KeyType key_type = 3;
  uint32 latest_version = 4;
}

service SigningService {
  rpc GetPublicKey(GetPublicKeyRequest) returns (GetPublicKeyResponse);
}

message GetPublicKeyRequest {
  string key_id = 1;
  optional uint32 version = 2;
}

message GetPublicKeyResponse {
  string key_id = 1;
  KeyType key_type = 2;
  bytes public_key = 3;
  uint32 version = 4;
}

ListCatalog lets an authorized local caller discover visible key names, key classes, key types, and latest versions. GetPublicKey returns the public bytes and KeyType for a chosen catalog key. These RPCs are not a trust bootstrap mechanism over an untrusted bridge or network: a man-in-the-middle could substitute its own recipient key unless the response is authenticated by a broker signing key the caller already trusts.

For bridged or direct remote invocation, the caller obtains broker public keys out of band. M3 does not add a JWKS, status, or public discovery surface for invocation keys. Clients pin at least the broker id, response signing public key, request-encryption public key, accepted key ids, and key type/version metadata before using a bridge. No reverse describe_key(public_key) API is required for sealed invocation; if a caller only has public bytes, it still needs an authenticated catalog key id to put in recipient_key_id.

8. gRPC API

Superseded by COSE: the implemented InvocationService.Invoke signature is Invoke(SealedRequest) returns (SealedResponse). The gRPC carrier contains opaque tagged COSE bytes in message plus an optional clear response_subject on responses for courier routing; it no longer accepts or returns protobuf SecureMessage values.

8.1 Recommended shape

Do not add authentication fields to every existing broker request yet. That is noisy and still leaves non-gRPC transports without a native shape. Add one transport-neutral invocation service for sealed bridged calls, while local typed gRPC services continue to use Unix peer credentials. The service is compiled into the agent but disabled by default at runtime. 

service InvocationService {
  rpc Invoke(SecureMessage) returns (SecureMessage);
}

Section 7.1 defines Header and SecureMessage. The request argument is a sealed request message. The return value is a sealed response message whose header carries a new sender-unique message_id and a request_id pointing back to the request. The gRPC status is only transport status: disabled service, malformed envelope, oversized payload, or inability to construct a sealed response. Broker operation success, denial, and failure status live inside the encrypted response body.

Rust and Go client libraries should hide InvocationService behind the same typed API methods. Direct local clients call typed gRPC services. Remote or NATS clients build sealed invocation envelopes.

8.2 Why not metadata-only auth?

gRPC metadata works for direct gRPC, but it is a poor universal envelope for NATS messages and other async transports. A sealed invocation gives one canonical object for queues, bridges, logs, replay protection, response signing, and response encryption. Metadata can still be added later as an optimization for direct remote gRPC if there is a real need. This does not refer to cgroup or Kubernetes evidence; those are principal kinds collected by authenticators. A metadata proof would be a direct gRPC-only signature/certificate proof attached to a normal RPC. That path is deferred.

8.3 Breaking admin explain API

ExplainRequest currently evaluates a uid. It should evaluate a subject instead. Offline explain should also accept a subject name and a mock evidence bundle for subject-resolution tests. 

message ExplainRequest {
  string subject = 1;
  string op = 2;
  string key = 3;
}

message ExplainResponse {
  string subject = 1;
  string op = 2;
  string key = 3;
  string decision = 4;
  MatchedRule matched = 5;
}

To keep common host-local policy concise, the module may generate Unix subjects from a source-level shortcut:

  "ops.wheel".group = "wheel";
};

The exporter resolves names to numeric uid/gid, writes those numeric values into policy.json, and preserves the name mapping in config.names for audit readability.

9.4 Broker identity API

services.basil.brokerIdentity = {
  id = "basil://prod/us-east-1/agent-a";
  responseSigningKeyId = "broker.response_signing.2026q3";
};
services.basil.invocation = {
  enable = false; # secure default
  audience = [ "basil://prod/us-east-1/agent-a" ];
  requestEncryptionKeyId = "broker.request_encryption.2026q3";
  maxTtlSeconds = 60;
  clockSkewSeconds = 30;
  replayCacheCapacity = 4096;
};

responseSigningKeyId names a backend-custodied catalog signing key used by the broker itself for response signatures. requestEncryptionKeyId names the public recipient key clients use to seal bridged invocation bodies to Basil. These are not granted through data-plane policy; they are agent identity and invocation transport settings. The loader must validate that the keys exist and have the required signing or encryption class.

The TOML agent config should mirror this split: broker identity keys are configured under broker identity, and the invocation service is enabled with an explicit runtime flag. This should not need a Cargo feature flag; the code-size cost is not expected to justify a separate build. 

[invocation]
enable = false

10. Implementation

10.1 Rust type sketch

struct Policy {
    schema_version: u32,
    subjects: BTreeMap<SubjectName, SubjectDefinition>,
    roles: BTreeMap<String, Vec<Op>>,
    rules: Vec<Rule>,
    config: Config,
    unauthenticated_subject: Option<SubjectName>,
}

struct Rule {
    id: String,
    subjects: Vec<SubjectName>,
    action: Vec<ActionTerm>,
    target: Vec<KeyGlob>,
    comment: Option<String>,
}

struct SubjectDefinition {
    break_glass: bool,
    match_: SubjectMatch,
}

enum SubjectMatch {
    AllOf(Vec<PrincipalSpec>),
    AnyOf(Vec<PrincipalSpec>),
}

enum PrincipalSpec {
    Unix { uid: Option<u32>, gid: Option<u32> },
    Unauthenticated,
    SignatureKey { algorithm: SignatureKeyKind, public: PublicKeySpec },
    Process { exe_path: Option<String>, sha256: String },
    Cgroup { /* future typed fields */ },
    Kubernetes { /* future typed fields */ },
    X509Pop { /* future typed fields */ },
}

10.2 Loader rules

  • Reject duplicate subject names after normalization.
  • Reject rules that reference undefined subjects.
  • Reject unsupported principal kinds and unsupported signature algorithms.
  • Reject unix principals that name neither uid nor gid.
  • Reject unauthenticated principals unless unauthenticatedSubject names the containing subject.
  • Reject subject expressions with empty allOf or anyOf.
  • Reject broker identity config that names a missing key or a key with the wrong use.
  • Reject wildcard target * unless every referenced subject has breakGlass: true.

10.3 PDP changes

Resolved policy no longer needs to preserve a parsed PeerPredicate on every rule. It can expand rule subjects into a set or sorted vector of subject names and match directly against the actor subject. 

struct ResolvedRule {
    subjects: BTreeSet<SubjectName>,
    grants: Vec<Grant>,
}

The world-readable public-class rule can remain unchanged, but audit should still require an authenticated actor. If no subject can be resolved, public reads should either be denied or resolved to a dedicated built-in anonymous.local subject. The simpler and safer first version is to deny missing subject.

10.4 Audit changes

The audit record should stop hard-coding actor_kind=unix_uid. Recommended fields: 

{
  "event_kind": "basil.audit.authz",
  "event_version": 2,
  "generation": 42,
  "op": "sign",
  "target_kind": "catalog_key",
  "target_id": "content.feed.signing_key",
  "actor_kind": "subject",
  "actor_id": "content.publisher",
  "authenticated_by": ["signature-key:nats-nkey:sha256:..."],
  "presenter_kind": "unix_peercred",
  "presenter_id": "svc-nats-bridge(9100)",
  "outcome": "allow",
  "reason": "subject_rule"
}

12. Security tests

12.1 Loader and schema tests

  • Undefined subject in a rule is a fatal load error.
  • Unsupported principal kind is a fatal load error.
  • Unsupported signature algorithm is a fatal load error.
  • Empty subjects, allOf, or anyOf is rejected.
  • unix principal with neither uid nor gid is rejected.
  • unauthenticated principal is rejected unless it is the configured unauthenticated subject.
  • Nix name resolution emits numeric uid/gid principals and preserves audit names.

12.2 PDP parity tests

  • A generated Unix subject grants the same operations as the previous user:uid rule.
  • A generated group subject uses the configured full membership set, not only primary gid.
  • A configured unauthenticated subject can reach only rules that explicitly name that subject.
  • Without unauthenticatedSubject, missing actor proof is denied even for public-class reads.
  • A denied subject does not inherit grants from the presenter.
  • Admin operations remain excluded from wildcard data-plane grants.
  • Public-class reads require a resolved subject, or the chosen anonymous behavior is tested explicitly.

12.3 Sealed request tests

  • InvocationService rejects requests while disabled in TOML config.
  • Valid Ed25519 sealed invocation resolves the configured subject and authorizes normally.
  • Valid NATS NKey sealed invocation resolves the configured subject and authorizes normally.
  • Wrong recipient subject is rejected before policy evaluation.
  • Expired request is rejected before policy evaluation.
  • Replay of the same request message_id is rejected.
  • Changing message type, key id, ciphertext, encapsulated_key (if using PQC encryption), nonce, sender subject, recipient subject, expiry, or request id breaks the signature.
  • Remote clients reject a recipient public-key descriptor unless it is configured out of band or signed by a previously trusted broker identity key.
  • Omitted expires_at_unix uses the signed default of issued_at_unix + 60 seconds.
  • Omitted sender_subject and recipient_subject are accepted only when they resolve unambiguously from sender_sign_id and recipient_key_id.
  • Each MessageType added in M2 has a Rust CBOR body struct, a documented body schema, and Rust fixtures that lock the plaintext CBOR bytes before encryption.
  • M2 and M3 Rust test vectors produce stable canonical transcript bytes, BLAKE3 transcript hashes, AAD bytes, and signatures for the same header and body bytes; deferred Go tests must later match those vectors byte-for-byte.
  • Test vectors are checked in as JSON fixtures containing named header fields, message type, structured body value, hex-encoded plaintext CBOR body, hex-encoded encrypted body, hex-encoded canonical transcript, hex-encoded AAD, expected BLAKE3 digest, signer public key, signature, and expected validation result.
  • Unknown signer and bad signature are indistinguishable at the public error level.

12.4 Bridge courier tests

  • A bridge cannot make an unsigned envelope resolve to a subject.
  • A bridge cannot change the envelope subject without breaking the sender signature.
  • A bridge cannot change the message type, target key, ciphertext, encapsulated_key (if using PQC encryption), nonce, sender subject, recipient subject, expiry, request id, or recipient key id without breaking the sender signature.
  • The bridge's own data-plane grants are not used for the sender subject.
  • Audit records include sender subject, presenter process, and signature proof.

12.5 Response protection tests

  • Client verifies a successful response against the configured broker public key.
  • Client rejects a response signed by an unknown broker key.
  • Client rejects a response with a mismatched request id or failed canonical request binding.
  • Denied responses and internal-error responses are signed.
  • Client ignores gRPC status for broker-operation results and trusts only the decrypted signed response body.
  • Bridge-visible response envelopes do not contain plaintext secret values or imported key material.
  • Client rejects a response encrypted to the wrong recipient key.
  • Policy generation in the response matches the generation that decided the request.

12.6 Future authenticator tests

  • Process digest principal fails closed when /proc/pid/exe cannot be read.
  • Process path alone is never accepted for high-trust examples unless explicitly configured.
  • Cgroup parser rejects malformed or ambiguous cgroup evidence.
  • Kubernetes token validation rejects wrong audience, wrong issuer, expired token, and forged signature.
  • X.509 proof-of-possession rejects valid certificates that do not sign the request envelope.

13. Operations

13.1 Observability

  • Count subject-resolution failures by stable reason: no subject, bad signature, unknown signer, decrypt failed, expired, replay, wrong recipient subject.
  • Count authorizations by actor subject, op, and outcome, with bounded labels.
  • Expose replay-cache size and eviction counters when metrics are enabled.
  • Log the broker identity key id at startup, never private material.

13.2 Operator guidance

  • Name subjects after operational roles or workloads, not after key formats.
  • Use direct signed proof when the real actor can sign the request.
  • Treat bridges as couriers. If a future delegation feature is added, model it as explicit delegation, not as bridge identity.
  • Distribute broker response-signing and request-encryption public trust material out of band before remote clients use a bridge.
  • Keep sealed request TTLs short. Long-lived reusable credentials are a different system design.
  • Rotate subject keys by adding a second principal, deploying clients, then removing the old principal.

13.3 Documentation impact

Once implemented, basil-doc pages that describe uid/gid policy will need a coordinated update. Until then, this document is the forward-looking design source and should not be treated as current user documentation.

14. Decisions and deferred features

Resolved decisions

  1. Unauthenticated requests are denied by default. Operators may configure unauthenticatedSubject, conventionally guest, and then grant that subject explicitly in policy.
  2. InvocationService is compiled in but disabled by default in the TOML config. Enabling it requires explicit configuration. No Cargo feature flag is proposed.
  3. Bridged invocation response signing and payload encryption are mandatory whenever InvocationService is enabled.
  4. Broker response-signing and request-encryption private keys live in the backend. Agent config names catalog key ids; the catalog carries metadata and backend location. Bridged clients obtain broker public trust material out of band.
  5. Response encryption is selected per request with response_encryption_key_id. The response omits that field and sets recipient_key_id to the validated requester key.
  6. The first NATS bridge is a separate Rust binary. It is authorized as a Unix uid/gid presenter through op:invoke on broker.invocation; actor authorization still comes only from the sealed envelope proof.
  7. Direct remote gRPC should use the sealed invocation service first. Metadata-based proof is not part of the first implementation.
  8. The canonical byte format is SecureMessage with a signed Header plus encrypted body. MessageType is the complete body discriminator. The BLAKE3 signature transcript starts with the basil-secure-message-v1 domain-separation string, then includes every Header field in protobuf field-number order, then the encrypted body. That deterministic transcript encoding is separate from protobuf's gRPC wire encoding and uses u32 little-endian integers, u32-length-prefixed strings and bytes, and u8 presence flags for optional fields. MessageType selects a documented CBOR schema for the plaintext message body. M2 and M3 define the schemas as Rust structs first and check in Rust-generated fixtures; Go helpers are deferred to a later issue and must produce identical CBOR bytes for every shared fixture before encryption when implemented. sender_subject and recipient_subject may be derived only when the sender or recipient key id maps unambiguously. Omitted expiry defaults to 60 seconds after issue time. Secret-bearing payloads use zeroizing storage and redacted debug output. Rust fixtures carry expected transcript, AAD, digest, and signature bytes for future cross-language implementations.

Deferred features

  1. ssh-ed25519 sealed-request principals are deferred behind raw Ed25519 and NATS NKey.
  2. Direct gRPC metadata proofs are deferred. This would mean a gRPC-only signature or certificate proof attached to a normal RPC, not cgroup or Kubernetes evidence.
  3. Go sealed-invocation helpers are deferred until after the Rust M3/M4 schemas, fixtures, and NATS bridge are stable.

This is a design proposal for Basil's pre-release API. It intentionally favors a clean breaking change over compatibility shims.
