# Subject-Based Authentication API Changes

This is a condensed implementation guide for the latest changes in
`designs/subject-based-authentication.html`. It describes the intended API shape
for updating code that already implements the older sealed invocation draft.

> **Superseded by COSE:** sections 1-8 below describe the earlier protobuf
> `Header`/`SecureMessage`/`MessageType` sealed-invocation draft. The current
> implementation follows `designs/basil-cose-api.md` and
> `designs/cose-sealed-invocation.html`: raw tagged COSE bytes on bridged
> transports, `SealedRequest`/`SealedResponse` as the thin gRPC carrier, and
> COSE claims for request/response correlation.

## 1. Replace The Envelope Shape

Use one `SecureMessage` shape for both requests and responses.

```proto
enum MessageType {
  MESSAGE_TYPE_UNSPECIFIED = 0;
  MINT_JWT_REQUEST = 1;
  MINT_NATS_USER_REQUEST = 2;
  SIGN_REQUEST = 3;
  SIGN_RESPONSE = 4;
}

message Header {
  uint32 version = 1;
  string message_id = 2;
  optional string request_id = 3;
  MessageType message_type = 4;
  uint32 issued_at_unix = 5;
  optional uint32 expires_at_unix = 6;
  string sender_sign_id = 7;
  optional string sender_subject = 8;
  string recipient_key_id = 9;
  optional string recipient_subject = 10;
  bytes encapsulated_key = 11;
  bytes nonce = 12;
}

message SecureMessage {
  Header header = 1;
  bytes body = 2;
  bytes signature = 3;
}
```

Breaking changes from the earlier draft:

- Delete `InvocationEnvelope`.
- Delete `InvocationResponseEnvelope`.
- Delete `status_hint` / `statusHint`.
- Delete the standalone `InvocationResponseBody` definition from the envelope API.
- Rename `actor_subject` to `sender_subject`.
- Rename `audience` to `recipient_subject`.
- Add `request_id` to `Header`.
- Move `message_type` and later header fields to the new field numbers shown above.

## 2. Update InvocationService

The bridged invocation RPC should be:

```proto
service InvocationService {
  rpc Invoke(SecureMessage) returns (SecureMessage);
}
```

The response is a sealed `SecureMessage`, not a response wrapper with cleartext
status.

The gRPC status is transport/protocol status only, for cases such as disabled
service, malformed envelope, oversized message, or inability to construct a
sealed response. Broker operation success, denial, and failure status must live
inside the encrypted response body selected by `MessageType`.

## 3. Request And Response Correlation

`message_id` is unique for the sender within the replay window. It does not need
128 bits of entropy. A UUIDv7 works; a durable sender-local monotonic counter is
also acceptable.

`request_id` is absent on first-leg requests.

For responses:

- `message_id` is a new sender-unique id generated by the responder.
- `request_id` equals the request envelope's `message_id`.
- The encrypted response body may also include request hash or other
  operation-specific binding data when that response schema needs stronger
  binding to the original request transcript.

Replay cache keys should use signer identity plus `message_id` until expiry plus
clock-skew allowance.

## 4. Body Encoding

The protobuf `SecureMessage.body` is encrypted CBOR bytes.

`MessageType` selects the CBOR schema for the plaintext body. Do not add
`body_type`, method strings, or protobuf type URLs to the envelope.

Each request and response body type should have:

- a stable `MessageType` enum value,
- a deterministic CBOR schema,
- Rust fixtures for plaintext CBOR bytes,
- encrypted-body and transcript test vectors as implementation stabilizes.

## 5. Protobuf Wire Encoding

Do not model the envelope as JSON. The wire shape is protobuf over gRPC.

Protobuf `bytes` fields are length-delimited raw bytes on the gRPC wire. They are
base64-encoded only by protobuf JSON/debug mappings.

`MessageType` is sent as its numeric enum value using protobuf's normal numeric
wire encoding. `uint32` fields are also protobuf numeric fields, not fixed-width
4-byte values on the protobuf wire.

## 6. Signature Transcript

The signature input is:

```text
BLAKE3(canonical_transcript)
```

Use normal unkeyed BLAKE3 parameters.

The canonical transcript is deterministic and separate from protobuf wire
encoding:

1. The fixed domain-separation string `basil-secure-message-v1`.
2. Every field in `Header`, in protobuf field-number order.
3. The encrypted `body`.

Transcript field encodings:

- integers and enum values: `u32` little-endian,
- optional fields: one `u8` presence flag, `0` or `1`, followed by the value when
  present,
- strings: `u32` little-endian byte length followed by UTF-8 bytes,
- byte fields: `u32` little-endian byte length followed by raw bytes.

This fixed transcript encoding exists only to make the BLAKE3 input
deterministic across implementations.

## 7. AEAD Associated Data

Do not add an `aad` field to `SecureMessage`.

AEAD associated data is derived, not transmitted. Recompute it from the canonical
header transcript, excluding `body`. Any unauthenticated metadata that must be
bound into encryption should become an explicit `Header` field.

## 8. Encryption And `encapsulated_key`

Every sealed message uses a fresh AEAD nonce.

If using PQC encryption, every sealed message also uses a fresh KEM
encapsulation. A response must not reuse the request's `encapsulated_key` or
`nonce`.

`encapsulated_key` is the raw KEM encapsulation output for the per-message body
key, if using PQC encryption. It is binary protobuf `bytes`, not base64 text.

For non-PQC encryption modes, keep the field semantics explicitly defined by the
chosen suite before implementation; do not infer ML-KEM behavior for a different
algorithm family.

## 9. Key IDs, Key Discovery, And Trust

`sender_sign_id` and `recipient_key_id` are Basil catalog key names. They are not
self-describing algorithm handles.

The caller must know the recipient key's public bytes and `KeyType` before
encrypting, because `KeyType` determines the public-key encoding and encryption
suite.

Existing local/trusted lookup APIs:

```proto
service SecretService {
  rpc ListCatalog(ListCatalogRequest) returns (stream CatalogEntry);
}

message CatalogEntry {
  string name = 1;
  CatalogKind kind = 2;
  optional KeyType key_type = 3;
  uint32 latest_version = 4;
}

service SigningService {
  rpc GetPublicKey(GetPublicKeyRequest) returns (GetPublicKeyResponse);
}

message GetPublicKeyRequest {
  string key_id = 1;
  optional uint32 version = 2;
}

message GetPublicKeyResponse {
  string key_id = 1;
  KeyType key_type = 2;
  bytes public_key = 3;
  uint32 version = 4;
}
```

These RPCs are not a trust bootstrap mechanism over an untrusted bridge or
network. A MITM could substitute its own recipient encryption key unless the key
descriptor is authenticated by a broker signing key the caller already trusts.

For remote/bridged invocation, clients must get the broker invocation-encryption
key from one of:

- static local configuration,
- a deployed trust bundle,
- a broker-signed key descriptor verified under a previously known broker
  identity key or trust root.

A signed key descriptor should bind at least:

- broker id,
- key id,
- key type,
- public key bytes,
- key version,
- issue time,
- optional expiry.

No reverse `describe_key(public_key)` API is required for sealed invocation. Raw
public bytes alone are insufficient because the envelope still needs an
authenticated catalog key id in `recipient_key_id`.

## 10. Response Protection

Responses are sealed messages:

- signed by the responder's configured signing key,
- encrypted to the caller's response recipient key,
- correlated with the request via `Header.request_id`.

Denials and errors must be signed and encrypted in the response body. Clients
must ignore clear gRPC status for broker-operation results and trust only the
decrypted, signed response body.

## 11. Validation Updates

Implementations and tests should reject:

- stale field names: `actor_subject`, `audience`,
- stale wrapper types: `InvocationEnvelope`, `InvocationResponseEnvelope`,
- cleartext response status as authoritative operation status,
- response reuse of request nonce or PQC `encapsulated_key`,
- unsupported recipient `KeyType`,
- remote recipient public-key descriptors that are neither configured out of band
  nor signed by a previously trusted broker identity key,
- signature transcripts with fields encoded in protobuf wire format instead of
  the canonical transcript format.

Update tamper tests so changing any of these breaks validation:

- `message_type`,
- key ids,
- ciphertext,
- `encapsulated_key` if using PQC encryption,
- `nonce`,
- `sender_subject`,
- `recipient_subject`,
- expiry,
- `request_id`.
