SPIFFE Workload API (gRPC) for Basil

Basil already attests the calling process by kernel credentials, makes a default-deny policy decision, and brokers crypto/secret operations where keys never leave the Vault-compatible backend. This document specifies the missing front door: a gRPC server implementing the standard SPIFFE Workload API (FetchX509SVID, FetchX509Bundles, FetchJWTSVID, FetchJWTBundles, ValidateJWTSVID), served by tonic over the same Unix socket that carries the sibling broker gRPC service. Workloads connect, are attested per connection, are mapped to a SPIFFE ID by the catalog/PDP, and receive short-lived SVIDs minted in place by the backend. Basil is the SPIFFE agent and server folded into one edge node: a lightweight single-node SPIRE alternative.

1. Problem statement & motivation

1.1 Why a Workload API at all

SPIFFE's value is that a workload obtains a cryptographic identity without holding a long-lived secret: it connects to a local endpoint, the platform attests which process is calling, and hands back a short-lived SVID scoped to that identity, rotating it before expiry. The standard endpoint for that exchange is the SPIFFE Workload API, a gRPC service reached over a Unix socket (SPIFFE_ENDPOINT_SOCKET). Speaking it verbatim means any SPIFFE-conformant client (go-spiffe, the rust-spiffe / c-spiffe SDKs, Envoy's SDS bridge, spiffe-helper) works against Basil unmodified.

Basil already has every ingredient SPIRE splits across an agent and a server:

SPIRE conceptBasil equivalentMaturity
Agent: attest the local workloadkernel SO_PEERCRED + /proc peer attestationImplemented
Agent: selectors → registration entrythe default-deny PDP (uid/gid → policy) + catalog trust_domain/spiffe_id labelsImplemented
Server: sign the SVID with the CA / JWT keyin-place signing in bao transit; bao PKI for X.509Implemented
Server: trust bundlethe CA chain / JWKS published from bao (§7)Implemented
Workload API endpointthis document (gRPC front door)Implemented

Basil folds the agent and server roles into one node (edge-first, single node). There is no upstream SPIRE server to cache from and no node-attestation step: Basil is the source of truth and is the minting authority.

1.2 Constraints (honored throughout)

  • Keys never leave the Vault-compatible backend. The JWT-SVID issuer key and the X.509-SVID CA key live in the backend (transit / PKI) and sign in place. The Workload API returns the SVID (a leaf cert + its leaf private key, or a signed JWT), never the issuer/CA private key. (The X.509 leaf key is a deliberate exception; §4.2.)
  • Provider-neutral core. The vault backend kind covers both OpenBao and HashiCorp Vault; issuance goes through the backend abstraction (plus a small extension, §4.4), so another PKI provider can drop in without touching the gRPC layer.
  • Edition 2024, no runtime panics. Every RPC handler returns a result that maps backend/PDP errors to a gRPC Status; no panic/unwrap/index on any runtime path. The broker is unrestartable; it fails closed.
  • Default-deny. No SVID is issued unless the PDP grants it for the attested uid (§3.4). An unattested connection (no SO_PEERCRED) gets nothing.
  • Edge-first single node. One Basil instance = one trust domain = one environment. Federation is a bundle-exchange add-on (§7.3), not a multi-node core.

1.3 Scope

In scope: the framework choice; the shared multi-service tonic server + attestation + PDP middleware (authoritative for the broker service too); X.509-SVID + JWT-SVID issuance via bao with issuer-key custody; attestation→SPIFFE-ID mapping; rotation/streaming vs the spec state machines; trust bundle + a federation path; revocation (§8); the .proto surface; the workload.spiffe.io=true header enforcement; the crate/module placement (§11).

Out of scope / future: node attestation (§7.4) / multi-node SPIRE federation control plane; the broker gRPC service surface itself (vault-8as); a REST/JSON-HTTP gateway; the SPIFFE Federation API auto-refresh transport (static federation only in v1, §7.3).

2. Framework choice + rationale

2.1 Decision: tonic

tonic (tokio + hyper + tower) is the gRPC stack. It is the de-facto Rust gRPC implementation, pure-Rust (no C++ toolchain), already sits on the runtime Basil uses, and, critically for §3, exposes the two seams this design needs: Unix-socket serving and per-connection connect-info carrying SO_PEERCRED, plus tower layers + interceptors for cross-cutting auth.

2.2 Alternatives considered

OptionVerdictWhy
tonicchosenPure-Rust on tokio/hyper/tower; first-class UDS + peer-cred at connect; multi-service registration on one server; tower/interceptor middleware for attestation + PDP; codegen via prost/tonic-build.
grpcio / grpc-rsrejectedWraps the C-core gRPC library → a C++/CMake build dep, foreign to this NixOS-only, pure-Rust tree (we deliberately use rustls to avoid even OpenSSL). Its UDS-peer-cred story is far less idiomatic.
hand-rolled over h2rejectedRe-implements gRPC framing, status mapping, and streaming for no gain; tonic is the thin layer over h2 we'd write.
axum / hyper RESTnot a gRPC optionThe Workload API is defined as gRPC and SPIFFE clients speak gRPC. axum is relevant only if a REST surface is later wanted (health/metrics), and even then it shares the same hyper/tower stack. Logged as a future option.

2.3 Why this matters for the broker service

The broker gRPC service (vault-8as) needs the identical transport + attestation + PDP plumbing. Picking tonic once, here, and registering both services on one server over one Unix socket is what lets the two services share the socket, the connect-info attestation, and the PDP gate, instead of standing up two servers or duplicating the middleware. §3 is the contract the broker design builds on.

3. Shared transport foundation Implemented

This section is the single source of truth for the gRPC transport, the per-connection attestation, and the per-RPC PDP gate. The Workload API service and the broker gRPC service are both registered on the server built here. The broker design (broker-grpc-api.html §5) references this section and does not re-specify it.

3.1 One tonic server, multiple services, one socket

A single tonic server serves both services over one Unix listener. Multi-service registration is native: the server routes by the gRPC path prefix (/SpiffeWorkloadAPI/… vs the broker package), so both live behind one listener with no manual demux.

  • One socket. The socket path is the SPIFFE endpoint socket for clients (e.g. unix:///run/basil/api.sock) and the same socket the broker clients dial. There is exactly one listener; there is no separate legacy path (§12).
  • Socket perms / shutdown reuse the existing server posture: bind, best-effort chmod 0600 (the kernel still attests every peer regardless), and a graceful drain on SIGINT/SIGQUIT/SIGTERM (the only sanctioned exit, §3.6).

3.2 Per-connection attestation: connect-info → PeerInfo

tonic captures peer credentials at connect time. For a UDS transport, the connection's connect-info carries the SO_PEERCRED snapshot (uid, gid, pid), exactly the kernel-trustworthy anchor the existing peer attestation already builds a PeerInfo from Implemented.

The connect-vs-RPC seam. Connect-info is populated once per connection; handlers run per RPC. tonic bridges the two: every request's extensions contain the connection's connect-info. So the flow is:

  1. At connect: the transport records the peer credential.
  2. Per RPC: the shared tonic transport lifts the credential and builds a PeerInfo by reusing the existing attestation enrichment (/proc exe, lineage), callable from the gRPC request path Implemented.
  3. The interceptor inserts the built PeerInfo into the request extensions, so every handler retrieves the attested (uid, gid, pid) with no re-attestation. A missing/uid-less credential is rejected fail-closed as UNAUTHENTICATED, never a panic.

Implementer caveat. An interceptor that rewrites the request type sees an erased request and can only read/modify metadata + extensions, sufficient here (we only read connect-info and insert a PeerInfo). Confirm against the pinned tonic version that connect-info is present at interceptor stage; if a version gates it, fall back to reading connect-info inside each handler (still per-connection data, just resolved at handler entry). Either way the source of truth is the same kernel credential. The connect-vs-RPC resolution must be the same choice in both gRPC services (§14).

3.3 The tower-middleware story

  • Attestation layer: wraps the whole server; on each request it ensures a PeerInfo is in the extensions (§3.2). A tower layer is the right tool when the behavior must apply to every service uniformly (both Workload API and broker). It may be a tonic interceptor if the simpler form reaches connect-info; the implementer picks the least-code option. The middleware never panics: a missing/uid-less credential → UNAUTHENTICATED.
  • PDP gate is per-RPC and per-method, not a blanket layer, because the (op, key) it decides on is method-specific. So the gate lives in a small helper each handler calls first (§3.4), reading the PeerInfo the layer deposited.

3.4 PDP gating per-RPC (default-deny)

Each handler, before doing any work, resolves the attested uid → policy decision via the existing PDP, reusing its exact action model and default-deny Implemented. Both gRPC services use one shared helper, owned by this section: 

// shared, owned here; consumed identically by the broker service (§14)
fn authorize(req_ext, op, key) -> Result<u32 /* attested uid */, Status>
//  1. read PeerInfo from request extensions
//  2. if uid is None            -> Err(UNAUTHENTICATED)   // fail closed
//  3. decision = pdp.decide(uid, op, key)
//  4. record DecisionRecord (audit) for allow AND deny
//  5. if decision.is_deny()     -> Err(PERMISSION_DENIED) // opaque; no key-existence leak
//  6. else                      -> Ok(uid)
Sync (§14): the helper returns the validated uid: both the SPIFFE-ID templating here (§5.1) and the broker's audit/mint paths need it. One name, one signature, in both services.
  • Op::Mint is the capability the Workload API exercises, the same mint capability the existing JWT minting uses, gated on the issuer key (JWT) or the CA key (X.509). mint is its own capability, separate from sign: a workload that can fetch an SVID does not thereby gain raw sign over the issuer/CA key.
  • The decision target is the issuer/CA catalog key, not the workload's own SPIFFE ID. The PDP answers “may this caller obtain an SVID minted by this issuer key”; the catalog binds the attested uid (user:<uid> / group:<gid>) to role:minter over the issuer key. The resulting SPIFFE ID is derived in §5.
  • Audit flows through the same decision-record / JSONL sink the existing path uses: one record per fetch, allow or deny.
  • Error mapping is opaque. A deny is PERMISSION_DENIED with a fixed message; an unknown key is also a deny (never “no such key”), so the gRPC surface leaks no more than the socket did.

3.5 No-panic error mapping

A single, total mapping function turns internal errors into a gRPC Status, never panicking. This table is shared with the broker service (§14) and is reconciled with that doc's error model:

Internal conditiongRPC Status code
PDP deny / unknown key (opaque)PERMISSION_DENIED
no attested uidUNAUTHENTICATED
missing workload.spiffe.io header (§10)INVALID_ARGUMENT
unsupported key type / algorithm (valid request, not built)UNIMPLEMENTED
backend transport failure (bao unreachable)UNAVAILABLE
backend reached but rejected / protocol decodeINTERNAL
invalid request (bad audience, malformed token, bad SPIFFE id)INVALID_ARGUMENT
backend “key not found” for a configured issuerINTERNAL (a misconfiguration, not the caller's fault)
Reconciled with the broker doc: an unsupported key type/algorithm maps to UNIMPLEMENTED (the additive-evolution slot), not FAILED_PRECONDITION. Both docs now agree.

3.6 Lifecycle / shutdown

Reuse the existing server posture: bind the listener, chmod 0600, serve with a graceful-shutdown future fed by SIGINT/SIGQUIT/SIGTERM, drain in-flight RPCs within a bounded window, unbind + remove the socket file, exit 0. The streaming handlers (§6) must observe the shutdown broadcast and end their streams cleanly (a select! over the rotation source and the shutdown receiver).

4. SVID issuance via bao (issuer-key custody)

4.1 The problem being fixed

The obsolete SVID minter generated an RSA-2048 key inside the broker process and signed RS256 locally. That path is retired Implemented. Issuer-key custody now belongs in the Vault-compatible backend. Two custody designs, one per SVID type.

4.2 X.509-SVID: Vault-compatible PKI engine (recommended) Implemented

An X.509-SVID is an X.509 leaf certificate whose single URI SAN is the SPIFFE ID (spiffe://<trust_domain>/<path>), CA-basic-constraints false, short-lived, returned to the workload with its leaf private key for mTLS. The recommendation is a Vault-compatible PKI secrets engine over step-ca:

Vault-compatible PKIstep-ca
CA key custodyCA key lives in the backend; signs leaves in place (never exported), satisfying the invariant by constructionseparate daemon, separate key store; another component to unlock/operate
URI-SAN SPIFFE IDa PKI role with allowed_uri_sans = the trust-domain pattern; request carries the uri_sanssupported (step is SPIFFE-aware) but via a second service
Provider-neutral fitone backend kind (vault) for transit + KV + PKIa second provider to abstract
Edge footprintreuses the backend the broker already authenticates toextra process, extra unlock surface
Test/devthe existing self-signed-TLS generator stays for test fixtures onlyalready present for tests

So: Vault-compatible PKI for production X.509-SVID issuance; step-ca stays test-only.

CA path + key custody. A PKI mount in the backend holds the trust-domain CA (root, or preferably an intermediate signed by an offline root). The CA private key never leaves the backend; issuance is an internal pki/issue/<role> that mints and signs the leaf inside the backend. The catalog gains a key entry for the SVID CA routed to the backend with engine = pki:

"spiffe.svid_ca" = {
  class = "asymmetric";            # a signing authority
  keyType = "ed25519";             # CA key type (rsa-2048 / ecdsa-p256 also valid)
  backend = "bao";
  engine = "pki";                  # NEW engine value (§4.4)
  path = "pki/issue/svid";         # the PKI role that mints SVID leaves
  writable = false;                # CA provisioned out-of-band (externally managed)
  missing = "error";
  labels = [ "trust_domain=example.org" "svid_kind=x509" ];
  description = "Vault-compatible PKI role that issues X.509-SVID leaves (CA key signs in place).";
};

4.2.1 Leaf-key custody: broker-generated for v1 (resolved)

An X.509-SVID requires the workload to hold the leaf key. The open question was whether to keep the leaf key out of the broker entirely by accepting a workload-supplied CSR, versus having the backend generate the keypair and the broker relay the leaf key to the workload. The two stated goals ((a) stay conformant with the SPIFFE Workload API and (b) keep private keys out of the broker) are not, in fact, in conflict once Basil is placed correctly in the SPIFFE model:

  • Goal (b) is about the issuer/CA key, and that stays in the backend regardless of how the leaf is handled. The custody invariant is fully met.
  • For the leaf, SPIFFE's own architecture has the agent generate the workload's key and hand it over locally. Basil is the agent (it folds agent + server, §1.1). So broker-side leaf generation is the conformant, idiomatic model, not a compromise of it. The leaf transits exactly one kernel-attested UDS hop to the very workload it identifies, then is zeroized.
  • The real conflict is the reverse: the standard X509SVIDRequest has no CSR field. A CSR flow would therefore require a non-conformant protocol extension or a side channel. That is, pursuing CSR to satisfy goal (b) is what would break goal (a).

Decision (v1): the backend generates the leaf keypair (the pki/issue call returns leaf key + certificate + chain); the broker hands the leaf key straight to the requesting workload over the attested local socket, then zeroizes its own copy. The CA/issuer key never leaves the backend; only the per-workload leaf transits, to the workload it identifies.

Escape hatch (future): if a specific high-value workload must keep its leaf key entirely out of the broker, that is offered as a separate Basil-proprietary broker RPC (CSR in, certificate out) on the broker package. It is additive, and it does not bend the standard FetchX509SVID. A CSR over the Workload API itself is rejected as non-conformant. Logged as a future option.

4.3 JWT-SVID: transit EdDSA (default), RS256-PKCS1v15 for compatibility

A JWT-SVID is a JWS (sub = SPIFFE ID, aud = the requested audience(s), iat, exp) signed by an issuer key in Vault-compatible transit: the existing minting flow already signs in place and only ever reads the public half Implemented. The only fix is that the issuer key must be a catalog/transit key, not the in-process RSA key.

  • Default: EdDSA (Ed25519) issuer. Transit Ed25519 signs cleanly with no extra params, and the existing minter already builds the alg:"EdDSA" JWS. Least footgun.
  • RS256-PKCS1v15 supported for compatibility Implemented. Some verifiers insist on RSA. Transit's sign defaults to PSS; a JWS-RS256-verifiable signature needs signature_algorithm=pkcs1v15 + hash_algorithm=sha2-256 on the transit call. The transit backend passes those two params for an rsa-2048 issuer. EdDSA remains the default; RS256-PKCS1v15 is first-class for compatibility.
  • JWKS / kid. The issuer's public key is published as a JWKS for verifiers (FetchJWTBundles, §7). The kid is derived deterministically from the transit key (e.g. a hash of the SPKI), so a rotation (new transit key version) is a new kid in the JWKS, replacing today's random kid, which can't be re-derived.
"spiffe.jwt_issuer" = {
  class = "asymmetric"; keyType = "ed25519"; backend = "bao";
  engine = "transit"; path = "spiffe-jwt-issuer"; writable = false;
  missing = "error";
  labels = [ "trust_domain=example.org" "svid_kind=jwt" "spiffe_id=spiffe://example.org/basil" ];
  description = "Transit Ed25519 key that signs JWT-SVIDs in place (seed never leaves the backend).";
};

4.4 Backend extension (provider-neutral) Implemented

JWT-SVID issuance needs no new backend method: in-place sign + public_key already cover it. X.509-SVID issuance uses one provider-neutral backend method so the gRPC layer stays provider-neutral and non-X.509 backends fail cleanly: 

// additive; default = Unsupported, mirroring the existing optional methods
struct X509Svid {
  cert_chain_pem: String,   // leaf + CA chain (PEM), leaf first
  leaf_key_pem:   String,   // the leaf private key (Zeroizing in the impl)
  bundle_pem:     String,   // the trust-domain CA bundle (roots)
  not_after:      u64,      // unix exp, for rotate-before-expiry (§6)
}

async fn issue_x509_svid(issuer_key_id, spiffe_id, ttl)
    -> Result<X509Svid, BackendError>;   // default: Err(Unsupported)
  • The Vault-compatible backend implements it via pki/issue/<role> with uri_sans=<spiffe_id>, ttl=<ttl>; the CA key signs in place.
  • The catalog Engine gains a Pki variant alongside Transit/Kv2, set explicitly only when the entry has engine = "pki" (PKI is never inferred; transit is the default for asymmetric).
  • Zeroize discipline: the leaf key and any transient PEM buffers are zeroized; the broker holds the leaf key only long enough to put it on the wire, then drops it.

4.5 In-process RSA issuer retired

The in-process RSA SVID minter is removed from the issuing path. Its only remaining legitimate use was the broker→bao authentication SVID (§7.2), a different direction; the flag-day cutover moves that to a backend-/catalog-backed key too (§13, resolved), so the broker process holds no signing key in memory. The Workload API never instantiates an in-process issuer.

5. Attestation → SPIFFE-ID mapping

The attested caller is mapped to a SPIFFE ID by reusing the catalog/PDP + the trust-domain templating that already exists Implemented; this design lifts the existing logic into the gRPC handlers, it does not invent a new mapping.

5.1 The mapping

  1. The attested (uid, gid, pid) come from PeerInfo (§3.2).
  2. The PDP decides (uid, Op::Mint, issuer_key), default-deny (§3.4), returning the validated uid.
  3. The SPIFFE ID is templated as spiffe://<trust_domain>/<name>, where <trust_domain> is the issuer key's catalog trust_domain label and <name> is the caller's symbolic name from the config name table, falling back to the bare uid when unknown.
  4. The result is validated as a well-formed spiffe://td/path (no control/whitespace) before use.

This is the registration-entry equivalent: the catalog (trust-domain label + the minter rule binding the uid) is the SPIFFE registration. No selectors DSL is needed for v1: uid/gid + the catalog rule are the selectors.

5.2 ID resolution on fetch

  • Default: the workload calls with no spiffe_id and gets the SVID(s) for its attested identity (the templated ID). The common “give me my identity” flow.
  • Explicit spiffe_id: issued only if the attested caller is authorized for that identity (the PDP grants mint on an issuer key whose trust-domain/policy covers it). An unauthorized explicit ID is PERMISSION_DENIED.
  • /proc selectors are advisory only. pid//proc/<pid>/exe are recorded for audit but are never a decision input; the kernel-trustworthy uid is the anchor. A richer selector model (exe path, cgroup) is a future option that extends the catalog rule grammar, not the gRPC layer.

5.3 Multiple IDs / hint

v1 returns one SVID per fetch (the caller's identity), hint empty. If the catalog later maps one uid to multiple IDs, the response can carry several SVIDs with distinct hints. Additive, no protocol change.

6. Rotation / streaming

We own the server half of the spec's Appendix A state machines; we skip the client half. Basil is the server/source: it mints in place and is the authority.

6.1 What we own (server side) Implemented

  • Serve the stream and push on change. FetchX509SVID, FetchX509Bundles, and FetchJWTBundles are server-streaming. The handler immediately yields the current complete set (the spec requires each message to be the complete set at that point in time), then stays open and yields a new complete message whenever the SVID rotates or the bundle changes, until the client disconnects or the server shuts down (§3.6).
  • Mint fresh, short-lived, with a re-fetch cadence. Because we mint in place, “rotation” is just minting a new SVID before the current one expires. The streaming handler holds a timer at TTL/2 (the SPIRE default rotation point, §13 Q4); when it fires it re-mints via bao and pushes the new response. Drives: TTL/2 elapsed, the issuer/CA key rotated, or the trust bundle changed.
  • Validate the request first. Header check (§10) + PDP gate (§3.4) on every stream open; an unauthorized/headerless open is rejected before any SVID is minted.

6.2 What we skip (client side)

We are not a client caching SVIDs from an upstream: there is no upstream SPIRE server, no node attestation, no durable cross-restart cache, and the workload owns its own backoff/reconnect. So we skip the client cache, the client rotate-before-expiry loop (we rotate by re-minting), and the client backoff. We keep a small coalescing cache keyed by (spiffe_id, audience) purely to avoid re-signing on bursty fetches; it is an optimization, not the spec's durable cache, and it is invalidated on issuer/bundle change.

6.3 The concerns, mapped

ConcernWhoHow in Basil
fetchserverthe handler mints on stream open / unary call via bao
cacheserver (optional)short-TTL in-memory (spiffe_id, audience) → SVID to coalesce bursts; invalidated on rotate

6.4 Streaming mechanics (no-panic)

Each streaming handler returns a boxed stream and internally select!s over: a rotation timer (at TTL/2), a watch receiver for bundle/issuer changes and revocation events (§8), and the shutdown receiver (§3.6). Every branch maps errors to Status (a bao outage mid-stream → yield UNAVAILABLE and either retry internally or end the stream so the client reconnects). No panic, and a dropped client just drops the stream.

7. Trust-bundle serving, federation + node attestation

7.1 Bundles served from bao

  • X.509 bundle: the trust-domain CA chain, read from the bao PKI mount. Returned as ASN.1 DER per the proto (map<trust_domain, bytes> of DER roots), local trust domain always present.
  • JWT bundle (JWKS): built from the JWT-SVID issuer's public key(s) in transit (current + grace-window versions, so a token signed just before a rotation still verifies). The kids match the JWS kid (§4.3).
  • Both are exposed via a watch updated on startup-reconcile, on issuer/CA rotation, and on revocation (§8), so the streaming handlers push updates.

7.2 Coexistence with the broker→bao auth SVID

The existing backend uses a JWT-SVID to authenticate the broker → bao, the opposite direction from the Workload API. Reconciled:

  • Basil as SVID presenter (existing): broker → bao auth. The broker self-issues an SVID to log into bao. Under the flag-day cutover that signing key moves into bao so the broker process holds no key (§4.5, §13).
  • Basil as SVID issuer (new, this doc): local workloads → Basil, using the bao-custodied issuer/CA key.
  • These share the trust domain and the JWKS: the same JWKS Basil publishes via FetchJWTBundles is what a verifier (including bao's jwt auth) uses to validate both the broker's own auth SVID and the SVIDs Basil issues, provided both are signed by the same catalog issuer key. One issuer key in bao, one JWKS, two consumers.

7.3 Federation path (deferred)

Federation = trusting SVIDs from another trust domain. The proto already carries federated_bundles map slots. Decision: federation is deferred (§13 Q8). We do not need to decide the transport now. When it lands, v1 would ship static federation (operator-loaded foreign bundles, stored in bao KV, published via the bundle maps); the SPIFFE Federation API auto-refresh (an HTTPS bundle-endpoint poller updating the watch) is a later, additive add-on with no change to the Workload API surface. Basil stays edge-first single-node; federation is for peering edges or bridging to a central SPIRE, not a multi-node control plane.

7.4 Node attestation: deferred; TPM-gated node identity the intended form Future

“Node attestation” means three different things against the folded single-node model, and only one is on the roadmap:

  • SPIRE agent→server node attestation: not applicable. When the agent and server are the same process, the node proving itself to the server is the process trusting itself; implementing the join-token / IID handshake internally would add ceremony with no security gain. Workload attestation (SO_PEERCRED + /proc) is the local job Basil does need, and already does.
  • Node attestation of Basil (TPM-gated node identity): the intended hardening. The reasonable form for an edge broker is to bind Basil's unlock (designs/unlock-and-bundle) and its broker→bao credential to a hardware node identity (a TPM DevID and/or a measured-boot quote) so the node must be the expected host in the expected state before it ever holds key material. This upgrades the current possession-based bootstrap (the sealed bundle) into a true node attestation, and it is single-node-compatible. It fits the existing tpm2 local-identity scaffolding. It lands when TPM support is fully integrated; the practical blocker is that the current dev platform exposes no TPM. An emulated TPM (swtpm under QEMU, with OVMF for measured boot) is the candidate substrate for exercising these code paths in CI: it validates the plumbing (DevID, quotes, sealing) but, being software-rooted, not hardware assurance. The testing shape is two lanes: an emulated lane (swtpm/QEMU, e.g. a nixosTest with virtualisation.tpm.enable) for functional coverage that runs anywhere, and a real-TPM lane (hardware/staging) for assurance before production relies on it. For the emulated lane to exercise EK/DevID certificate validation, the EK/DevID trust anchor must be configurable: accept a test CA in CI rather than hardcoding only vendor roots.
  • Node attestation by Basil of other nodes: travels with federation. A workload on another host shares no kernel, so SO_PEERCRED does not apply; serving it requires a network transport (tonic over TCP + mTLS), agent-SVID issuance, a node-attestor abstraction, and node selectors in the catalog/PDP rule grammar (the richer-selector future option of §5.2). That is the multi-node SPIRE-server role this design declines; it is deferred together with federation (§7.3).

8. Revocation Future

The default posture is short TTLs (§13 Q4) so that an un-renewed credential simply expires. On top of that, active revocation is supported (resolved, Q6), split by SVID type because JWTs have no CRL:

8.1 X.509-SVID revocation: bao PKI CRL (v1)

  • bao PKI already revokes a serial and rotates a CRL. Revocation is a pki/revoke + pki/crl/rotate; the broker reads the current CRL and populates the crl field of X509BundlesResponse (empty until now).
  • A revocation updates the bundle watch, so the server-streaming FetchX509Bundles handlers push the new CRL to connected verifiers immediately (§6.4, §7.1).

8.2 JWT-SVID revocation: a persistent jti deny-list Future · P3

There is no CRL for JWTs. Revocation = short TTL (default) + issuer-key rotation (the nuclear option: a new kid invalidates every token under the old key) + an explicit jti deny-list checked in ValidateJWTSVID (§9). The deny-list is priority 3 (after the core gRPC surface and X.509 CRL):

  • Persistent. The broker is unrestartable in normal operation, but it can still crash; a revocation must survive a crash. The deny-list is stored durably (bao KV) and reloaded at startup so a restart cannot silently un-revoke a token. Entries auto-expire when they pass the revoked token's own exp (no unbounded growth).
  • Checked at validation. ValidateJWTSVID rejects a token whose jti is on the deny-list as INVALID_ARGUMENT (opaque, like any other invalid token).

8.3 Revocation notifications on the admin Watch

Yes, revocation is surfaced on the broker's admin Watch API. The broker design's AdminService.Watch (now in scope, see broker-grpc-api.html §7) emits rotation/bundle and revocation events from the same internal event source that drives this doc's streaming push (§6.4). So a consumer that cares about “a credential was revoked” (a sidecar invalidating a cache, an operator dashboard) subscribes once to Watch rather than polling. X.509 revocation already rides the bundle watch (CRL update); a jti deny-list addition emits a revocation event on the same channel. This is the unification that makes the deny-list worth its persistence cost: one event bus, two streaming consumers (the SPIFFE bundle streams here, and AdminService.Watch there).

9. The .proto surface

9.1 Canonical source: vendor it, don't re-author

The service is the standard SPIFFE Workload API, defined canonically in the upstream spiffe/spiffe repository (standards/workloadapi.proto, proto3). Vendor that file verbatim into the tree (pinned to a recorded upstream commit) and generate the Rust types with tonic-build/prost. Do not hand-write or paraphrase the messages: conformance depends on byte-for-byte field/tag fidelity, and real SPIFFE clients are generated from this exact file.

Because the vendored file is proto3, the sibling broker proto is also proto3 + optional (not protobuf editions), so one tonic-build invocation in basil-proto builds both packages under one syntax (§11, and broker doc §7). This resolves the editions-vs-proto3 question in favor of proto3.

The proto declares one service, SpiffeWorkloadAPI:

service SpiffeWorkloadAPI {
  rpc FetchX509SVID(X509SVIDRequest)          returns (stream X509SVIDResponse);
  rpc FetchX509Bundles(X509BundlesRequest)    returns (stream X509BundlesResponse);
  rpc FetchJWTSVID(JWTSVIDRequest)            returns (JWTSVIDResponse);
  rpc FetchJWTBundles(JWTBundlesRequest)      returns (stream JWTBundlesResponse);
  rpc ValidateJWTSVID(ValidateJWTSVIDRequest) returns (ValidateJWTSVIDResponse);
}

FetchX509Bundlesserver-streamyield {td → DER roots} + the crl (§8.1); push on bundle/CRL change Implemented

  • ValidateJWTSVID validates locally against the published JWKS (verification needs only the public key) plus the deny-list; a malformed token, bad signature, wrong/absent audience, expired, or revoked token is INVALID_ARGUMENT with an opaque message (no oracle detail). It is PDP-gated; see §9.3.

9.3 ValidateJWTSVID is PDP-gated (resolved)

The open question was whether validation should be open to any attested local caller or PDP-gated. Decision: add a validate capability to the policy and gate ValidateJWTSVID on it. Because every caller is already declared in the startup config, the policy can name exactly which principals may use Basil as a JWT verifier. The gate is (uid, Op::Validate, jwt_issuer_key) through the same shared helper (§3.4): a caller granted validate on the issuer key may validate tokens minted under that issuer. This adds an Op::Validate to the action model alongside the existing operations.

9.4 New build/deps

tonic, prost, tonic-build (build-dep), tokio-stream (for the UDS incoming stream). Additive workspace deps; the runtime, serde, the bao HTTP client, JWS assembly, and zeroize are already present.

10. The workload.spiffe.io=true header

The spec requires every Workload API call to carry the metadata header workload.spiffe.io: true; a request without it MUST be rejected. This is an anti-confused-deputy / anti-CSRF measure (it forces the caller to be a deliberate gRPC client).

  • Where: a method-scoped check at the top of every Workload API handler (and, for streaming, before the first yield). It is not in the shared attestation layer, because the header is required only on the Workload API methods. The sibling broker service does not require it (§14). A tiny shared helper keeps it one line per handler.
  • Status code: INVALID_ARGUMENT if absent, returned before any attestation result is acted on and before any SVID is minted.
  • Order within a handler: (1) header check → (2) read PeerInfo (must have a uid, else UNAUTHENTICATED) → (3) PDP gate (§3.4) → (4) derive SPIFFE ID (§5) → (5) mint via bao (§4) → (6) respond/stream.
  • Alternative considered: a per-service tower layer applied only to the Workload API service. Rejected for v1 as more wiring than a one-line helper, but a clean refactor if the method count grows.

11. Crate layout & module placement

The gRPC cutover lands together with the crate restructure (the authoritative target is recorded in AGENTS.md). The previous split of a server library plus a thin daemon binary is replaced:

11.1 Where the Workload API lives

  • The SpiffeWorkloadAPI handlers are service/spiffe, behind the spiffe feature. With the feature off, the node serves the broker services only (no SPIFFE code, no SPIFFE-specific deps).
  • The shared transport foundation (§3) lives in transport/ and is always compiled; both services consume it. It comprises the tonic server, the attestation layer, the connect-info→PeerInfo bridge, the authorize helper, and the error mapping.
  • The SVID-specific minting (the SPIFFE-ID templating, the JWT-SVID claim assembly, the X.509 issuance) lives in service/spiffe. The sign-in-place primitive it shares with the broker's NATS/generic minting lives in core/, so service/spiffe never depends on service/minting: the module graph stays acyclic and services never reach sideways (§14, and broker doc §6).

12. Coexistence (flag-day, no dual-run)

12.1 With the broker gRPC service: one server, one socket

Both gRPC services are registered on the one tonic server over the one socket (§3.1), share the one attestation layer (§3.3), the one PDP helper + shared state, and the one audit sink. The only Workload-API-specific additions on top of the shared foundation are the workload.spiffe.io header check (§10) and the SVID-minting handlers. The broker design supplies its own services + its own per-RPC (op, key) mapping but reuses everything in §3.

12.2 The legacy JSON socket is gone

The JSON socket is retired: flag-day, no dual-run. The project is pre-release with no external users, so there is no run-both phase and nothing “runs alongside during migration.” The broker ops and the Workload API both arrive on the one tonic server over the one socket; the hand-rolled JSON wire, its codec, and the JSON client paths are deleted in the same cut, and the in-tree clients are regenerated as gRPC.

  • The old svid JWT preset is removed, not migrated. JWT-SVIDs are issued only by the Workload API (FetchJWTSVID); the bao-custody fix (§4.3) lands directly in the Workload-API issuer. There is no JSON preset to keep alive.
  • The broker→bao auth SVID moves to a bao/catalog key in the same cut (§4.5). The broker process holds no in-memory signing key.
  • Socket: the one gRPC socket is the advertised SPIFFE_ENDPOINT_SOCKET; there is no separate JSON path.

13. Decisions

Every prior open question is resolved; the resolutions are folded in below.

  • tonic is the gRPC stack; grpcio (C-core), hand-rolled h2, and axum (REST) rejected (§2).
  • One tonic server + one socket serving both the Workload API and the broker service; §3 is authoritative and shared (Q3: one listener, multiple services).
  • Per-connection SO_PEERCRED lifted into a PeerInfo (reusing the existing attestation, refactored to build from a raw credential) and inserted into request extensions by the attestation layer; handlers read it per-RPC (§3.2).
  • Per-RPC PDP gate, default-deny, gated on the issuer/CA catalog key with Op::Mint; opaque deny; audited. The shared authorize helper returns the validated uid and is identical in both services (§3.4, §14).
  • No-panic handlers; one total error→Status mapping, reconciled with the broker doc: unsupported key type/algorithm is UNIMPLEMENTED, not FAILED_PRECONDITION (§3.5).
  • The in-process RSA issuer is removed from issuance. Custody is in the Vault-compatible backend: transit EdDSA for JWT-SVIDs, PKI for the X.509 CA (§4).
  • JWT-SVID default = EdDSA (the modern standard); RS256-PKCS1v15 is supported for compatibility (transit passes signature_algorithm=pkcs1v15 + hash_algorithm=sha2-256) (Q2, §4.3).
  • X.509-SVID via Vault-compatible PKI, not step-ca (step-ca stays test-only); the per-workload leaf key is the one deliberate export, broker-generated for v1 and zeroized, conformant with the SPIFFE agent model; the CA key never leaves the backend. A CSR-based endpoint is a future broker-proprietary RPC, never a Workload-API extension (Q1, §4.2.1).
  • One new backend method (issue_x509_svid, default Unsupported) + a catalog Engine::Pki variant; JWT issuance needs no new method (§4.4).
  • SPIFFE-ID derivation reuses the existing trust-domain templating (spiffe://<trust_domain>/<name|uid>); uid is the anchor, /proc advisory only (§5).
  • Trust domain is per-Basil-node (Q3, see §13.1).
  • SVID TTL + rotation: SPIRE defaults (~1h X.509 leaf, short JWT; re-push / rotate at TTL/2) as the starting point (Q4, §6).
  • ValidateJWTSVID is PDP-gated via a new validate capability on the issuer key (Q5, §9.3).
  • Revocation is supported, not just TTL: X.509 via bao PKI CRL (populating the crl field); JWT via short TTL + issuer rotation + a persistent jti deny-list at priority 3 (reloaded after a crash); revocation is surfaced on the admin Watch (Q6, §8).
  • We own the server state machine (serve + push-on-rotation), skip the client one; rotate = re-mint at TTL/2 and push; optional short-TTL coalescing cache only (§6).
  • Trust bundles served from bao (PKI for X.509; transit-public JWKS for JWT) via a watch so streams push updates (§7).
  • The standard workloadapi.proto is vendored verbatim (pinned commit) and built with tonic-build as proto3, matching the broker proto's syntax in one basil-proto build (Q from broker §7, §9.1, §11).
  • workload.spiffe.io=true enforced per Workload-API method (one-line helper), INVALID_ARGUMENT if absent; not required by the broker service (§10, §14).
  • The broker→bao auth SVID moves to a bao/catalog key now: flag-day; the JSON socket is already retired (Q7, §4.5, §12).
  • Federation is deferred (Q8); static federation is the eventual v1, the Federation API auto-refresh later, both additive (§7.3).
  • Node attestation (§7.4): the SPIRE agent→server form is not applicable to the folded single-node model; TPM-gated node identity (binding unlock + the bao credential to a hardware node identity) is the intended future hardening, deferred pending full TPM integration (the current dev platform has no TPM; an emulated TPM via swtpm/QEMU is the candidate test substrate); node attestation of remote nodes travels with federation and stays deferred.
  • Flag-day migration: the JSON wire is deleted in the same change that lands gRPC; no dual-run (§12).
  • Crate layout: basil-proto extracted; the server library folds into basil-agent (core/service/transport/main); the Workload API is service/spiffe behind the spiffe feature; the shared sign-in-place primitive lives in core/ (§11).

13.1 Trust-domain identity: per-node (resolved)

The question was whether the trust domain is per-Basil-node or shared across peered edges. Decision: per-node. The tradeoffs:

  • Per-node (chosen). Matches “one broker = one environment” and the edge-first single-node honesty of this design. Simplest data model. Kept as a per-issuer-key trust_domain label (not a single hard-coded global), so the catalog stays uniform and the data model does not have to change if federation later bridges two domains. Two nodes that need to trust each other do so via federation (deferred, Q8), not by sharing a domain.
  • Shared td across peered edges (rejected for v1). Multiple nodes issuing in the same trust domain with different issuer keys means every verifier needs all those keys in the JWKS/bundle, and you have built a multi-issuer trust domain with no control plane to coordinate it, which is exactly the SPIRE-server role this design declares Basil is not. The failure mode (a verifier missing one node's key silently rejects valid SVIDs) is the kind of cross-node drift the single-node model avoids.

14. Sync points with the broker-gRPC design

These must stay consistent between this doc and broker-grpc-api.html:

  • Transport (§3.1): ONE tonic server + ONE socket; the broker registers its services on the server built here. Neither doc stands up a second server.
  • Attestation extensions (§3.2/§3.3): the shared attestation layer and the PeerInfo-in-extensions contract are defined here; the broker consumes the same extension. The connect-vs-RPC seam resolution must be the same choice in both.
  • PDP gate + error mapping (§3.4/§3.5): the authorize(req_ext, op, key) → uid helper and the internal-error→Status table are shared; the broker maps its ops through the same helper, default-deny, same audit record. Unsupported key/algorithm → UNIMPLEMENTED in both.
  • Minting boundary (§4): keys never leave bao: the broker service must also sign in place and must not reintroduce an in-process signing key. The shared sign-in-place primitive lives in core/; SVID minting is service/spiffe, NATS/generic minting is service/minting (broker doc §6).
  • Header policy (§10): workload.spiffe.io=true is required by Workload API methods only; the broker methods MUST NOT require it.
  • Proto syntax (§9.1): both packages are proto3 in one basil-proto build.
  • Watch / revocation events (§8.3): the broker's AdminService.Watch and this doc's streaming bundle handlers consume the same internal rotation/bundle/revocation event source.
  • Migration (§12): both gRPC services land on the same socket in the flag-day cut; there is no legacy JSON socket.
