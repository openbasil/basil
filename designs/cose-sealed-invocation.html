COSE Sealed Invocation Conversion

A plan for replacing the bespoke SecureMessage sealed invocation envelope with COSE (RFC 9052 / RFC 9053): standard signed and encrypted CBOR messages carried through a deliberately thin gRPC and NATS envelope. This is a complete replacement of the wire format shipped by the subject-based-authentication milestones M2–M5, with no backwards-compatibility path.

This plan supersedes the wire format in subject-based-authentication §7.1–7.4 and §8.1 (the Header / SecureMessage envelope, canonical BLAKE3 transcript, and Invoke(SecureMessage) RPC). Everything else in that design — subjects, principals, PDP, audit, replay semantics, broker identity keys, and the courier bridge model — is unchanged and remains authoritative.

1. Summary

The shipped sealed invocation path invented three things COSE already standardizes: a signed header structure (Header + the canonical little-endian transcript), an AEAD binding rule (AAD = header transcript), and a signed-ciphertext envelope (SecureMessage). Each invention needs its own specification, fixtures, and cross-language reimplementation. COSE gives the same properties with registered algorithms, published test vectors, and third-party verifiers.

Core decision: a sealed invocation becomes a tagged COSE_Sign1 whose payload is a tagged COSE_Encrypt. Basil claims ride in the COSE_Encrypt protected header, so one placement is covered by both the sender signature and the AEAD tag and remains readable before decryption for replay and expiry preflight. The gRPC envelope shrinks to opaque COSE bytes plus one clear routing field (response_subject); the NATS payload becomes the raw COSE bytes with no protobuf wrapper at all.

The v1 cipher suite is the one the short-term use case needs: EdDSA (Ed25519, including NATS NKey-encoded principals) for signatures and ECDH-ES + HKDF-256 over X25519 with A256GCM for encryption. ML-KEM sealing is dropped from the invocation path until COSE codepoints exist (it survives unchanged in the separate WrapEnvelope/UnwrapEnvelope AEAD API). Interop is proven against the independent cose_minicbor crate plus imported COSE working-group vectors.

3. Target message format

3.1 Structure

18(                                ; COSE_Sign1 (RFC 9052 tag 18)
  [
    protected:   { 1: -8,                     ; alg = EdDSA
                   4: 'publisher.signing.2026q3' } ; kid = sender_sign_id
    unprotected: {},
    payload:     << 96(                       ; COSE_Encrypt (tag 96), embedded
      [
        protected: {
          1: 3,                               ; alg = A256GCM (content)
          3: "basil.sign-request",            ; content type of the plaintext body
          15: { 1: "content.publisher",       ; CWT claims: iss = sender subject (optional)
                3: "basil://prod/use1/agent-a", ; aud = recipient subject (optional)
                4: 1782740060,                ; exp (optional; effective iat+60 when absent)
                6: 1782740000,                ; iat
                7: h'0197...' },              ; cti = message id (bstr, sender-unique)
          -70003: "publisher.signing.2026q3", ; basil: sender key id (MUST equal outer kid)
          -70004: "publisher.response_encryption.2026q3", ; basil: response key id (requests)
          -70005: "_INBOX.abc123",            ; basil: response subject (requests, optional)
          2: [3, 15, -70003, -70004]          ; crit: all present basil-critical labels
        },
        unprotected: { 5: h'<12-byte IV>' },  ; AEAD nonce
        ciphertext:  h'<encrypted CBOR body>',
        recipients: [[
          protected:   { 1: -25 },            ; alg = ECDH-ES + HKDF-256
          unprotected: { -1: { 1: 1, -1: 4,   ; ephemeral COSE_Key: OKP, crv X25519
                               -2: h'<ephemeral public>' },
                         4: 'broker.request_encryption.2026q3' }, ; kid = recipient_key_id
          ciphertext:  h''                    ; empty: direct key agreement derives the CEK
        ]]
      ]
    ) >>,
    signature: h'<Ed25519 over Sig_structure>'
  ]
)

Sign-over-ciphertext is preserved: the broker verifies the outer signature and runs replay, expiry, and audience preflight before any decryption, exactly like today. The claims sit in the COSE_Encrypt protected header, which is plaintext CBOR (readable pre-decrypt), authenticated by the AEAD tag through Enc_structure, and covered by the outer signature because the whole COSE_Encrypt is the COSE_Sign1 payload. Nothing needs the old external transcript: Sig_structure and Enc_structure (both with empty external_aad) replace it.

Private labels use the RFC 9052 private-use range (below −65536), allocated in one place as a Rust constant registry. All basil labels present in a message, plus content type (3) and CWT claims (15), are listed in crit (2) so any conforming third-party consumer fails closed rather than silently ignoring them.

3.3 Algorithms

PurposeCOSE algorithmCodepointNotes
Envelope signatureEdDSA−8 Covers both ed25519 and nats-nkey principals: an NKey is an Ed25519 key in a different at-rest encoding, so on the wire there is one algorithm and the NKey encoding remains a policy-side public-key decoding detail.
Key agreementECDH-ES + HKDF-256−25 X25519 (OKP, crv 4), direct CEK derivation, fresh ephemeral per message. Matches the existing X25519 arm (ECDH + HKDF-SHA256 + AEAD) almost 1:1.
Content encryptionA256GCM3 Single suite for v1. ChaCha20/Poly1305 (24) can return later as a pure alg value change with no format break.
Request binding hashSHA3-256— Not a COSE codepoint; used only inside the -70002 response claim, so the choice is ours — take the newer primitive per the strict-validation decision (§11).

The broker validates that the wire alg values equal the algorithms implied by the catalog records for the named key ids and rejects mismatches before use. This preserves the old rule that algorithms resolve from configured key records: the wire field exists because COSE requires it, but it is checked, never trusted (§5, item 7).

ES256/ES384/ES512 (−7/−35/−36) are standard COSE and the catalog already has P-256/P-384/P-521 key types, but envelope principals stay Ed25519-family in v1, matching what is implemented today. Adding ECDSA envelope signing later is a policy/catalog change, not a format change.

3.4 Verification order and cross-checks

  1. Strict decode: tagged COSE_Sign1, definite lengths, RFC 8949 §4.2 deterministic encoding, no unknown labels outside the registry, all crit labels understood. Payload must be a tagged COSE_Encrypt; any other nesting is rejected.
  2. Resolve the outer kid against signature-key principals; verify the wire alg against the configured key; verify the signature over Sig_structure.
  3. Read claims from the (now signature-trusted) inner protected header: check aud against the configured audience, iat against skew, effective exp against max TTL, and cti against the replay cache. All before decryption, as today.
  4. Enforce -70003 (sender key id) equals the outer kid. A courier or attacker who strips the signature and re-signs the untouched ciphertext under its own key now fails here (and would additionally fail subject cross-checks), so signature-stripping cannot re-attribute a sealed request.
  5. Authorize Op::Decrypt for the resolved actor on the recipient key id; ECDH-decapsulate and AEAD-decrypt (the AEAD tag authenticates the protected headers via Enc_structure, so ciphertext transplant under altered claims fails).
  6. Strictly decode the plaintext body by content type (unknown body fields still rejected), then dispatch the operation and authorize it normally (Op::Sign, mint ops, …).

3.5 Responses

Same shape, roles swapped: the broker signs a COSE_Sign1 with the configured response-signing key (outer kid = response signing key id) over a COSE_Encrypt sealed to the request's validated -70004 response key. Response claims: iss = broker identity id, aud = requester subject when known, fresh iat/cti, -70001 = request cti, and -70002 = SHA3-256 of the exact request bytes. Content type selects the response body schema; InvocationStatus (OK / DENIED / INVALID_REQUEST / INTERNAL_ERROR), policy_generation, and operation results stay inside the encrypted body. Denials and errors are signed and encrypted; gRPC status stays transport-only; the fail-closed response-protection-failed path is unchanged.

Client verification: check broker signature against pinned trust, check alg, match -70001 to the sent cti, recompute and match -70002, confirm the recipient key is the requested response key, decrypt, then trust only the signed decrypted status. A response never reuses the request's nonce, and the fresh ephemeral per message replaces the old fresh-encapsulation rule.

4. Thin gRPC and NATS envelope

The old envelope forced every carrier hop through a structured protobuf Header even though the carrier routes on none of it. The replacement carries opaque COSE bytes plus the one field a courier genuinely needs for response routing: 

service InvocationService {
  // Submit a sealed COSE invocation message.
  rpc Invoke(SealedRequest) returns (SealedResponse);
}

message SealedRequest {
  // Tagged COSE_Sign1 request bytes.
  bytes message = 1;
}

message SealedResponse {
  // Tagged COSE_Sign1 response bytes.
  bytes message = 1;
  // Carrier subject to publish `message` to (for NATS, often the
  // requester's inbox). Echoed from the signed request's response-subject
  // claim; absent when the request named none.
  optional string response_subject = 2;
}
  • NATS payloads are raw COSE bytes in both directions — no protobuf wrapper on the message bus at all. Any COSE/CBOR tool can inspect the (public) envelope structure of a captured message.
  • Response routing: the requester states its response subject inside the signed, AEAD-bound request claim (-70005); Basil echoes it in clear on SealedResponse. The bridge publishes the response bytes there, falling back to the NATS request/reply inbox when absent. The bridge never parses COSE; a misrouting bridge is equivalent to a message-dropping bridge because payloads stay sealed either way.
  • Bridge behavior otherwise unchanged: size limit checks, queue-group subscription, op:invoke presenter grant, and the Basil-Bridge-Error/-Message/-Retryable headers for bridge-level failures without a sealed response.
  • gRPC status stays transport-only: disabled service, malformed/oversized envelope, or response-protection failure.

5. What we give up

Ordered roughly by weight. Items 1–3 are real capability or property changes; the rest are invariants that must be deliberately re-established in the new format rather than losses.

  1. ML-KEM sealing on the invocation path. The KemAlgorithm ML-KEM-512/768/1024 arms are removed from request/response sealing because COSE has no final ML-KEM or HPKE codepoints (only Internet-Drafts, e.g. draft-ietf-cose-hpke, as of this writing). Accepted for v1; the general-purpose WrapEnvelope/UnwrapEnvelope API keeps ML-KEM, so post-quantum envelope wrapping remains available outside invocation. When COSE HPKE/ML-KEM codepoints land, PQC sealing returns as a new recipient alg with no format redesign.
  2. Post-quantum signature headroom on the envelope. Nothing shipped uses it (envelope principals are Ed25519/NKey only), but the old bespoke format could have adopted ML-DSA-44/65/87 from the crypto provider at any time. Under COSE, ML-DSA envelope signatures and a PQC broker response-signing key wait on draft-ietf-cose-dilithium becoming an RFC, or on us knowingly shipping draft codepoints.
  3. BLAKE3, and digest-only signing. COSE registers no BLAKE3 codepoint. ECDSA arms use SHA-2 (matching the ES256/384/512 transit shaping that already exists), and Ed25519 becomes PureEdDSA over the full Sig_structure — so the sign-in-place call to the backend carries the whole structure (bounded by the existing max message size, ~1 MiB) instead of a 32-byte BLAKE3 digest. OpenBao/Vault transit signs arbitrary-length Ed25519 input, so this is a bandwidth/latency cost per bridged call, not a capability loss. The response request-binding hash moves from BLAKE3 to SHA3-256.
  4. The single flat transcript invariant. Today one canonical transcript is simultaneously the signature input and the AAD, so every header field is trivially covered by both. COSE splits coverage across Sig_structure and Enc_structure; the equivalent guarantee comes from deliberate placement (claims in the encrypted layer's protected header) plus the -70003-equals-outer-kid cross-check. Equivalent strength, but it is a design discipline to maintain rather than a structural given — hence the new adversarial tests in §9.
  5. Fail-closed unknown-field rejection by construction. The v1 envelope rejected unknown fields everywhere. COSE convention is to ignore unrecognized non-critical header parameters, and third-party libraries (including cose_minicbor) follow that. We keep the property by policy: our decoder rejects labels outside the registry, and every basil label rides in crit so conforming external consumers fail closed too. Body decoding stays strict and unchanged.
  6. The explicit envelope version field. COSE structures carry no version; discrimination collapses into alg values, the content-type registry, and crit-gated labels. Acceptable pre-release; a future incompatible profile would introduce new content-type strings.
  7. “No caller-supplied algorithm fields.” The old design resolved every algorithm from catalog key records and put none on the wire. COSE requires alg in protected headers. The invariant is preserved in weakened form: wire alg must equal the catalog-derived algorithm or the message is rejected, and algorithm-confusion becomes an explicit test class instead of an unrepresentable state.
  8. Protobuf-native envelope introspection. Header was inspectable with protobuf tooling and typed Go stubs. The envelope is now CBOR; debugging shifts to standard CBOR diagnostic tooling, which is arguably a net gain but is a tooling change for anyone scripting against the old shape.
  9. Sunk M2–M5 format artifacts. The canonical transcript specification, invocation-secure-message-v1.json fixture, transcript documentation in basil-doc, and the Go proto stubs for Header/SecureMessage are all discarded and replaced. Rework, not capability loss.

Explicitly not lost: replay protection, short expiry with signed defaults, audience binding, requester-selected response encryption keys, signed denials and errors, verify-before-decrypt preflight ordering, courier opacity (bridge never sees plaintext bodies in either direction), actor-vs-presenter audit, and the nats-nkey principal kind (it simply signs as EdDSA on the wire).

6. What we gain

  • Two RFCs replace three bespoke constructions (transcript encoding, AAD derivation rule, envelope struct) that each needed spec text, fixtures, and a promised cross-language port.
  • Independent verification: third-party COSE implementations can decode and verify basil messages, which is the basis of the §8 interop suite — something impossible for a bespoke format.
  • One CBOR encoder for envelope and bodies (bodies were already deterministic CBOR; the protobuf/CBOR split inside one message disappears).
  • Future room without format breaks: multi-recipient sealing is a recipients-array feature (via ECDH-ES + A256KW), ChaCha20 and ECDSA are alg-value changes, PQC returns via future registered codepoints.
  • Cheaper Go client parity later: mature Go COSE libraries (e.g. veraison/go-cose) replace a hand-ported transcript encoder.
  • Cross-project convergence: edgebox-rs's bespoke RPC uses the same two shapes (Ed25519-signed, and X25519-encrypted + Ed25519-signed). Converting it to COSE lets it consume basil-cose directly — one implementation, one interop suite, and shared fixtures guarding both projects instead of two bespoke formats evolving apart.

7. Implementation and rip-out inventory

7.1 Foundation

Decision: build basil's COSE layer on the coset crate (structure encode/decode only, crypto-agnostic — keys stay in providers/backends, nonces stay basil-generated), with basil-side strict-decode validation on top. C1 must verify that coset emits deterministic bytes for our structures; if it cannot, fall back to extending the existing hand-rolled deterministic CBOR encoder in basil-proto to the COSE structures. Do not build on cose_minicbor: it is the interop counterparty and must stay independent.

Decision: the profile lives in a new broker-free, publishable crate crates/basil-cose (precedent: basil-nats), not a module inside basil-proto. It provides exactly two constructions — signed (COSE_Sign1) and sealed (COSE_Sign1 over COSE_Encrypt) — under the strict deterministic profile, with signing and decryption behind traits so the broker backs them with backend sign-in-place while clients and external consumers use local keys. The crate owns the generic sealed-RPC machinery: the private label registry and claim structs (in-reply-to, request-hash, sender-key-id, response-key-id, response-subject), the skew/TTL/audience validation (bounds passed as parameters), the crit and strip-and-re-sign cross-checks, and the deterministic-encoding enforcement. Basil-specific values — the content-type strings, body schemas, and catalog key-id semantics — stay in basil-proto. Motivation: edgebox-rs carries a bespoke RPC with these same two shapes (Ed25519-signed; X25519-encrypted + Ed25519-signed) and should be able to adopt this implementation directly instead of maintaining a parallel format.

A review of the edgebox-rs protocol surfaces (the signed control RPC/push draft, the downloader command envelope, the enrollment sealed framing, and hub-endpoints) fixes four crate requirements: (1) caller-supplied external_aad on both Sig_structure and Enc_structure — edgebox binds the delivery NATS subject into control-RPC signatures and a purpose/version context into enrollment AEAD; basil invocation passes empty; (2) a seal-only bare COSE_Encrypt as a third construction — enrollment's outer layer is encrypt-only, with tamper evidence from the AEAD plus an inner signed envelope; (3) exposed PartyU/PartyV identity in the ECDH KDF context structure, the standard home for enrollment's direction-separating d2h/h2d HKDF info strings; (4) ChaCha20/Poly1305 content encryption from the start (§11). Detached payloads (nil COSE payload, supplied externally at verify time) are available in COSE if a future consumer needs to sign stored artifacts without embedding them; not a v1 requirement.

8. Interop test suite

8.1 Counterparty and scope

Primary counterparty: cose_minicbor (v0.1.x, from the suit-rs project) as a dev-dependency in the basil-tests crate (publish=false). It is a decode-and-verify-first, no_std COSE implementation on an unrelated codebase — exactly what an interop counterparty should be. Its current coverage shapes the suite: CoseSign1 verification with Ed25519 and ES256 features, structural decode, and feature-gated ECDH-ES/AES-KW decryption. Its X25519 ECDH-ES support must be confirmed at implementation time; its signing/encryption (produce) side is limited, so interop is primarily one-directional: basil produces, cose_minicbor independently consumes.

8.2 Test matrix

  • Structural decode: every message type in the content-type registry, request and response, round-trips basil-encode → cose_minicbor-decode with matching header parameters, claims, and recipient structure.
  • Signature interop: cose_minicbor verifies basil-produced Ed25519 COSE_Sign1 messages; every basil tamper vector (§9) must also fail verification in cose_minicbor, proving the two implementations agree on what is covered by the signature.
  • Decryption interop: if cose_minicbor's ECDH-ES feature supports X25519, it decrypts basil-sealed messages given the recipient private key. If it is P-256-only, this leg is covered by checked-in known-answer vectors instead, and we note the gap (or contribute upstream) rather than weakening our suite.
  • Inbound vectors: basil verifies/decrypts selected vectors from the COSE working group Examples repository (EdDSA COSE_Sign1, ECDH-ES cases with published key material), checked into fixtures, so basil is tested as a consumer of externally-produced COSE, not only of its own output.
  • Determinism: byte-identical re-encoding of every fixture (encode→decode→encode), guarding the deterministic profile that signatures depend on.

8.3 Fixtures

JSON fixtures replace invocation-secure-message-v1.json: hex of complete tagged COSE bytes per message type, the claim set, signer/recipient key material (test keys only), plaintext body hex, expected accept/reject verdict and rejection reason. These serve the same role the transcript fixtures served for the deferred Go client: the future Go implementation must consume them byte-for-byte.

9. Security tests

Every adversarial property from subject-based-authentication §12.3–12.5 is ported by re-expressing the mutation against COSE fields: tampering with content type, claims (iss/aud/iat/ exp/cti), key ids, ciphertext, ephemeral key, IV, or the response correlation claims must break the signature, the AEAD tag, or an explicit cross-check; replay, expiry, TTL, skew, audience, disabled-service, unknown-signer indistinguishability, and signed-denial coverage all carry over. New classes required by the format change:

  • Strip-and-re-sign: re-signing an intact COSE_Encrypt payload under a different valid principal's key is rejected by the -70003/outer-kid equality check (and never reaches policy as the new subject).
  • Algorithm confusion: wire alg values that disagree with the catalog key (signature, recipient, or content) are rejected before use; unknown or absent alg is rejected.
  • Nesting confusion: a COSE_Sign1 whose payload is not a tagged COSE_Encrypt (plain body, COSE_Encrypt0, COSE_Mac0, double-wrapped Sign1) is rejected at decode.
  • Criticality enforcement: messages missing crit, missing a basil label listed in crit, or carrying unknown labels (critical or not) are rejected; claims placed in unprotected headers are rejected.
  • Encoding strictness: indefinite lengths, non-deterministic map ordering, untagged structures, and duplicate labels are rejected.
  • Response subject abuse: SealedResponse.response_subject equals the signed request claim exactly; a response to a request without the claim carries none.

11. Decisions and deferred features

Resolved decisions

  1. Nesting is COSE_Sign1 over embedded tagged COSE_Encrypt: keeps verify-before-decrypt preflight and courier-visible-but-sealed structure.
  2. Claims live in the COSE_Encrypt protected header (one placement covered by signature and AEAD, readable pre-decrypt), with the sender key id duplicated there and checked against the outer kid.
  3. v1 suite: EdDSA / ECDH-ES+HKDF-256 (X25519) / A256GCM, single suite, no negotiation. Wire alg is validated against catalog records, never trusted.
  4. Strict validation wherever the RFCs allow flexibility — assume nothing. Enum-shaped fields (alg, kty, crv, content types, claim labels, tags) reject unknown variants; there are no forward-compatible Unknown arms in the profile's decoders. Unknown or duplicate header labels, claims in unprotected headers, indefinite lengths, non-deterministic encodings, untagged structures, and wrong CBOR types for a known label are all rejected. Where the RFCs leave a primitive choice open, take the more conservative option: the private request-hash claim uses SHA3-256 rather than SHA-2 (registered COSE signature/KDF codepoints fix their own hashes and offer no choice). This carries basil's secure-by-default, least-privilege, fail-closed principles into the profile.
  5. The carrier envelope carries exactly one routing field (response_subject), echoed from a signed request claim; NATS payloads are raw COSE bytes.
  6. Basil's implementation builds on coset (or the in-tree deterministic encoder if coset fails the determinism check) and lives in the standalone broker-free basil-cose crate, scoped to profile machinery so edgebox-rs can adopt it for its RPC; interop tests use cose_minicbor as an independent counterparty.
  7. Complete replacement: the SecureMessage format, transcript spec, and fixtures are deleted, not deprecated.

Deferred features

  1. ML-KEM / HPKE sealing for invocation — blocked on COSE codepoints (draft-ietf-cose-hpke et al.); returns as a recipient alg addition.
  2. ML-DSA envelope signatures — blocked on draft-ietf-cose-dilithium.
  3. ECDSA (ES256/384/512) envelope principals and broker response signing — standard already; policy/catalog work when needed.
  4. ChaCha20/Poly1305 (24) content encryption as a second suite. Likely wanted early: edgebox-rs's existing seal suite (x25519-hkdf-sha256-chacha20poly1305-v1) is exactly ECDH-ES + HKDF-256 with ChaCha20/Poly1305 content, so edgebox parity needs it; the basil-cose API should treat the content alg as a parameter from the start.
  5. Multi-recipient sealing via ECDH-ES + A256KW.
  6. Go sealed-invocation helpers (unchanged deferral; now targets a Go COSE library plus the §8.3 fixtures).

This is a conversion plan for Basil's pre-release sealed invocation API. It intentionally favors a clean breaking change over compatibility shims.
