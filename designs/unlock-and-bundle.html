0. TL;DR

The broker is a Unix-socket signing/secrets proxy in front of a Vault-compatible backend (OpenBao or HashiCorp Vault). To talk to its backends it needs its own bootstrap credentials (a Vault token / AppRole secret-id today; a db-keystore DEK or other backend creds later). Today those live in plaintext: a --vault-token flag, $VAULT_TOKEN, or ~/.vault-token. That is the gap §17.7 "Unlock & bootstrap" closes.

This document specifies:

  • a pluggable master unlock that recovers a master KEK, and
  • a 0600 multi-slot sealed bundle the KEK opens, holding a versioned map of per-backend credentials.

Each unlock method is an independent slot that wraps the same master KEK. Methods: age-yubikey-plugin (operator-presence), bip39 (break-glass recovery), passphrase (file-sourced Argon2id passphrase), tpm (later). Adding or removing a method rewraps one slot; it never re-encrypts the payload.

This is not the data plane. Data-plane secrets (signing keys, NATS seeds, opaque values) live in the Vault-compatible backend (transit + KV v2); backend seal/unseal + KV versioning already handle DEK/KEK/rotation for data (§17.1). The sealed bundle is a small local store for the broker's own bootstrap creds, nothing else.

1. Problem statement & constraints

1.1 What the broker must hold at rest, and why

The broker authenticates to each backend before it can serve any request. Those backend credentials cannot themselves live in a backend the broker can only reach after it authenticates: a bootstrap chicken-and-egg. They must be present locally at process start. Today (crates/basil-agent/src/main.rs) they are:

CredSource todaySensitivity
Vault-compatible token (X-Vault-Token)--vault-token / $VAULT_TOKEN / ~/.vault-tokenFull transit sign/mint authority for every key the broker fronts.
AppRole secret_id (planned)not yet wiredEquivalent: exchanged for a token.
SPIFFE/JWT-SVID signing key (planned, --backend spiffe)implicit/self-issued todayUsed to obtain a short-lived bao token via the jwt auth method.
db-keystore DEK (future value-store backend, §17.7)n/a yetDecrypts a local encrypted DB; equivalent to all values in it.

Every one of these is a long-lived, high-value secret. A leaked bao token is the keys to the transit engine. Storing it as a world-or-owner-readable plaintext file or in an environment variable (visible via /proc/<pid>/environ, leaked into core dumps, inherited by children) is the weakness this design removes.

The sealed bundle is the single local at-rest home for all of these. The broker unlocks it once at startup, hands each credential to the backend that needs it, and zeroizes the plaintext.

This implementation replaces the old plaintext path outright. vault-vh1 removes the --vault-token / $VAULT_TOKEN / ~/.vault-token loading and the implicit SPIFFE-flag handling in crates/basil-agent/src/main.rs. There is no compatibility fallback to plaintext bootstrap creds: leaving one in place would be permanent technical debt and would defeat the at-rest protection this design exists to provide. The sealed bundle becomes the only supported source of bootstrap credentials.

1.2 Threat model

In scope. The bundle file and the sealed-method state sit on the broker host's persistent disk. We assume an attacker may obtain:

  1. The 0600 bundle file itself (backup theft, disk clone, snapshot, lost laptop).
  2. Read access to the broker's runtime as a non-root, non-broker user.

What an attacker with the bundle file but no unlock method gets: nothing usable. The payload is AEAD-encrypted under a key (the master KEK) that exists only after a successful unlock. Each slot wraps that KEK under a method-specific key the attacker does not have:

  • age-yubikey slot: the master KEK is wrapped to an age recipient whose private half is on a YubiKey (PIV/FIDO2). Without the physical token and its PIN/touch, the slot is opaque. A cloned disk does not clone the token (matches §9 removable-token / operator-presence).
  • bip39 slot: wrapped under a key derived (memory-hard KDF) from a 24-word recovery phrase the attacker does not hold (it is printed and stored offline, never on the host).
  • tpm slot (later): sealed to the TPM + PCR policy of this host; non-exportable, does not move with a disk image.
  • passphrase slot: the master KEK is wrapped under an Argon2id key derived from a passphrase read from a 0600 file or systemd credential file. This is production-supported, but operators must protect the file source; Basil best-effort wipes the file after startup unless unlock-passphrase-no-wipe is configured.

Explicitly out of scope (document, do not claim to defend):

  • An attacker who is root on a running, already-unlocked broker can read the live bao token from the broker's memory or impersonate it on the socket. The bundle defends at-rest state, not a compromised live host. (This mirrors §13's stance and §17.3's "only path to the backend" caveat.)
  • A stolen device with the YubiKey attached / PIN cached unlocks, documented in §9 removable-token risk. operator-presence (touch-per-boot) and tpm raise this bar.
  • Physical full-disk rollback to an older bundle (anti-rollback for the bundle itself is §6.4, best-effort; it is not a TPM-monotonic guarantee, the same scope caveat as §13.1).

1.3 Constraints

  • Secure by default, least privilege (§17 guiding principles). Bundle file is mode 0600, owned by the broker user, in the broker's private runtime/state dir. No env-var or argv path for the unwrapped creds.
  • Provider-neutral / generic. No edge or "realm" terms. Backend ids are opaque strings.
  • zeroize the master KEK and every materialized credential as early as possible (§5, §17.7 "Audit & process" already mandates zeroize for materialized key bytes).
  • Crypto with no footguns (§17 principle): broker-owned AEAD nonces, no caller-nonce path, versioned format, deny-by-default builds (unsafe_code forbid, pedantic/nursery deny per .agent-notes).
  • No runtime panics: fail closed, never crash. The broker is an unrestartable service: a crash, panic, or aborted process halts the entire signing/secrets plane (there is no supervisor that can safely restart it mid-flight). Therefore the unlock/crypto code (and all broker runtime code) must contain no runtime panics: no unwrap(), expect(), panic!, unreachable!, todo!, unimplemented!, panicking slice/array indexing, or arithmetic overflow on any runtime path. Every fallible step returns a Result and fails closed gracefully: a failed unlock produces a clean non-zero exit, never a panic or abort. The only sanctioned process exit is the SIGINT/SIGQUIT shutdown path. This is enforced repo-wide by clippy restriction lints (unwrap_used / expect_used / panic / unreachable / todo / unimplemented / indexing_slicing set to deny, allowed only in #[cfg(test)] code).
  • Must compose with the existing tpm2-style feature gating already in the workspace (crates/basil-agent/Cargo.toml).

2. Sealed-bundle on-disk format

2.1 Goals of the layout

  • Wrapped-DEK / multi-slot envelope (§8): one symmetric master KEK encrypts the payload; N slots each independently wrap that KEK under a different unlock method. Any one slot recovers the KEK (this is how the bundle supports "two or more recovery methods"; see §16 Hub recovery).
  • Adding/removing a method = add/remove a slot, without re-encrypting the payload.
  • Self-describing & versioned so the opener can refuse formats it does not understand.
  • Single file, atomically replaceable (write temp in same dir, fsync, rename).

2.2 File container

The bundle is a single file at e.g. ${STATE_DIR}/bundle.sealed (0600). Encoding is JSON: a fixed ASCII magic + u16 version prefix, then a JSON body (header + slot table + sealed payload), so the file is greppable and the version is checkable before any deserialization. Binary fields (nonces, ciphertexts, salts, the bundle_id) are base64, no padding (URL_SAFE_NO_PAD, the same alphabet basil-nats uses elsewhere in the tree). JSON uses serde_json, already a workspace dependency: this adds no new container dependency (CBOR was dropped for exactly this reason; §7). Conceptually three regions:

+-----------------------------------------------------------+
| MAGIC  "BASILBDL\x00"        (9 bytes, fixed)             |  framing prefix (pre-JSON)
| format_version  u16          (= 1)                       |  (cleartext; version checkable
+-----------------------------------------------------------+   before any JSON parse)
| JSON BODY:                                                |
|   "header": {                                             |  header object: the EXACT on-disk
|     format_version, suite ids (2.5),                     |  bytes of this object are the AAD
|     bundle_id (b64-nopad, 16B), created_unix,            |  (read back verbatim; never
|     epoch (monotonic; anti-rollback)                     |  re-serialized, see below)
|   },                                                      |
|   "slots": [ Slot, ... ]   (slot table, 2.4),           |  one entry per unlock method
|   "payload": SealedPayload  (2.3)                        |  the per-backend cred map (§4)
+-----------------------------------------------------------+

The header is bound as AAD over both the payload AEAD and every slot's KEK-wrap AEAD, so an attacker cannot downgrade format_version, swap the suite ids, or splice slots/payload from a different bundle_id/epoch without breaking authentication. To avoid any JSON-canonicalization ambiguity, the AAD is the literal on-disk header bytes: the opener reads the exact byte span of the header object from the file and feeds those bytes to the AEAD. The header is never re-serialized for verification, so whitespace/key-ordering differences cannot affect the tag.

2.3 Sealed payload

struct SealedPayload {
    nonce:      [u8; 12],   // base64-nopad; broker-generated, fresh random per encryption (never caller-supplied)
    ciphertext: Vec<u8>,    // base64-nopad; AES-256-GCM(key = master_kek, aad = header_bytes) over the JSON cred map (§4)
    // 16-byte GCM tag is appended to ciphertext by the AEAD
}
  • AEAD: AES-256-GCM. The master KEK is a 32-byte symmetric key. Each encryption draws a fresh 12-byte (96-bit) nonce from the OS CSPRNG: the nonce is broker-owned; there is no caller-supplied-nonce path. A random 96-bit nonce is safe here because the bundle is re-sealed only rarely (re-seal / rotate / add-slot, never on a hot path), so the birthday-bound risk of nonce collision under one key is negligible at this frequency. This also matches the AES-256-GCM symmetric primitive the wire protocol already uses (.agent-notes wire-v2).
  • The payload is encrypted once under the master KEK. Slots never touch the payload; they only wrap the KEK. This is the property that makes add/remove-method cheap.

2.4 Slot table

Each slot independently wraps the same 32-byte master KEK:

struct Slot {
    slot_id:    u32,            // stable id, for add/remove/CLI reference
    method:     MethodKind,     // tagged enum: AgeYubikey | Bip39 | Passphrase | Tpm  (2.6)
    label:      String,         // operator-facing, e.g. "primary-yubikey-5C", "break-glass-2026"
    created_unix: u64,
    params:     MethodParams,   // method-specific public material (NOT secret); see below
    wrap:       KekWrap,        // how the master KEK is wrapped for this method
}

struct KekWrap {
    // For methods that yield a 32-byte slot key (bip39, passphrase, tpm):
    //   AES-256-GCM(key = slot_key, aad = header_bytes || slot_id) over the master KEK.
    nonce:      [u8; 12],       // base64-nopad; fresh random per wrap from the OS CSPRNG
    ciphertext: Vec<u8>,        // base64-nopad; wrapped master KEK + 16-byte GCM tag
}

MethodParams holds only non-secret material the method needs to reconstruct its slot key:

Methodparams contentswrap shape
AgeYubikeythe age recipient string (public); optional plugin/identity hintThe master KEK is encrypted with age to that recipient. wrap.ciphertext is the age stanza/file; no separate slot_key; age does the wrap. nonce unused (age manages its own).
Bip39KDF salt (16B), Argon2id params (mem = 64 MiB, t = 3, p = 1)slot_key = Argon2id(mnemonic_entropy, salt, params); then AES-256-GCM-wrap the master KEK under slot_key.
PassphraseKDF salt (16B), Argon2id params (mem = 64 MiB, t = 3, p = 1)slot_key = Argon2id(file_passphrase, salt, params); then AES-256-GCM-wrap the master KEK under slot_key.
Tpm (later)PCR selection, sealed-blob handle/public areaTPM unseal yields slot_key (or unseals the KEK directly). AES-256-GCM-wrap as above.

Why age for the yubikey slot, not a hand-rolled wrap: age-plugin-yubikey is the §16 first-class decision ("Support PIV and age-plugin-yubikey first"). Reusing age's recipient stanza for the KEK-wrap means the YubiKey slot is exactly an age decryption: well-reviewed, handles PIV/touch/PIN, and is what operators already use. The other methods produce a raw 32-byte slot_key and go through the broker's own AES-256-GCM wrap for uniformity.

2.5 Suite ids (versioning)

format_version = 1 pins the suite. The header carries small enum ids so a future v2 can change a primitive without ambiguity, and the opener refuses unknown ids:

  • container = json (ASCII magic + u16 version prefix + JSON body; binary fields base64-nopad)
  • payload_aead = aes-256-gcm
  • kek_wrap_aead = aes-256-gcm
  • recovery_kdf = argon2id (mem = 64 MiB, t = 3, p = 1), used by bip39 / passphrase
  • age_recipient (yubikey slot)

No silent algorithm negotiation; one named suite per format_version.

2.6 Method enum (wire/struct)

enum MethodKind { AgeYubikey, Bip39, Passphrase, Tpm }

Passphrase is a production file-sourced Argon2id method. Tpm is a reserved variant that returns NotImplemented until the TPM slot lands (§9 / §16 require fail-closed for tpm-bound before support exists).

2.7 Add / remove slots

  • Add: unlock via any existing slot → recover master KEK in memory → build the new slot's KekWrap (e.g. age-encrypt KEK to the new YubiKey recipient) → append Slot → atomically rewrite the file. Payload untouched. (epoch unchanged, since content didn't change, only access.)
  • Remove: drop the Slot from the table → atomically rewrite. Refuse to remove the last slot (would brick the bundle); CLI requires --force-empty only with an explicit out-of-band confirmation. Production policy SHOULD warn when removing the last recovery slot, mirroring §16 ("prompt for two or more recovery methods").

3. Unlock-method abstraction (Rust trait sketch)

Sketch only: vault-vh1 owns the implementation in crates/basil-agent (or a new crates/basil-seal crate). Mirrors the existing Backend trait style (crates/basil-agent/src/core/backend/mod.rs): a small object-safe trait, errors as a thiserror enum, async not required (unlock is local and one-shot, but a method may shell out to age/TPM, so keep it sync + blocking, run off the async accept loop at startup). Every method below returns a Result on every fallible step: no unwrap/expect/panicking index anywhere (§1.3).

/// 32-byte symmetric key material that zeroizes on drop.
/// (Newtype over `zeroize::Zeroizing<[u8; 32]>`.)
pub struct KeyMaterial(/* zeroizing [u8; 32] */);

#[derive(Debug, thiserror::Error)]
pub enum UnlockError {
    #[error("method unavailable: {0}")]      Unavailable(String),   // no token plugged, no TPM, ...
    #[error("authentication failed")]         AuthFailed,            // wrong PIN / phrase / disk key
    #[error("method not implemented: {0:?}")] NotImplemented(MethodKind), // tpm (later)
    #[error("feature not enabled: {0:?}")]    FeatureDisabled(MethodKind),// disk in prod build
    #[error("crypto: {0}")]                   Crypto(String),
    #[error("io: {0}")]                       Io(#[from] std::io::Error),
}

/// One unlock method = the logic behind one slot kind.
/// Implementations are stateless w.r.t. the master KEK; they only know how to
/// turn their `params` + an out-of-band secret (token touch / phrase / key file)
/// into the ability to unwrap *this slot's* `KekWrap`.
pub trait UnlockMethod: Send + Sync {
    fn kind(&self) -> MethodKind;

    /// Is this method usable *right now* on this host? (token present, TPM node,
    /// key file readable). Used to order attempts and to report status.
    fn available(&self) -> bool;

    /// Recover the master KEK from one slot. The method performs whatever
    /// interaction it needs (age/YubiKey touch, prompt for phrase, TPM unseal)
    /// and returns the unwrapped master KEK, or an `UnlockError` (never panics).
    fn recover_kek(
        &self,
        slot: &Slot,
        header_aad: &[u8],
    ) -> Result<KeyMaterial, UnlockError>;

    /// Build a fresh `KekWrap` for this method over an existing master KEK
    /// (used by add-slot / re-seal). `params` captures the public material.
    fn wrap_kek(
        &self,
        kek: &KeyMaterial,
        header_aad: &[u8],
    ) -> Result<(MethodParams, KekWrap), UnlockError>;
}

3.1 How each method recovers the KEK

  • AgeYubikey (unlock-age-yubikey, default-on): recover_kek feeds slot.wrap.ciphertext (the age stanza) to an age decryptor configured with the YubiKey identity (via age-plugin-yubikey); the plugin drives PIN/touch. Output is the master KEK. wrap_kek age-encrypts the KEK to the slot's recipient.
  • Bip39 (unlock-bip39, default-on): prompt for the 24-word phrase on a controlled input (tty / systemd AskPassword / agent socket, never argv/env). Validate the BIP39 checksum, derive entropy, Argon2id(entropy, slot.params.salt, mem = 64 MiB, t = 3, p = 1) → slot_key, AES-256-GCM-unwrap the master KEK.
  • Passphrase: read a passphrase from a 0600 file or systemd credential file into zeroizing memory, optionally best-effort wipe the file, then Argon2id(passphrase, slot.params.salt, mem = 64 MiB, t = 3, p = 1) → slot_key → AES-256-GCM-unwrap.
  • Tpm (unlock-tpm, later): tpm available? check reusing the existing identity::tpm_available() probe; TPM2_Unseal under the PCR policy → slot_key/KEK. Until implemented, recover_kek/wrap_kek return NotImplemented (fail-closed, §9).

3.2 Method registry / policy

The broker builds an ordered list of enabled UnlockMethods from config (which maps onto §9 VaultUnlockPolicy: operator-presence → require an interactive method like age-yubikey/bip39; passphrase → file-sourced passphrase; tpm-bound → tpm). On startup it walks the bundle's slot table, and for each slot whose method has an enabled, available() handler, attempts recover_kek until one succeeds. Default-deny: if no slot can be opened, the broker fails to start (it has no way to reach its backends) rather than serving an empty/degraded data plane.

3.4 Passphrase file source

Passphrase slots are always available in production builds. The agent reads [unlock].unlock-passphrase-file into zeroizing memory, derives the slot key with the slot-stored Argon2id salt and params, and best-effort wipes the file after a successful read. Set [unlock].unlock-passphrase-no-wipe = true for read-only credential mounts such as systemd credentials. Wipe failures warn but do not block startup after the passphrase has been read.

4. Per-backend cred payload schema

The decrypted payload (the AEAD plaintext, §2.3) is a JSON map: backend id → credential. Backend ids are the same opaque strings the catalog routes keys to (§17.7 "per-key routing"). The broker hands creds[backend_id] to the backend constructor at startup (§5).

struct CredBundle {
    schema_version: u16,                       // independent of format_version
    backends: BTreeMap<String, BackendCred>,   // "vault-transit" -> ...
}

enum BackendCred {
    /// Vault-compatible static token (today's `--vault-token`). Dev / simple setups.
    VaultToken { token: Zeroizing<String>, addr: Option<String> },

    /// Vault-compatible AppRole: secret-id (+ role-id) exchanged for a short-lived token at startup.
    /// Recommended production default (see below).
    VaultAppRole { role_id: String, secret_id: Zeroizing<String>, addr: Option<String> },

    /// SPIFFE/JWT-SVID signing key for the `--backend spiffe` path: a private key
    /// the broker uses to self-issue an SVID, exchanged via bao `jwt` auth.
    SpiffeSigner { key_pem: Zeroizing<String>, spiffe_id: String },

    /// Future value-store backend (db-keystore): the DEK that opens the local DB.
    DbKeystoreDek { dek: Zeroizing<[u8; 32]> },

    /// Escape hatch for a backend kind not yet modelled: opaque labelled bytes.
    Opaque { kind: String, secret: Zeroizing<Vec<u8>> },
}

Static token vs AppRole: why AppRole is the production default. A VaultToken is one long-lived bearer secret: the broker sends it on every call, and if it leaks, the holder has the token's full authority until it expires or is manually revoked (a large blast radius). VaultAppRole splits the credential into a non-secret role_id and a secret secret_id; at startup the broker exchanges the pair with the backend's Vault-compatible AppRole auth method for a short-lived token that it then uses for calls. So the secret sitting at rest in the bundle is short-TTL / single-use-capable / independently revocable, which means a leak of the bundle's secret_id has a much smaller blast radius than a leaked static token. For that reason AppRole is the recommended pattern for production hubs; the static VaultToken is retained for dev and simple/single-operator setups.

Notes:

  • Every secret field is Zeroizing so the whole decrypted payload zeroizes on drop, not just the KEK.
  • addr is optional per-cred so the bundle can pin which bao a token is for; otherwise the broker's --vault-addr/$VAULT_ADDR applies.
  • This map is the only place these creds live at rest. The CLI to seed it (init / set-cred) is part of vault-vh1; it reads a token/secret-id from a controlled input and writes a new sealed payload.

5. Startup flow

This flow replaces the old plaintext bootstrap-cred loading in crates/basil-agent/src/main.rs entirely (§1.1): the --vault-token / $VAULT_TOKEN / ~/.vault-token and implicit-SPIFFE paths are removed, with no compat fallback. Every step is fallible-by-Result and fails closed without panicking (§1.3): any error → clean non-zero exit.

1. Load bundle file (0600); parse + verify MAGIC, format_version, suite ids.
   - bad perms (not 0600 / wrong owner) -> warn-or-refuse per policy.
2. Build enabled UnlockMethod registry from config (§3.2 / §9 policy).
3. For each slot with an enabled, available() method (ordered):
     kek = method.recover_kek(slot, header_aad)?   // touch / phrase / unseal
     on first success -> break
   No slot opens -> FAIL CLOSED (clean non-zero exit; broker cannot reach backends).
4. payload = AES-256-GCM_decrypt(kek, header_aad, sealed_payload)
   -> CredBundle (§4).            // authenticity failure also fails closed (no panic).
5. ZEROIZE kek immediately (no longer needed after payload decrypt).
6. For each (backend_id, cred) in CredBundle.backends:
     construct the Backend (e.g. VaultBackend::new(addr, token, mount),
     or exchange AppRole role_id+secret_id for a short-lived token first)
     handing it its cred.  A down/invalid backend fails only that key's ops,
     not the broker (§17.7 routing).
7. ZEROIZE the CredBundle once every backend holds what it needs
   (VaultBackend stores its own copy of the token; db-keystore holds the DEK).
8. Drop bundle plaintext. Proceed to bind the socket / accept loop.
  • No panics on any path (§1.3): bad magic, version mismatch, malformed JSON, base64 decode failure, a wrong-length nonce/key, AEAD tag mismatch, or "no slot opened" all return errors that propagate to a graceful non-zero exit. There is no unwrap/expect/panicking index in this path; the only sanctioned exit besides a fail-closed startup error is the SIGINT/SIGQUIT shutdown.
  • Zeroize discipline (§17.7 "Audit & process" mandates zeroize for materialized key bytes): master KEK zeroized right after step 4; full CredBundle zeroized after step 7; any transient buffers (age output, KDF output) use Zeroizing. The broker keeps only what each backend intrinsically must hold live (e.g. the Vault token inside VaultBackend).
  • The bundle is not kept open. Re-reads (re-seal, add-slot) re-run unlock. There is no in-memory long-lived KEK after startup.
  • Audit (§17.7): emit a JSONL event {op:"unlock", slot_id, method, outcome}, never the cred bytes.

6. Recovery & rotation flows

6.1 Break-glass with bip39 (recovery)

At init, the broker generates a 24-word BIP39 phrase, derives the recovery slot_key (Argon2id, mem = 64 MiB, t = 3, p = 1), wraps the master KEK into a Bip39 slot, and displays the phrase once for the operator to print/store offline (§16: "optional printed BIP39 passphrase"). The phrase is never written to the host. If the primary (YubiKey/TPM) method is lost (token destroyed, motherboard replaced), the operator runs unlock --method bip39, types the phrase on a controlled input, and the bundle opens. Recommended posture (§16 "two or more recovery methods"): every production bundle has ≥1 primary + ≥1 break-glass slot; CLI status warns if only one slot exists.

6.2 Key rotation / re-seal (rotate the master KEK)

Rotating the KEK is needed if the KEK is suspected exposed, or as periodic hygiene. Because the payload is encrypted under the KEK, rotating the KEK re-encrypts the payload (small, just the cred map), and rewraps every slot. Each new encryption draws a fresh CSPRNG nonce (§2.3); these re-seal events are rare and operator-driven, which is why a random 96-bit nonce is safe:

re-seal(reason):
  kek_old   = unlock()                       // via any slot
  payload   = decrypt(kek_old)               // CredBundle
  kek_new   = 32 bytes from the OS CSPRNG
  for slot in slots:
     (params, wrap) = method(slot).wrap_kek(kek_new, header_aad')   // re-touch YubiKey, re-derive bip39, ...
     slot.params, slot.wrap = params, wrap
  sealed_payload' = encrypt(kek_new, header_aad', payload)          // AES-256-GCM, fresh 12B nonce
  epoch += 1                                  // anti-rollback (§6.4)
  atomically replace file
  ZEROIZE kek_old, kek_new, payload
  • Rewrapping a slot may require the operator to re-present that method (YubiKey touch, re-enter the bip39 phrase to re-derive its key). The CLI walks the operator through each slot.
  • Rotating a backend credential (e.g. a new bao token / secret_id after revocation) is the same re-seal, except the step replaces an entry in the payload map rather than the KEK. set-cred <backend> <...> → decrypt → mutate map → re-encrypt under the same KEK (no slot rewrap needed) → epoch += 1.

6.3 Distinction from data-plane rotation

This is bundle/bootstrap rotation only. Data-plane key rotation (transit key versions, 7-day retention, grace periods; see §17.7 "Key lifecycle") is the Vault-compatible backend's job and is out of scope here. The bundle KEK has no fixed retention; rotate on compromise or policy.

6.4 Anti-rollback (epoch sidecar, ratified MVP decision)

epoch is a monotonic counter in the header, bound as AAD. Bumped on every content change (re-seal, cred rotate, remove-slot). For the MVP this is implemented as a 0600 sidecar file holding the last-seen epoch: the broker persists epoch outside the bundle and refuses a bundle whose epoch is lower: catching a swap-in of an old bundle that still has a revoked token. This is logical anti-rollback only (matches §13.1 scope); it is not a defense against a full-disk rollback of host + sidecar together. TPM NV-counter binding is deferred as the future hardening (same as §13.1); the sidecar is the accepted MVP mechanism.

8. Decisions

The crypto-primitive and dependency choices below were reviewed and ratified by the operator on 2026-06-22. They are baked into format_version = 1 and override any earlier proposal text in this document.

  1. Payload + KEK-wrap AEAD = AES-256-GCM. Each encryption uses a fresh 12-byte (96-bit) nonce from the OS CSPRNG; the nonce is broker-owned, with no caller-supplied-nonce path. A random 96-bit nonce is safe at the bundle's rare re-seal frequency (re-seal / rotate / add-slot only). Master KEK = 32 bytes. (Pins the §2.5 suite ids payload_aead = aes-256-gcm, kek_wrap_aead = aes-256-gcm.)
  1. Recovery KDF = Argon2id, mem = 64 MiB, t = 3 (time cost), p = 1 (parallelism). Argon2id cost is memory + a small time cost + parallelism, not a large iteration count. This profile is a sound boot-time default for keys derived from a human recovery phrase or file-sourced passphrase, and is baked into the Bip39 / Passphrase slot params for format_version = 1.
  1. YubiKey slot = age + age-plugin-yubikey. Confirmed; age is pulled into the broker's dependency tree (along with bip39 and argon2). Reuses a reviewed tool that already handles PIV/PIN/touch (§2.4, §3.1).
  1. AppRole is the recommended production default; static token is for dev / simple setups. A static VaultToken is a single long-lived bearer secret with a large blast radius if leaked; AppRole keeps a non-secret role_id + a secret secret_id and exchanges them at startup for a short-lived token, so the secret-at-rest is short-TTL / single-use-capable / independently revocable. Production hubs use AppRole (§4).
  1. Anti-rollback = 0600 epoch sidecar, acceptable for MVP. TPM NV-counter binding is deferred to a later hardening pass (consistent with §13.1). See §6.4.
  1. Container = JSON, CBOR dropped. A fixed ASCII magic + u16 version prefix followed by a JSON body (header + slot table + sealed payload); binary fields are base64 no-pad (URL_SAFE_NO_PAD, matching basil-nats). The AAD is the literal on-disk header bytes (read back as bytes, never re-serialized), which sidesteps JSON non-canonicalization. JSON uses serde_json, already in the tree, so this adds no new dependency: the reason CBOR was dropped (§2.2, §7).
