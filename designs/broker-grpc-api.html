Broker gRPC API for Basil

An idiomatic gRPC re-design of the broker functionality (the key lifecycle, signing, AEAD, KV, minting, and admin operations) sitting as a sibling to the SPIFFE Workload API services on one tonic server over one Unix socket. This document specifies the implemented basil-proto contract, the op-by-op mapping from the now-archived JSON wire, the error model, the minting boundary, the admin event stream, and the flag-day migration. The shared transport (server, attestation, PDP middleware) is owned by spiffe-workload-api.html §3 and only referenced here (§5).

Compatibility with the JSON wire: none, flag-day. The length-prefixed-JSON socket is removed in the same change that lands gRPC (§8). The provider-neutral core, no-panic gate, edition 2024, and “keys never leave the Vault-compatible backend” invariants are unchanged. The point of this issue is not a mechanical port: a 1:1 transcription of the JSON tagged union would throw away per-RPC typing, bytes, canonical status codes, the rich error model, and streaming. The surface is re-designed to be gRPC-idiomatic while preserving every security property the JSON handler enforced.

1. Problem statement & motivation

The broker historically spoke a length-prefixed-JSON, externally-tagged-enum protocol over a Unix socket: tagged-union request/response, strict request/response lock-step per connection, a closed error-code enum, and binary carried as JSON number arrays. That protocol was fully built and mock-tested Implemented.

Phase 2 adds a SPIFFE Workload API gRPC service (X.509-SVID + JWT-SVID issuance, vault-gqn). Rather than run a second transport, the broker ops are also re-exposed over gRPC so the whole node presents one coherent surface: one tonic server over one Unix listener, multiplexing the SPIFFE services and the broker services, all reusing the same per-connection SO_PEERCRED attestation and the same default-deny PDP. The JSON broker is deleted in the same cut (§8).

Design goals (and the invariants they must not break)

  1. Idiomatic, not transcribed. Per-op request/response messages; well-named RPCs grouped into logical services; bytes for binary; google.rpc.Status + error-details instead of an error-code enum; server-streaming where it fits.
  2. Same attestation + same PDP, per-RPC. Every key-scoped RPC is gated by the PDP decision against the kernel-attested uid, default-deny, exactly as the JSON handler did. No new trust anchor.
  3. Preserve the leak-proof properties. “forbidden” and “not found” stay indistinguishable on a PDP deny (no key-existence oracle); List is per-key PDP-filtered; decrypt failure stays opaque; Get never returns crypto-key material; minted private material never crosses the wire. Non-negotiable (§4.3, §6).
  4. Keys never leave the Vault-compatible backend. gRPC changes the transport, not the sign/mint/encrypt-in-place backend contract.
  5. Evolvable. Field-number discipline, reserved ranges, explicit presence, and a package-version story so a second algorithm or op is additive.

2. Service split: five services, one package

Decision: split the broker into five focused services rather than one monolithic Broker service.

SigningService : NewKey, Import, ImportSet, Sign, Verify, GetPublicKey (asymmetric key lifecycle + use)
AeadService    : Encrypt, Decrypt                                     (symmetric AEAD, broker-owned nonce)
SecretService  : GetSecret, SetSecret, RotateSecret, ListCatalog      (secret values + catalog listing)
MintingService : MintJwt + typed MintNats* + IssueCertificate         (NATS / generic JWT / TLS cert, NOT SVID, §6)
AdminService   : Status, Watch                                        (discovery / health / event stream §7)

All five live in one proto package, basil.broker.v1, generated into the basil-proto tonic module, and are registered on the same tonic server alongside the SPIFFE services (basil.workload.v1) by the shared foundation's router (§5). They are “the broker,” addressed as a unit; the split is an organisational device, not a deployment boundary.

Why split, not one Broker service

  • The PDP already partitions the ops by capability Implemented. The policy roles are exactly these clusters: signer = {sign, verify, get_public_key}, cryptor = {encrypt, decrypt}, reader/operator = the KV ops, minter = {mint}. The service boundaries mirror the role boundaries, so “this credential can reach only the Signing service” is a natural statement.
  • Different evolution cadences. AEAD suites, KV semantics, and JWT presets evolve independently; separate services let one grow without versioning the others.
  • Generated-client ergonomics. A consumer that only signs links a signing client and never sees encrypt/mint.
  • It reads better. A service is a cohesive set of related RPCs; one service with a dozen unrelated RPCs is the anti-pattern the JSON tagged union already was.

Keeping all five in one package + one tonic build keeps the shared message types (KeyType, CiphertextEnvelope, the error details) in a single compilation unit and versioning coordinated. The SPIFFE services stay in their own package because one of them is a wire-compatible implementation of the upstream SPIFFE proto and must not be entangled with broker-private types.

3. The .proto

This is the implemented protobuf contract, not prose. It is proto3 with optional for explicit presence, matching the vendored SPIFFE workloadapi.proto, so one tonic-build invocation in basil-proto builds both packages under one syntax (§7, §8; and spiffe doc §9.1). Field numbers leave reserved ranges for the known-future enum values. 

syntax = "proto3";

package basil.broker.v1;

import "google/protobuf/duration.proto";   // ttl as a Duration
import "google/protobuf/timestamp.proto";  // expires_at / event time as a Timestamp
import "google/protobuf/struct.proto";     // free-form JWT claims (§3, §9 Q5)
import "google/rpc/status.proto";          // referenced by the error model (§4)

// ===== Shared enums (package scope): wire-self-describing algorithm slots.
// A non-MVP value is UNIMPLEMENTED with an ErrorInfo reason (never a reshape).

enum KeyType {
  KEY_TYPE_UNSPECIFIED  = 0;   // rejected as INVALID_ARGUMENT
  KEY_TYPE_ED25519      = 1;
  KEY_TYPE_ED25519_NKEY = 2;   // NATS NKey envelope
  KEY_TYPE_RSA_2048     = 3;   // SVID / RS256 issuer
  KEY_TYPE_ML_DSA_44    = 4;
  KEY_TYPE_ML_DSA_65    = 5;
  KEY_TYPE_ML_DSA_87    = 6;
  KEY_TYPE_ML_KEM_512   = 7;
  KEY_TYPE_ML_KEM_768   = 8;
  KEY_TYPE_ML_KEM_1024  = 9;
  reserved 10 to 31;           // future asymmetric / KEM suites
}

enum SigningAlgorithm {
  SIGNING_ALGORITHM_UNSPECIFIED  = 0;  // legacy default path; key type still gates
  SIGNING_ALGORITHM_ED25519      = 1;
  SIGNING_ALGORITHM_ED25519_NKEY = 2;
  SIGNING_ALGORITHM_RS256        = 3;
  SIGNING_ALGORITHM_ML_DSA_44    = 4;
  SIGNING_ALGORITHM_ML_DSA_65    = 5;
  SIGNING_ALGORITHM_ML_DSA_87    = 6;
  reserved 7 to 31;
}

enum AeadAlgorithm {
  AEAD_ALGORITHM_UNSPECIFIED       = 0;  // rejected as INVALID_ARGUMENT
  AEAD_ALGORITHM_CHACHA20_POLY1305 = 1;
  AEAD_ALGORITHM_AES_256_GCM       = 2;
  reserved 3 to 15;                      // xchacha20-poly1305, SIV, …
}

enum KemAlgorithm {
  KEM_ALGORITHM_UNSPECIFIED = 0;
  KEM_ALGORITHM_ML_KEM_512  = 1;
  KEM_ALGORITHM_ML_KEM_768  = 2;
  KEM_ALGORITHM_ML_KEM_1024 = 3;
  reserved 4 to 31;
}

enum EnvelopeAlgorithm {
  ENVELOPE_ALGORITHM_UNSPECIFIED       = 0;
  ENVELOPE_ALGORITHM_AES_256_GCM       = 1;
  ENVELOPE_ALGORITHM_CHACHA20_POLY1305 = 2;
  reserved 3 to 31;
}

enum CatalogKind {
  CATALOG_KIND_UNSPECIFIED = 0;
  CATALOG_KIND_SIGNING     = 1;  // sign / verify / get_public_key / mint
  CATALOG_KIND_VALUE       = 2;  // get
  CATALOG_KIND_ENCRYPTION  = 3;  // encrypt / decrypt
}

// BYOK material for Import: write-only, never echoed. A oneof so the variant
// must agree with key_type; only ed25519_seed is honored at MVP.
message KeyMaterial {
  oneof material {
    bytes ed25519_seed = 1;  // 32-byte raw seed (MVP)
    bytes pkcs8_der    = 2;  // reserved; PKCS#8 DER import later
  }
}

// Self-describing AEAD ciphertext. Broker owns the nonce; deliberately NO
// request nonce field anywhere.
message CiphertextEnvelope {
  AeadAlgorithm alg = 1;
  uint32 key_version = 2;   // version that produced it (rotation grace-period decrypt)
  bytes nonce        = 3;   // broker-generated; 12 bytes for both MVP suites
  bytes ciphertext   = 4;   // AEAD ciphertext incl. the 16-byte tag
}

message KemEnvelope {
  KemAlgorithm kem_algorithm = 1;
  EnvelopeAlgorithm envelope_algorithm = 2;
  uint32 key_version = 3;
  bytes encapsulated_key = 4;
  bytes nonce = 5;
  bytes ciphertext = 6;
}

// Leak-proof list entry: names + kind + type + version COUNT only.
message CatalogEntry {
  string name = 1;
  CatalogKind kind = 2;
  optional KeyType key_type = 3;  // present for signing/encryption; absent for values
  uint32 latest_version = 4;      // version count, NOT contents
}

// ===== SigningService: asymmetric key lifecycle + signature ops.
service SigningService {
  rpc NewKey       (NewKeyRequest)       returns (NewKeyResponse);
  rpc Import       (ImportRequest)       returns (NewKeyResponse);   // BYOK; same reply shape
  rpc ImportSet    (ImportSetRequest)    returns (ImportSetResponse); // all-or-nothing auth, sequential imports
  rpc Sign         (SignRequest)         returns (SignResponse);
  rpc Verify       (VerifyRequest)       returns (VerifyResponse);
  rpc GetPublicKey (GetPublicKeyRequest) returns (GetPublicKeyResponse);
}

message NewKeyRequest  { string key_id = 1; KeyType key_type = 2; }
message NewKeyResponse { string key_id = 1; bytes public_key = 2; }
message ImportRequest  { string key_id = 1; KeyType key_type = 2; KeyMaterial material = 3; }
message ImportEntry    { string key_id = 1; KeyType key_type = 2; KeyMaterial material = 3; }
message ImportSetRequest  { repeated ImportEntry entries = 1; }
message ImportedKey       { string key_id = 1; bytes public_key = 2; }
message ImportSetResponse { repeated ImportedKey keys = 1; }
message SignRequest    { string key_id = 1; bytes message = 2; SigningAlgorithm algorithm = 3; }
message SignResponse   { bytes signature = 1; }
message VerifyRequest  { string key_id = 1; bytes message = 2; bytes signature = 3; SigningAlgorithm algorithm = 4; }
message VerifyResponse { bool valid = 1; }
message GetPublicKeyRequest  { string key_id = 1; optional uint32 version = 2; }
message GetPublicKeyResponse { string key_id = 1; KeyType key_type = 2; bytes public_key = 3; uint32 version = 4; }

// ===== AeadService: symmetric AEAD plus KEM/envelope encryption; broker-owned nonce.
service AeadService {
  rpc Encrypt        (EncryptRequest)        returns (EncryptResponse);
  rpc Decrypt        (DecryptRequest)        returns (DecryptResponse);
  rpc WrapEnvelope   (WrapEnvelopeRequest)   returns (WrapEnvelopeResponse);
  rpc UnwrapEnvelope (UnwrapEnvelopeRequest) returns (UnwrapEnvelopeResponse);
}

message EncryptRequest {
  string key_id = 1;
  AeadAlgorithm algorithm = 2;   // mandatory; must match the key's catalog keyType
  bytes plaintext = 3;
  optional bytes aad = 4;        // bound into the AEAD; not stored in the envelope
  reserved 5;                    // (was: caller nonce), forbidden by constraint
}
message EncryptResponse { CiphertextEnvelope envelope = 1; }
message DecryptRequest  { string key_id = 1; CiphertextEnvelope envelope = 2; optional bytes aad = 3; }
message DecryptResponse { bytes plaintext = 1; }
message WrapEnvelopeRequest { string key_id = 1; KemAlgorithm kem_algorithm = 2; EnvelopeAlgorithm envelope_algorithm = 3; bytes plaintext = 4; optional bytes aad = 5; }
message WrapEnvelopeResponse { KemEnvelope envelope = 1; }
message UnwrapEnvelopeRequest { string key_id = 1; KemEnvelope envelope = 2; optional bytes aad = 3; }
message UnwrapEnvelopeResponse { bytes plaintext = 1; }

// ===== SecretService: secret values, rotation, catalog listing. ListCatalog is server-streaming.
service SecretService {
  rpc GetSecret(GetSecretRequest)    returns (GetSecretResponse);
  rpc SetSecret(SetSecretRequest)    returns (SetSecretResponse);
  rpc RotateSecret(RotateSecretRequest) returns (RotateSecretResponse);
  rpc ListCatalog(ListCatalogRequest)   returns (stream CatalogEntry);   // server-streaming (§7)
}

// ===== MintingService: NATS + generic JWT. NOT SVID (that is the Workload API §6).
// Dispatch is the METHOD: there is no JwtPreset enum; each NATS kind is its own

service MintingService {
  rpc MintJwt          (MintJwtRequest)          returns (CredentialResponse); // generic custom-claim JWT
  rpc MintNatsUser     (MintNatsUserRequest)     returns (CredentialResponse); // U-prefix
  rpc MintNatsAccount  (MintNatsAccountRequest)  returns (CredentialResponse); // A-prefix
  rpc MintNatsOperator (MintNatsOperatorRequest) returns (CredentialResponse); // O-prefix
  rpc MintNatsSigner   (MintNatsSignerRequest)   returns (CredentialResponse); // account/operator signing key
  rpc MintNatsServer   (MintNatsServerRequest)   returns (CredentialResponse); // N-prefix
  rpc MintNatsCurve    (MintNatsCurveRequest)    returns (CredentialResponse); // X-prefix (xkey)

  google.protobuf.Duration ttl = 3;     // absent => non-expiring (expires_at unset)
  google.protobuf.Struct claims = 4;    // non-reserved extra claims; merged (§9 Q5)
}

service AdminService {

}

message StatusRequest {}
message StatusResponse {
  string backend = 1;   // backend label, e.g. "vault" or "spiffe-vault"
  string version = 2;   // broker crate version
  uint32 protocol = 3;  // gRPC API generation (= 1 for basil.broker.v1); package name is the truth
}

message WatchRequest { repeated EventKind kinds = 1; }   // empty => all kinds

enum EventKind {
  EVENT_KIND_UNSPECIFIED    = 0;
  EVENT_KIND_KEY_ROTATED    = 1;  // a catalog key rotated (transit bump / PKI re-issue)
  EVENT_KIND_BUNDLE_CHANGED = 2;  // a trust bundle / JWKS / CRL changed
  EVENT_KIND_REVOKED        = 3;  // a credential was revoked (X.509 serial or JWT jti)
  reserved 4 to 15;
}

message Event {
  EventKind kind = 1;
  google.protobuf.Timestamp at = 2;
  oneof detail {
    KeyRotated    key_rotated    = 3;   // per-key; PDP-filtered like List (§7)
    BundleChanged bundle_changed = 4;   // trust-domain scoped; public bundle material
    Revoked       revoked        = 5;
  }
}
message KeyRotated    { string key_id = 1; uint32 new_version = 2; }
message BundleChanged { string trust_domain = 1; }
message Revoked       { string trust_domain = 1; string id = 2; }  // id = X.509 serial or JWT jti

// ===== Error model: see §4. Carried in google.rpc.Status.details.
// machine-readable reason, attached as an ErrorInfo detail with domain
// = "basil.broker.v1". The canonical gRPC Code carries the category (§4).
message BrokerErrorInfo {
  string reason = 1;   // SCREAMING_SNAKE, e.g. "UNSUPPORTED_KEY_TYPE"
  string op = 2;       // the RPC op token, e.g. "sign"
}

Idiomatic choices baked into the sketch

  • bytes everywhere binary. Message, signature, public key, plaintext, ciphertext, nonce, AAD, BYOK material are protobuf bytes: the JSON number-array workaround is gone, along with its on-wire bloat.
  • oneof only where it models a real union (KeyMaterial, Event.detail), not for op dispatch. The op is the method. That is the whole point of not porting the tagged union. This also retires the old JwtPreset enum: each NATS kind is now its own RPC (§9 Q7).
  • optional for true presence. Version, AAD, prefix, subject, ttl, expires_at, and key_type on CatalogEntry use explicit presence so “absent” is distinct from a zero/default.
  • Well-known types. ttl → Duration; expires_at/event time → Timestamp; free-form claims → Struct (§9 Q5). Self-documenting units.
  • Enums carry _UNSPECIFIED = 0, treated as INVALID_ARGUMENT so a client that forgets to set key_type fails fast rather than silently meaning Ed25519.

4. Error model: google.rpc.Status + error-details

gRPC's native failure channel is the status of the RPC itself: a canonical Code, a message, and a details list. The broker uses that instead of an in-band error body.

4.1 The two-layer mapping

Every former error code maps to (a) a canonical gRPC Code (the category, for transport-level handling and retries) and (b) a BrokerErrorInfo detail (reason + op, the machine-readable specific, for branching without string-matching). The handler builds one Status; the generated client reads back both.

4.3 Preserved security property: no key-existence oracle (critical) Implemented

The JSON handler makes “forbidden” and “not found” indistinguishable on the gated path. The gRPC mapping MUST keep this:

  • On any key-scoped RPC, a PDP deny (whether unknown-key, not-writable, or not-permitted) returns PERMISSION_DENIED with reason = "UNAUTHORIZED", identical bytes regardless of whether the key exists. NOT_FOUND is never returned from a gated RPC reached through the PDP.
  • DECRYPT_FAILED → INVALID_ARGUMENT with no further detail: the same single opaque outcome for bad tag / bad AAD / wrong version, so no padding/AAD oracle.
  • List is per-key PDP-filtered, server-side. The stream yields a CatalogEntry only for keys the caller may see; a caller with no attested uid gets an empty stream (fail-closed), never a distinguishing error. Value-free metadata only.
  • BYOK material never crosses the wire outbound. Import replies with the public half only; Get is the sole value-returning RPC and the manager rejects a crypto-key class before any backend read.

4.4 No-SO_PEERCRED peer → fail closed

When the kernel cannot report SO_PEERCRED, every gated op is denied. In gRPC this is the one place we use UNAUTHENTICATED + reason = "NO_PEER_CRED": we could not establish who is calling at all (distinct from “we know who you are, you're not allowed”). List from such a peer still returns an empty stream rather than an error.

4.5 unknown_op is gone

The whole lenient unknown-op decode machinery disappears. gRPC dispatches by method; an unknown method is tonic's transport-level UNIMPLEMENTED and never reaches a handler, and a malformed message is a transport decode error scoped to that one RPC, neither closes the HTTP/2 connection. The connection-keepalive property the JSON wire worked for is free here.

5. Attestation + PDP integration (consumes the shared foundation)

The shared transport foundation is owned by spiffe-workload-api.html §3. This design consumes it and does not re-specify it. The contract:

  1. One tonic server, one socket, multiplexed services. The five broker services are added to the same router alongside the SPIFFE services; gRPC multiplexes them by fully-qualified method name on one connection.
  2. Per-connection SO_PEERCRED attestation in request extensions. Every broker handler can read the attested PeerInfo (in particular uid: Option<u32>) from the request extensions; the foundation's layer attests and injects it.
  3. PDP decide callable per-RPC, default-deny. The handler reuses the existing PDP decision unchanged. The decision is recorded for every gated RPC, allow or deny.
  4. Shared backend, sign-in-place. The same backend manager the JSON handler held is shared with the gRPC handlers; gRPC is a second front door onto the same state. Keys never leave the Vault-compatible backend.
  5. No-panic handlers. Every handler returns a result; a backend failure becomes a Status (§4), never an unwind.

Implementation shape. Rather than re-implement the gate in every handler, the broker reuses the one shared helper owned by the foundation (spiffe §3.4): authorize(req_ext, Op, key) → Result<u32, Status>. It reads PeerInfo from extensions, fails closed UNAUTHENTICATED if there is no uid, calls the PDP, records the decision, returns PERMISSION_DENIED on deny (identical bytes for unknown/forbidden, §4.3), else the validated uid. A tower interceptor cannot do the full job alone (it does not know the per-RPC key_id, which is inside the decoded message), so the gate is called at the top of each handler with the message in hand; the interceptor's job is only to attest and inject PeerInfo.

Sync (spiffe §14): the helper signature and the internal-error→Status table are shared and identical across both services; the helper returns the validated uid.

6. Minting boundary & feature split

Minting is a clean sibling to SVID issuance, not a duplicate. Decision: split by credential kind, at the service boundary.

  • X.509-SVID and JWT-SVID issuance → the SPIFFE Workload API (basil.workload.v1, vault-gqn). That is the SPIFFE-standard surface real consumers expect. The SVID minting logic (caller-templated SPIFFE id, the required audience, the alg-from-issuer) lives there. It is no longer reachable through the broker.
  • NATS JWTs, generic custom-claim JWTs, and DNS/IP-SAN TLS certificates → the broker MintingService (basil.broker.v1). The generic + NATS minters and backend-PKI TLS leaf issuance stay broker-side.

6.1 Feature-flag behavior (resolved, §9 Q1)

mint_jwt was an overloaded term. JWT-SVID minting is part of the SPIFFE implementation and is not duplicated to be available when the spiffe feature is disabled. NATS-format and generic JWTs still mint with spiffe disabled. So the no-spiffe build is the broker gRPC services with NATS + generic JWT intact and zero SPIFFE code or SPIFFE-specific deps.

6.2 Module split that makes this work (resolved, §9 Q1; spiffe §11)

The minting code splits across the target module layout so the dependency graph stays acyclic and services never reach sideways:

PieceModuleCompiled when
The sign-in-place primitive (build signing input → backend signs → assemble JWS)core/always
Generic + NATS minting and TLS certificate issuance (MintJwt, MintNats*, IssueCertificate)service/mintingalways

Both service/minting and service/spiffe call the shared sign-in-place primitive in core/, so service/spiffe never depends on service/minting and there is no logic duplication. Both gate on the Mint op against the issuer key; same capability model, same default-deny, same audit op.

7. Evolution, versioning, Watch & streaming

7.1 Package version

The package is basil.broker.v1; the major version is in the package name (standard gRPC convention). Per the flag-day policy that is a clean cut, not a v1-alongside-v2 window. StatusResponse.protocol = 1 echoes the generation as a cheap runtime probe, but the package name is the source of truth; supported_ops is dropped because gRPC server reflection + the service descriptors already enumerate the methods.

7.2 Field-number discipline

  • Never renumber or reuse a field/enum number. Deleting reserves it. The proto reserves EncryptRequest field 5 (the forbidden caller-nonce), the PQ ranges in KeyType/AeadAlgorithm, and EventKind 4–15.
  • Additive evolution only. A new algorithm is a new enum value in a reserved range; a new op is a new RPC; a new optional field gets the next free number.
  • Explicit presence (proto3 optional) wherever absence is meaningful, so a newer client sending a field an older server ignores stays wire-valid.
  • Enum zero values are _UNSPECIFIED and rejected, so adding a value never silently changes the meaning of an unset field.

7.3 proto3, not editions (resolved, §9 Q2)

The proto is proto3 + optional. The deciding factor is consistency: the SPIFFE side vendors the upstream workloadapi.proto verbatim, and that file is proto3 (spiffe §9.1). Mixing a protobuf-editions broker proto with a proto3 SPIFFE proto in one basil-proto build would be gratuitous inconsistency, and the message/field design is identical either way. One syntax, one tonic-build invocation.

7.4 AdminService.Watch: in scope (resolved, §9 Q8) Implemented

The earlier draft deferred Watch because rotation was synchronous and the audit sink was JSONL-to-disk with no event bus. It is now implemented, scoped to rotation / bundle / revocation events (audit tail remains deferred). The key realization: the SPIFFE side already introduces an internal watch over issuer/CA rotation and bundle/CRL changes to drive its streaming push (spiffe §6.4, §7.1, §8.3). Watch surfaces that same internal event source: there is one event bus, two streaming consumers (the SPIFFE bundle streams and this RPC), not two mechanisms.

  • Event kinds: KEY_ROTATED (a catalog key rotated), BUNDLE_CHANGED (a trust bundle / JWKS / CRL changed), REVOKED (an X.509 serial or JWT jti was revoked; see spiffe §8).
  • Leak-proof, like List: Watch requires an attested uid (empty/closed stream otherwise). KEY_ROTATED events are per-key visible-filtered (a caller sees a rotation only for keys it may see). BUNDLE_CHANGED / REVOKED carry only public bundle material (already published via FetchJWTBundles / the CRL), so they are emitted to any attested caller.
  • No-panic streaming: the handler select!s over the event source and the shutdown receiver, maps errors to Status, and a dropped client just drops the stream, same posture as the SPIFFE streams.
  • Audit-tail subscription remains deferred (it would need the JSONL sink to become a pub/sub); additive later as another EventKind or a separate RPC.

7.5 Streaming scope

  • List server-streaming: in scope. The per-key PDP filter is naturally a stream: yield each visible CatalogEntry as it passes the predicate, backpressured, without buffering the whole catalog.
  • Watch server-streaming: in scope (§7.4).
  • No client-streaming / bidi. None of the ops need it; the AEAD ops are bounded by a size cap (a chunked encrypt stream for huge blobs is a possible future additive AeadService RPC).

8. Migration plan & crate layout

8.1 Flag-day, not dual-run

The project is pre-release with no external users, so the length-prefixed-JSON wire is deleted in the same change that lands gRPC: no --protocol both, no parallel transports, no compatibility shims. gRPC becomes the one and only wire. There is no transition window and nothing “runs alongside.”

  1. Proto + codegen are in the shared basil-proto crate (tonic-build/prost), shared with the SPIFFE side. Workspace deps include tonic, prost, tonic-types (error details), and prost-types.
  2. gRPC handlers reuse the core verbatim. The services are a thin gRPC adapter over the same backend manager / PDP / minters (decode message → authorize(Op, key) → core call → typed response / Status). No backend, PDP, or minter logic is rewritten.
  3. Delete the JSON wire in the same cut. Remove the tagged-union request/response types, the JSON codec, the JSON serve loop, and the number-array binary workaround. The hand-rolled domain types and the error-code enum are replaced by the generated proto types, not retained as compat newtypes (§9 Q5/Q6).
  4. Clients become gRPC-only. The client crate is reimplemented over the tonic-generated service clients behind the same facade, so call sites use client.sign(key, message); the blocking client wraps a runtime around the async one. The typed MintNats*, ImportSet, and IssueCertificate RPCs replace raw-tonic escape hatches.
  5. One transport. The agent serves a single tonic server (the five broker services plus the SPIFFE Workload API when the spiffe feature is enabled) over the one socket the foundation owns. No JSON listener exists.

8.2 Crate layout & module placement

The cutover lands with the crate restructure recorded in AGENTS.md (the previous server-library + thin-binary split is replaced):

Tuple returns are likewise replaced by named structs such as KeyHandle, SecretValue, MintedJwt, IssuedCertificate, and AgentStatus.

The no-spiffe build is gRPC with fewer services, never a JSON fallback: it still speaks gRPC, just without the Workload-API services and SPIFFE-specific deps (§6.1).

9. Decisions

Every prior open question is resolved; the resolutions are folded in below.

  • Five services, one package (SigningService / AeadService / SecretService / MintingService / AdminService in basil.broker.v1), mirroring the PDP role clusters; registered on the one shared tonic server (§2).
  • No giant oneof. Per-op messages; dispatch is the gRPC method. This also retires the JwtPreset enum: each NATS kind is its own RPC (§3, §3.1).
  • bytes for all binary, dropping the JSON number-array workaround (§3).
  • google.rpc.Status Code + BrokerErrorInfo detail replace the error-code enum: Code = category, reason = machine-specific, op preserved. Unsupported key/algorithm → UNIMPLEMENTED, reconciled with the spiffe doc (§4).
  • Security properties preserved: PDP deny (incl. unknown-key) → uniform PERMISSION_DENIED; DECRYPT_FAILED opaque; List per-key-filtered server-stream, empty on no-uid; no-uid → UNAUTHENTICATED; BYOK material request-only (§4.3, §4.4).
  • Minting split by credential kind: SVID → Workload API; NATS/generic → broker. JWT-SVID is not duplicated when spiffe is off; NATS/generic JWT still mint. The shared sign-in-place primitive lives in core/; SVID minting in service/spiffe, NATS/generic in service/minting; the graph stays acyclic (§6; Q1).
  • proto3 + optional, not protobuf editions, to match the vendored SPIFFE proto3 in one build (§7.3; Q2).
  • One listener, multiple services (Q3): the broker registers on the shared tonic server / socket owned by the foundation (§5).
  • Shared basil-proto crate as in AGENTS.md (Q4; §8.2).
  • claims as google.protobuf.Struct (Q5): idiomatic JSON-in-proto. Numeric extra claims round-trip as f64, accepted; the reserved numeric claims (iat/exp/jti) are server-generated and never cross the wire, and a caller needing an exact large integer encodes it as a string. The JSON wire is dropped entirely; no number-array workaround remains.
  • No client-side error-code enum retained (Q6): clients branch on the gRPC Code + reason directly; nothing kept for “churn” reasons (no-backwards-compatibility).
  • Typed NATS mint family (Q7): MintNats{User, Account, Operator, Signer, Server, Curve}, each a first-class RPC with its own request shape (the inputs genuinely differ; the issuer NKey prefix is derived from the issuer key, not a wire field). Curve is the X-prefix (xkey). The full family is implemented Implemented (§3, §3.1).
  • IssueCertificate lives in MintingService: it issues TLS leaves from backend PKI roles and is gated by Mint on the issuer key; it is deliberately separate from SPIFFE SVID issuance.
  • AdminService.Watch implemented now (Q8), scoped to rotation / bundle / revocation events on the shared internal event source; audit tail deferred (§7.4).
  • Attestation + PDP consumed from the foundation, applied per-RPC via the shared authorize(req_ext, Op, key) → uid helper + an attest-only interceptor (§5).
  • Versioning by package name (…v1), _UNSPECIFIED zero values, reserved ranges, optional presence, unknown_op machinery removed (§4.5, §7).
  • Flag-day migration: the JSON wire is deleted in the same change that lands gRPC; clients regenerated behind the renamed Client facade (§8).

9.1 Options considered (and rejected)

  • One Broker service with ~13 RPCs: rejected. The JSON tagged union with a new coat of paint; ignores the PDP's capability partition and gives every scoped client the whole surface.
  • Port the tagged union 1:1. Rejected by the hard requirement. Throws away typing, bytes, status codes, error-details, and streaming.
  • Keep the in-band error as a successful response: rejected. gRPC's failure channel is the RPC status.
  • NOT_FOUND for a missing key on a gated RPC, rejected; it would leak existence (§4.3).
  • Keep an svid broker mint preset: rejected. One way to get an SVID (the standard Workload API); the broker mints only NATS/generic (§6).
  • A JwtPreset enum with NATS arms. Rejected in favor of typed per-kind RPCs (Q7); the preset enum was a mini-tagged-union the method dispatch replaces.
  • protobuf editions for the broker proto, rejected for consistency with the vendored proto3 SPIFFE file (§7.3).
  • supported_ops string list in Status: rejected. gRPC reflection + descriptors are discovery.
  • A transport toggle (JSON ⇄ gRPC). Moot: flag-day means there is no transition window; the one compile-time feature is spiffe, which gates the SPIFFE services + deps, not the wire.
