1. Goal

Basil needs post-quantum signing and KEM/envelope encryption before OpenBao or Vault transit necessarily supports the required NIST algorithms. The public contract must remain the same broker gRPC operation contract: clients ask Basil to sign, verify, encapsulate, decapsulate, wrap, or unwrap. Clients never receive private key material and never choose nonce or key-handling details.

This design introduces a provider-neutral crypto layer inside basil-agent and a policy-gated local software custody mode. Vault-compatible transit (crypto_provider=vault-transit, backed by OpenBao or HashiCorp Vault) remains the preferred provider whenever it supports an algorithm. A local software provider is an explicit fallback for PQC algorithms that need in-process operation until a backend-native implementation exists.

2. Threat Model

Software-custodied PQC keys are more sensitive than transit keys because Basil must decrypt the private key record into process memory for one operation. The risk is bounded by:

  • Default-deny policy: local software custody is unavailable unless the catalog opts the key into that mode and policy permits the operation.
  • Operation-scoped unwrap: private material is decrypted only for the current sign, verify, decapsulate, wrap, or unwrap operation and is stored in Zeroizing buffers where dependencies permit.
  • No export path: no gRPC, CLI, catalog, list, or metadata operation returns private material.
  • No oracle behavior: malformed ciphertext, AAD mismatch, unsupported provider, and policy denial collapse to canonical errors that do not reveal secret state.
  • Audit durability: each operation records key id, version, algorithm, provider, caller uid, custody mode, and outcome, without secret bytes.

3. Catalog Labels

Until a stronger schema field is justified, provider metadata lives in labels. Reserved labels are validated by the catalog loader in the PQC label task:

  • crypto_provider=vault-transit|local-software
  • crypto_provider_version=<opaque provider version or crate version>
  • pqc_algorithm=ml-dsa-44|ml-dsa-65|ml-dsa-87|ml-kem-512|ml-kem-768|ml-kem-1024
  • pqc_custody=backend-native|software-encrypted
  • pqc_storage_key=<backend AEAD key id>
  • crypto_provider_policy=backend-preferred|backend-required|local-software
  • migration_target=<catalog key id or backend locator>

Unknown labels remain preserved. Duplicate reserved labels are fatal because provider routing must be deterministic.

4. Provider Policy

Each key resolves to one of three policy modes:

  • backend-required: use only backend-native provider support. If Vault-compatible transit cannot do the algorithm, fail closed with UNIMPLEMENTED or the service's canonical unsupported-provider status. No local unwrap is attempted.
  • backend-preferred: use backend-native support when available; otherwise use the local software provider only if pqc_custody=software-encrypted and policy explicitly permits local software custody for the caller and operation.
  • local-software: use the local software provider. This mode is allowed only for keys whose encrypted private key record is present and whose policy grants the operation. It is never inferred.

The default is backend-required for non-PQC keys and for PQC keys without a recognized provider policy label.

5. Crypto Provider Trait

The dispatch layer lives under crates/basil-agent/src/core and remains independent of service::*. It adapts existing Vault-compatible transit operations and the new local PQC provider behind one internal trait. The initial implementation is core::crypto_provider: it parses the reserved catalog labels, selects vault-transit or local-software according to provider policy and custody metadata, delegates legacy signing/import/generate operations to the backend adapter, and returns closed unsupported/provider-policy errors for PQC operations until native or local provider support lands.

#[async_trait]
pub trait CryptoProvider: Send + Sync {
    fn provider_id(&self) -> &'static str;

    async fn generate_key(&self, request: GenerateKey) -> Result<KeyRecord, ProviderError>;
    async fn sign(&self, request: SignRequest<'_>) -> Result<Signature, ProviderError>;
    async fn verify(&self, request: VerifyRequest<'_>) -> Result<bool, ProviderError>;
    async fn encapsulate(&self, request: EncapsulateRequest<'_>) -> Result<KemCiphertext, ProviderError>;
    async fn decapsulate(&self, request: DecapsulateRequest<'_>) -> Result<SharedSecret, ProviderError>;
    async fn wrap_envelope(&self, request: WrapEnvelopeRequest<'_>) -> Result<Envelope, ProviderError>;
    async fn unwrap_envelope(&self, request: UnwrapEnvelopeRequest<'_>) -> Result<Vec<u8>, ProviderError>;
}

Concrete request/response structs carry the catalog key id, backend path, key version, algorithm, AAD, and provider metadata. They do not carry raw private material except inside provider-internal local software unwrap code.

6. Software Key Storage

Software-custodied PQC private keys are stored encrypted at rest in Vault-compatible KV or an equivalent backend value store. Records include:

  • schema version
  • key id and key version
  • public key bytes
  • encrypted private key bytes
  • algorithm and parameter level
  • provider id and provider version
  • custody mode
  • backend AEAD wrapping key id
  • creation and rotation metadata

The record is malformed if any identity field disagrees with the catalog labels or the requested operation. Malformed records fail closed before private material is decrypted.

Private material unwrap is scoped to a single provider call:

  1. Read encrypted key record.
  2. Validate metadata against catalog labels and request algorithm.
  3. Decrypt into a Zeroizing<Vec<u8>> or provider-native zeroizing key type.
  4. Execute the operation.
  5. Drop private material before returning.

7. Backend-Native Migration

When OpenBao/Vault gains native ML-DSA or ML-KEM support, the migration target is recorded in migration_target. Operators rotate or re-provision the key into the backend-native provider, then change labels to crypto_provider=vault-transit, pqc_custody=backend-native, and crypto_provider_policy=backend-required.

There is no compatibility shim or dual-running path. Basil is pre-release, so the clean migration is a flag-day catalog change plus changelog entry.

8. Public API Impact

The broker gRPC API is extended for algorithm names and KEM/envelope operations, but the custody abstraction stays private to basil-agent. Clients see normal signing and envelope responses plus canonical errors:

  • unsupported provider or algorithm: UNIMPLEMENTED
  • policy denial: PERMISSION_DENIED
  • malformed ciphertext/AAD/version: opaque invalid-argument/decrypt-failed shape
  • backend unavailable: UNAVAILABLE

No client-visible operation exports private material, software key records, or provider-internal metadata beyond non-sensitive list/metadata fields.
