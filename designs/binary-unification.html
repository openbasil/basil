Unify into a single basil binary

RATIFIED: decisions recorded (§8); ready to implement
Tracking: br basil-ju4s  ·  Related: br basil-8hws (OTLP feature-gate)
Date: 2026-06-25  ·  Pre-release; flag-day migration (no backwards-compat shims, per repo policy).

1. Summary

Collapse the two human-facing binaries (basil from basil-cli, and basil-agent) into one basil binary, following the vault/consul model: the long-running daemon becomes a subcommand (basil agent), and every other command (the over-socket client commands and the offline config commands) lives on the same binary. The bulk of the code moves into a new basil-core library; a thin basil-bin clap crate produces the binary; the published basil client library stays lean (proto-only); and the cross-crate live/interop test suite + harness move into a dedicated basil-tests crate.

Primary driver is ergonomics: one binary in $PATH for every admin/dev operation. Today the split is by implementation (does the command need server-side core?), which leaks into the user's mental model: basil list talks to the broker, but basil-agent check validates its config. Users should not have to remember which binary hosts which command.

2. Goals & non-goals

Goals

  • One basil binary for all admin/dev operations; basil agent launches the daemon.
  • A clean, acyclic crate topology with the bulk in basil-core and a thin binary crate.
  • Keep the published basil client library exactly as lean as today (proto-only): embedders unaffected.
  • A dedicated home for the cross-crate live/interop suite + the shared harness.
  • Bonus, secondary: reduce total shipped bytes by compiling the gRPC/async/clap stack once (it is currently duplicated across both binaries), and coordinate with OTLP feature-gating (br basil-8hws).

Non-goals

  • No change to the gRPC contract (basil-proto) or the client library's public API.
  • No backwards-compat aliases/dual-paths: this is a flag-day rename (repo policy).
  • The OTLP feature-gating itself is a separate ticket (basil-8hws); independent, coordinate only.
  • Not chasing binary size via crate-splitting: cargo bloat shows the per-command glue is kilobytes against a ~10 MiB .text; size lives in the framework/crypto/OTEL generics, not the subcommands.

3. Current state (the problem)

BinaryCrateCommandsNeeds
basilbasil-cli sign, verify, encrypt, decrypt, get, set, list, import-set, mint-jwt, mint-nats-user, issue-cert, status, health, ready, reload the basil client (gRPC over the socket)
basil-agentbasil-agent run (daemon), init, check, explain, doctor, bundle server-side core (loader/PDP/schema, seal/unlock, backend probe)

The seam is "does it need server-side core", an implementation fact, not a user-meaningful one. Two relevant findings from cargo bloat on release builds:

  • Both binaries are dominated by monomorphized generics from the shared framework stack (h2/tonic/tokio/clap) plus crypto and OTEL, spread across thousands of small methods, not in the subcommands. The biggest command-specific symbol is ~20 KiB against a ~10 MiB .text.
  • The "lean" CLI is already ~10 MiB because it carries the full gRPC client stack (tonic/h2/hyper/tokio) to talk to the broker. "Keep the CLI lean" was protecting a binary that is already large for unavoidable reasons.

So the topology decision is essentially free to optimize for UX. Size is a profile/codegen + feature-gating problem (strip/LTO; gate OTLP via basil-8hws), not a crate-topology one.

Dependency edges (a clean DAG)

basil (client)  -> basil-proto
basil-core      -> basil-proto, basil-nats, (backends/crypto/seal)        # NOT the client crate
basil-bin       -> basil-core, basil (client), clap
basil-tests     -> basil-core, basil (client), basil-proto, (interop deps)

Bulk = basil-core.  Bin = thin.  Client lib = lean.  No cycle.

Command split (by what each needs)

  • Offline / server commands (init, check, explain, doctor, bundle) + the daemon serve logic → basil-core as library fns; basil-bin dispatches into them.
  • Over-socket client commands (sign, verify, encrypt, decrypt, get, set, list, import-set, mint-jwt, mint-nats-user, issue-cert, status, health, ready, reload, + future revoke) call a running broker → they need only the basil client, so they live in basil-bin (thin basil::Client wrappers).

This keeps basil-core a pure server/offline library: no "server depends on client" smell. basil-bin is the one crate that wires both halves, which is exactly its job as the unified frontend.

Folded-in cleanup

Today basil-agent imports proto types through the client crate (use basil::proto::…): the server reaching through the client. In basil-core, import from basil-proto directly (sweep basil::proto:: → basil_proto::) and drop the client-crate dependency. Only basil-bin touches the client crate.

Tests

basil-tests holds only the system/interop tests (boot the binary + a dev backend, drive over the socket, shell the Go OIDC verifier). The harness becomes the crate's src/lib.rs (a real, documented API: boot_basil, boot_dev_backend, alloc_addr, the dev-server semaphore, fetch_jwt_svid, rotate_transit_key, kill_backend, Harness), compiled once and linked by all test binaries (better than today's per-binary mod common;). Unit tests stay inline in basil-core's modules (they need internals). Wins: isolates heavy dev-deps (spiffe, spiffe-rustls, jsonwebtoken, reqwest, x509-parser, the Go driving) off the production crates, and collapses the justfile recipe to cargo test -p basil-tests --features live-e2e (no more --test foo --test bar enumeration).

Per D-7, basil-tests is also where the CLI/command coverage lives: no mock server and no dedicated unit tests for the basil-bin command bodies. Instead: every over-socket command gets a gRPC client test (positive + negative); every CLI command with flag variations is driven by invoking the basil binary in a subshell; and authorization is exercised against the running uid/gid via a small root-only set that is normally #[ignore]d.

Wrinkle: a separate crate gets no free CARGO_BIN_EXE_basil; the harness already builds + locates target/debug/<bin> via ensure_binaries_built, so this keeps working (or use escargot).

8. Decisions

These were the open questions; all are now resolved. Identifiers map to the original questions (D-1 = OQ-1, etc.) for traceability.

  1. D-1 Command namespacing.
    Decision: basil agent (bare) starts the service, and does nothing else. The offline config commands group under a dedicated basil config namespace: basil config init, basil config check, basil config bundle, basil config explain. Rationale: reserving basil agent solely for launching the service means there is no risk of accidentally starting a long-running daemon while merely checking the config. doctor is top-level (basil doctor) because it is broader than config checking: it probes dependencies, socket permissions, and the environment. Data-plane, minting and admin commands (basil sign, basil reload, basil revoke, …) stay top-level. A later CLI reorg may group some other level-1 commands; that is deferred.
    (Differs from the original recommendation, which nested config/lifecycle under basil agent …; the ratified choice is a separate basil config namespace with a bare basil agent.)
  2. D-2 Daemon invocation.
    Decision: basil agent (bare) launches the daemon: no run subcommand, no server. Reinforces D-1: the basil agent verb is the daemon and only the daemon.
  3. D-3 Clean rename vs a basil-agent compat shim.
    Decision: clean flag-day rename, no basil-agent symlink/alias (repo policy: no compat layers; pre-release, no external users). The Nix unit is updated in the same change.
  4. D-4 Binary crate name.
    Decision: basil-bin. The crate hosts the daemon dispatch too, so "cli"/"agent" would both mislead.
  5. D-5 Default feature set of the unified binary.
    Decision: keep the current defaults (spiffe on; OTLP off once basil-8hws lands; unlock-disk/live-e2e test-only). basil-bin re-exposes the meaningful features so distro/Nix can toggle them. See D-8 for how a feature is declared across the basil-bin → basil-core edge.
  6. D-6 Daemon-only artifact.
    Decision: defer. The daemon retains core regardless, so the savings are just the tool glue (small). Treat as a later packaging knob, not part of this change.
  7. D-7 Testing the CLI / over-socket command bodies.
    Decision: do not build a mock server, and we do not need dedicated Rust unit tests for the socket-command bodies in basil-bin, provided the following coverage exists (it lives in basil-tests):
    • Every over-socket command has a test where a client connects over gRPC and runs both positive and negative cases.
    • Every CLI command with flag variations is exercised by an automated test that invokes the basil binary in a subshell.
    • Authorization is tested against the running uid/gid: one small set of tests must run as root and is normally #[ignore]d.
    (Supersedes the original recommendation to factor the command bodies into render-into-Writer free fns; the ratified approach tests them end-to-end via gRPC + subshell instead.)
  8. D-8 Coordination with basil-8hws (OTLP feature-gate).
    Decision: land basil-8hws first, or at the same time as the unification. The OTLP feature must be declared in both basil-bin and basil-core: normally off in basil-core, and enabling it in basil-bin enables it in basil-core (the bin propagates the feature down). This is the template for how any meaningful feature flows basil-bin → basil-core (D-5).
  9. D-9 One basil-core, or split later?
    Decision: one basil-core now; revisit only if compile times or boundaries demand it.
